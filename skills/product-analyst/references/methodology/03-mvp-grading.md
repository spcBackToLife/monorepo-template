# MVP 分级 P0 / P1 / P2 / P3

## 分级标准

```
P0：MVP 必须，没它不成产品
P1：基础体验，没它产品很难用
P2：增长体验，没它不影响核心使用
P3：未来规划，明确不在本期
```

## 切分原则

### P0 标准（缺一就不成产品）
- 不做这个，产品演示都跑不通
- 不做这个，用户的核心流程断裂
- 例：社交产品的"登录 + 看内容 + 发内容"

### P1 标准（基础体验底线）
- 不做这个，用户的核心流程能跑但体验明显残缺
- 用户会问"怎么连这个都没有"
- 例：社交产品的"通知 / 个人资料 / 隐私设置"

### P2 标准（增长体验）
- 不做这个，核心使用不受影响
- 但留存 / 转化 / DAU 会差一截
- 例：搜索高级筛选 / 数据看板 / 批量操作

### P3 标准（未来规划）
- 明确写出"本期不做"
- 留作后续迭代记录
- 防止反复争论"这个要不要做"

## 砍模块的理由必须留痕

每个被砍到 P2 / P3 的模块必须在 md 中说明：
- 为什么本期可以不做
- 不做对用户体验的影响范围
- 后续什么条件触发实施

这些理由进 `decisions/` 或 `00-overview.md` 的"替代方案与否决"段。

## 沉淀到 schema

```jsonc
// modules 中标注 priority
project.meta.modules = {
  M1: { name: "...", priority: "P0", summary: "..." },
  M5: { name: "...", priority: "P2", summary: "..." }   // 含本期不做的理由
}

// 关键砍模块决策可入 constraints.decisions
project.meta.constraints.decisions = [
  { id: "D1", summary: "本期不做支付——理由：MVP 核心流程不依赖；接入合规成本高" }
]
```
