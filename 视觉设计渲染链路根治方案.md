# 视觉设计渲染链路根治方案

> **背景**：用 design-planner 技能给 `校园社交-登录页` 做视觉设计，schema integrity 0 错通过，出场门禁全过，但实际渲染严重 broken（无 FormCard 卡片视觉 / SubmitBtn 无主色底 / 装饰 blob 不可见 / ModeToggle 两 tab 挤一起 / GetCodeBtn 与 placeholder 重叠 / BrandLogo broken image…）。
>
> **目标**：本文档**不**讨论"怎么修这次 demo 的视觉"——demo 是测试手段，不重要。本文档用第一性原理拆解"为什么这种 bug 会从写到提交全程没人拦下来"，提出**事前治理**的根治方案。
>
> **覆盖**：design-planner 技能 / design-mcp 工具 / 前端 SchemaRenderer / 后端 design-schema service / 流水线产物契约。

---

## 1. 第一性原理：什么是"视觉设计正确"？

```
正确 = 设计师在 schema 里写的视觉意图
       1:1 等价于
       最终用户屏幕上看到的画面
```

中间的因果链上每一个环节都可能让"意图 ≠ 画面"。当前 bug 不是某一个点出错，**是这条链上多个环节没有"契约对账"**：

```
[1] 设计师/AI 视觉意图
       ↓ ① SKILL.md 告诉怎么写  (人类文档)
[2] AI 落到 schema 的 styles / visualState / materialSpec
       ↓ ② MCP 工具校验          (Zod schema 校验，但只查字段类型不查 token 合法性)
[3] 后端 service 持久化 schema
       ↓ ③ API 返回原样          (无校验)
[4] 前端拿到 schema
       ↓ ④ SchemaRenderer 解析 token + 应用 styles  (实现细节，文档外)
[5] DOM + CSS
       ↓ ⑤ 浏览器渲染            (标准)
[6] 用户看到的画面
```

每条 `↓` 都需要"契约一致"才能保证 [1] = [6]。当前的失败本质是：

- ① 是**说出来的契约**（文档），不是机器对账
- ② 是**字段级类型校验**，不验证语义（如 token 引用合法性、CSS 属性兼容性）
- ④ 是**黑盒实现**，文档里没写清楚边界，AI 无法预知
- 整条链路**没有任何一个点"自然显示渲染失败"**——直到用户人肉看截图才发现

## 2. 五个根源（按"为什么 → 第一性原理解"组织）

### 根源 1：契约靠"人类文档"而非"代码契约"

#### 为什么会发生

- `schema-spec/node-styles.md §2` 表写 `$token:typography.fontSize.body` 这种深层引用
- 前端 SchemaRenderer 实际可能只支持 `$token:colors.X` / `$token:spacing.X` 两层
- AI 严格按文档写 → schema 通过校验 → 渲染失败
- **文档与代码的真相源不一致**，且没有自动化机制让它们同步

#### 第一性原理

> 契约的本质是"双方对承诺的共识"。让两方靠"看同一份文档"达成共识，这种共识在任何一方变更时立刻漂移。**真正的契约必须是双方都会自动消费的东西**——如代码、类型定义、运行时 validator。

#### 治理方向

```
建立 packages/design-token-spec
├── src/
│   ├── token-grammar.ts      // TokenRef 类型 + 合法语法的 TS 类型
│   ├── token-resolver.ts     // 给定 ThemeConfig + TokenRef，返回 resolved value 或 ResolveError
│   ├── style-property-spec.ts // 哪个 CSS 属性允许 TokenRef、允许哪种引用层级
│   └── validators.ts         // 运行时 validator（Zod refinement）

消费方（共享 import 同一份）：
- 前端 SchemaRenderer:          import { resolveToken } 实际渲染时用
- 后端 design-schema service:   import { validateTokenRef } 写入时校验
- design-mcp tools:             import { validateStyles } style/update 入参拒非法
- design-planner SKILL.md:      schema-spec/node-styles.md 由该包自动生成 + 内嵌示例
```

**效果**：
- AI 写 `$token:typography.body.fontSize`（深层）→ MCP `style/update` 调用直接 422 拒掉，不让进 schema
- AI 收到错误信息：「TokenRef 不合法：渲染器仅支持扁平引用 `$token:colors.X / spacing.X / radius.X / shadows.X / transitions.X` 或 typography group `$token:typography.body`」
- 文档不再是"承诺"，是"自动生成的说明"——前端改了 token 解析能力，AI 这边的工具立刻同步

### 根源 2：阶段产物"在画布上不自洽"

#### 为什么会发生

每个阶段的设计假设：

| 阶段 | 假设 | 现实 |
|------|------|------|
| product | 节点树建完，留 src/styles 给后续阶段 | type=img 没 src → 画布显示 broken image icon |
| theme | token 全齐 | typography 实际嵌套结构与渲染器期望不一致 |
| interaction | events / bind / 表达式建完 | 表达式作用域里的 `$.now()` `Math.floor` 实际渲染期不可用 |
| design | styles 全写完 | 字符串内嵌 token 不解析 / animation 缺 @keyframes / safe-area 在画布无意义 |
| executor | 应用素材后才"完整可视" | 直到这一步前，没有任何阶段画布是"接近最终"的 |

**整条流水线只有最后一棒（executor）才能看到"完整画面"**——前面 4 棒标 done 时实际画布都是部分 broken 的，没人能在出场前发现问题。

#### 第一性原理

> 反馈延迟越长，bug 越难暴露。一个产物如果"暂时不能用、要等到后面阶段才能用"，那这个产物的"暂时状态"必须被显式定义和验证，而不是被忽略。**每个中间阶段的产物，都应该在自己阶段的画布上"自洽可视"**——哪怕用 placeholder 替代真素材。

#### 治理方向

**建立"画布渲染契约"——每个阶段出场前，画布必须无 broken 视觉。**

```
渲染器内建 5 类 fallback：

1. type=img 空 src        → 显示 placeholder 卡片（带尺寸 + alt 文字 + 浅灰底 + 角标"待 executor 上传"）
2. token 引用不合法       → 控制台警告 + 回退到 theme.tokens 中第一个匹配前缀的值 + 节点角落显示 ⚠️ 角标
3. 字符串内嵌 token 不支持 → 同上
4. animation 缺 @keyframes → 渲染器自动在 head 注入常用 keyframe（spin/fadeIn/slideUp）
5. env(safe-area-inset-*) 在编辑画布 → 映射到模拟值（top:20 / bottom:20）

加 query/canvas_render_status { screenId }：
返回 [
  { nodeId, kind: "broken-image" | "token-fail" | "overlap" | "overflow", message, severity },
  ...
]

design-planner SKILL.md §6 出场门禁加 1 项：
□ canvas_render_status 返回的 severity:error 列表为空
（这不是"加事后截图任务"——是阶段出场契约从 schema 完整性升级到"画布完整性"）
```

**效果**：design 标 phase=designed 之前，自动跑画布检查 → 发现 BrandLogo broken / BgBlob token 失败 → 拒推进 → AI 必须真把视觉做对才能 handover。

### 根源 3：MCP 工具是"盲写工具"，缺反馈循环

#### 为什么会发生

- 真实设计师是"画 → 看 → 调 → 看"循环
- AI 设计师调 `style/batch_update` 写 33 节点 → 标 D-styles done → 走完所有任务 → 用户截图发现 broken
- **写和看完全解耦**，AI 没有反馈直到流水线终点
- 即使加"截图任务"，也是事后预防，不是事前内建

#### 第一性原理

> 设计本质是**视觉反馈回路**，不是线性流水线。任何把"反馈"延迟到流水线末尾的设计，都违反了设计活动的本质。**反馈必须内建到每个写入操作**，而不是作为独立任务事后跑。

#### 治理方向

**MCP 工具"应用即返回预览" 改造**：

```ts
// 当前
style/batch_update { projectId, updates } → { success, affectedNodeIds }

// 改造后
style/batch_update { projectId, updates, returnPreview?: 'before-after' | 'after' | 'none' }
  → {
      success,
      affectedNodeIds,
      preview?: {
        beforeUrl?: 'snapshots/.../seq-N-before.png',
        afterUrl: 'snapshots/.../seq-N-after.png',
        renderWarnings: [{ nodeId, kind, message }, ...]
      }
    }

returnPreview 默认 'after'。AI/设计师 写一组样式 → 立刻拿到预览 URL + 渲染警告。
```

**效果**：
- AI 写 SubmitBtn styles 后，下一行就能 Read 预览图，"哦 backgroundColor 没生效，这个 token 不对"
- 不需要等到 D-handover 才发现
- 配合根源 1 的契约前置（写入时拒非法 token），AI 几乎不会留下"事后截图发现 broken"的产物

### 根源 4：流水线时序假设错位（最深层认知错误）

#### 为什么会发生

当前流水线设计：
```
product → theme → interaction → design → executor
（节点骨架）（token）（行为）   （视觉）  （素材）
```

这个排序的假设：**每一棒都能在前一棒的产物上独立完成自己的工作**。

但视觉设计活动的本质，**需要看到素材 + 看到行为 + 看到节点 同时存在的画面才能调试**：
- 设计师调 SubmitBtn shadow，需要同时看到按钮 hover 时的 spinner 出现
- 设计师调 装饰 blob 位置，需要同时看到背景色 + 卡片
- 设计师调字号梯度，需要同时看到所有文字节点的对比

把"视觉"抽到一个独立棒次（第 4 棒），假设它能不看素材完成—— **是把"线性流水线"思维强加到"反馈回路"任务上**。

#### 第一性原理

> 流水线适合"每一步是独立可验证的产物"——如编译。设计/视觉本质上**所有元素互相影响**，强行线性化会让中间产物"不可独立验证"。
>
> **正确的角色分工不应该是"线性时序"，而应该是"职责分层 + 同时可见"**：每个阶段写自己的字段，但所有字段在每一刻都"组合渲染"，每个角色都能看到当前画面。

#### 治理方向

**重新定义"阶段"语义：从"时序流水线"改"职责分层 + 共享画布"**

```
当前模型：
  product 完成 → theme 开始 → interaction 开始 → design 开始 → executor 开始
  （时间序列，每棒接力）

改造后模型：
  product 角色：随时可写 meta.product / 节点骨架 / dataSource
  theme 角色：随时可写 themeConfig.tokens
  interaction 角色：随时可写 events / bind / state.view
  design 角色：随时可写 styles / visualState / materialSpec
  executor 角色：随时可应用素材 / 截图

每次任意角色写入：
  → 自动跑画布渲染
  → 返回 当前画面 + warnings
  → 角色之间通过 plan tasks + UpstreamChallenge 协调"什么时候算 ready"

阶段不是时序门，是"完成度的快照"：
  phase=analyzed: meta.product 完整
  phase=interaction-defined: events 全
  phase=designed: styles 全
  phase=verified: 素材应用 + 渲染无 warning

任何角色都可以在任何时间补字段——只要不动其他角色已写的就行。
```

**效果**：
- design 角色写 styles 时，executor 已经先把 BgBlob 装饰用 css-gradient 渲染上 → design 写完立刻能看
- design 写 SubmitBtn shadow 时，interaction 已经把 loading 态绑好 → design 能看到 loading 切换效果
- 每个角色不需要"等待上游"—— UpstreamChallenge 协议解决冲突
- **设计本质上的反馈回路得以保留**，不被流水线时序破坏

### 根源 5：技能/契约的"事后检测"思维（元层面）

#### 为什么会发生

观察本次出场门禁清单：
```
□ 所有屏 phase = "designed"
□ 每节点 styles 已写 + visualStates 矩阵完整
□ 需要素材节点 materialSpec 完整
□ 跨屏一致性 audit 通过
□ $token: 引用率 ≥ 95%
□ 所有 plan 任务 status ∈ {done, skipped}
□ query/integrity 0 个 R-* 错误
□ 每个 done 任务的 md 已存在
```

**全部是"对 schema 的检查"，没有一项是"对画面的检查"**。技能默认"schema 完整 = 视觉正确"，但这个等价关系**就是出 bug 的地方**。

#### 第一性原理

> 检测分两类：
> - **事前**：让 bug 不可能发生（contract → 写入时拒）
> - **事后**：让 bug 发生后被发现（test → 跑完后查）
>
> 事前 > 事后，因为事后只能"发现某些 bug"，无法发现没人想到去测的 bug；事前能让一整类 bug 永远不出现。
>
> 当前所有红线（R-STRUCTURE-02 / R-VISUALSTATE-01 / R-TOKEN-COVERAGE 等）都是**事后红线**——它们告诉 AI "你写完后还差什么"。但根源 1-4 说的问题，**事后红线根本检测不到**：
>
> - R-TOKEN-COVERAGE 95% 通过 → 但深层 token 引用渲染失败 ❌
> - R-STRUCTURE-02 通过 → 但 broken image 没有红线管 ❌
> - R-VISUALSTATE-01 通过 → 但 visualState 在编辑器画布默认显示哪个态没人管 ❌
>
> **事后红线是"检查作业"，事前契约是"不让你写错"**。

#### 治理方向

**把红线从"integrity checker 事后扫"前置到"MCP 写入时拒"**

```
当前 R-* 红线 → checker 跑完看错误
改造后 → MCP 工具入参 validator + service 写入时校验

例：R-STRUCTURE-02（styles 用硬编码颜色）：
  当前：integrity checker 扫 styles 找硬编码 → 报警告
  改造：style/update 入参 validator
        if (containsHardcodedColor(value) && !isWhitelistedException(value))
          throw 422 { message: "color 必须 token 引用，硬编码 #FF6F91 不允许" }
        ↓
  AI 调 style/update 时立刻被拒，连 schema 都进不去

例：R-VISUALSTATE-01（按钮缺 disabled 态）：
  当前：checker 扫节点 visualStates → 报警告
  改造：节点标 phase=designed 时 service 强制校验
        type=button 节点 → 必须有 disabled state ∈ states
        否则 update_plan_task 拒掉

R-TOKEN-COVERAGE 95%：
  当前：扫所有 styles 看引用率
  改造：根本不需要 95%——所有非法 token 引用在 style/update 入参就被拒
        实际引用率自动 100%（剔除合理偏差列表）
```

**效果**：80% 的事后红线**消失**——因为它们想检测的 bug 已经在写入时被拒。剩下的 20% 真的需要全局视角的（如跨屏一致性 audit）才保留为事后。

## 4. 怎么验证这套方案有效？

不要靠"再做一个 demo 看看好不好"——**继续走 P0/P1，然后**：

1. 建一个最小 `design-render-roundtrip-test` 包：
   - 内含 N 个故意有 bug 的 schema（深层 token / 字符串内嵌 / type=img 空 src / 缺 @keyframes / overlapping styles）
   - 跑 P0 改造后的 MCP/Service/Renderer，期望：每个 bug schema 在"写入"或"出场"阶段被拒
   - **不看截图**——看的是"是否在写入时被拒掉了"

2. 反向验证：拿当前这次 demo 的 schema，喂给 P0 改造后的 MCP
   - 期望：~6/8 个本次发现的视觉 bug 在"写入时"就被拒
   - 剩下 2 个（如 GetCodeBtn calc 公式 / ModeToggle button reset）由 renderer fallback 兜底显示警告

3. 如果以上两条都达成 → 证明根治成功；不达成 → 找出哪类 bug 仍漏，回到根源 1-5 看缺哪一层

---

## 5. 不做什么（避免治错方向）

明确拒绝以下"改进"——它们都是事后预防伪装的事前治理：

- ❌ "在 SKILL.md 加 D-X-snapshot 任务" — 事后预防，AI 还是会忘
- ❌ "在 references/ 加 render-capabilities.md / fallback-patterns.md" — 加一层文档，再加一层漂移
- ❌ "在 D-handover 加截图核对清单" — 同上
- ❌ "让 AI 每次写完 styles 主动调 generate_snapshots 自查" — 把人能想到的事写进 prompt，下次还有想不到的
- ❌ "训练模型让它学会自动写正确的 token 语法" — 模型再聪明，文档/代码漂移它无法预知

事前治理的判定标准：**改完后，类似的 bug 不再依赖任何人/AI 的自觉就不会出现。**

## 6. 行动呼吁

P0 的 5 项里 (1)(2)(5) 是改 design-mcp + 后端 service，(3)(4) 是改前端 SchemaRenderer 和 MCP。所有 5 项都是"代码改造"，不是"文档/技能改造"。
