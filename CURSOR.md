# Agent 约束（本仓库）

## 红线：素材与画布能力必须通过 MCP 验证

凡涉及 **素材编辑器 / 画布几何 / 导出应用到设计节点** 的工作，**必须**走 **design-mcp**（如 `canvas`、`material_project`、`material_slot`、`execute_operations_batch` 等工具）以及 **design-api 上正式的素材/设计操作接口**，以便与真实产品链路、操作日志、协作行为一致。

**禁止「绕过 MCP」**，包括但不限于：

- 为求快直接向 `apps/design-api/uploads/`（或任意静态资源目录）**手写 / 脚本生成** SVG、PNG 等，再手动把 URL 填进 schema，而**不**经过 MCP 驱动的素材工程与导出流程；
- 用一次性 `curl`/脚本只打 HTTP、却**不**通过 MCP 工具编排，来冒充「已用素材能力测过」；
- 任何等价于**跳过素材画布与 MCP 工具链**、使测试结果无法代表 MCP 与编辑器真实行为的捷径。

若 MCP 能力不足或阻塞，应 **先说明限制**，再讨论改 MCP、改 API 或拆任务；**不得**用文件落盘代替 MCP 完成本属素材/MCP 测试范围的需求。

## 红线：设计节点 ↔ 素材工程 **必须走素材槽**（严禁「只 apply 样式」冒充已绑定）

**产品约定**：要让某个 div 在**设计编辑器里**出现 **「设计素材…」→「+ 添加素材…」** 这条入口，并在槽位维度可追踪、可换绑、与协作/撤销一致，**必须**在 **`node_material_slots`（素材槽）** 上建立 **nodeId + materialProjectId + cssTarget** 等绑定。

**禁止**作为「完成绑定」的唯一手段：

- 仅调用 `applyMaterialDesign`（或等价）往节点上写 `materialProjectId`、`backgroundImage` 等样式，却 **不** 通过 **`material_slot` MCP**（`create` / `list_by_node` / `update`）或 **等价前端操作** 写入/对齐槽位；
- 任何让「右键没有正确素材上下文、或槽位与设计器脱节」的捷径。

**MCP 侧推荐顺序**（与 `canvas.export_and_apply` 注释一致）：

1. 需要「导出 PNG + 应用到节点 + 槽位」时，**优先** `canvas` → **`export_and_apply`**（实现里会调 `ensureMaterialNodeBinding` 写槽位）；  
2. 若只做槽位 CRUD、不导出：用 **`material_slot`** 工具（`list_by_node` / `create` / `update` / `delete`），**核对** `cssTarget`（如 `background-image`、`border-image`）与节点样式策略一致。

**编辑器侧对应操作（与下图一致）**：在画布上选中目标节点 → **右键「设计素材…」→「+ 添加素材…」**（或已有子菜单中的槽位管理路径），**不要**假设只改 JSON 样式即等价于走通产品链路。

![设计编辑器：右键菜单必须为「设计素材…」→「+ 添加素材…」以建立素材槽](./assets/cursor/material-slot-context-menu.png)

> 若视觉效果与参考稿仍有差距（例如霓虹外晕、透明度 taper），属于 **素材 primitive / 导出渲染** 与 **CSS 合成** 的能力边界，须在专项里迭代；**槽位与入口**仍必须按本条执行，不得省略。

## 红线：槽位 `cssTarget` 与样式写入一致（禁止在应用路径里「偷改槽类型」）

**约定**：`node_material_slots.cssTarget` 表示该素材工程绑定到节点的 **哪一条 CSS 通道**（如 `background-image`、`border-image`、`mask-image`）。**应用 / 导出**（前端「应用到元素」、`canvas.export_and_apply`、`applyMaterialDesign` 等）**只按当前槽位的 `cssTarget` 写入对应样式**，不得在同一流程里把槽位从 `border-image` 改成 `background-image` 或反向「帮用户纠正」。

**若产品策略变更**（例如统一弃用某一类槽位）：**只允许**通过 **显式数据迁移**（一次性 SQL/脚本、`material_slot` MCP 的 `update`、或专用 migration 任务）改库；**禁止**在常规编辑器/MCP 业务代码里夹带隐式槽位改写。

**禁止「删槽又自动长回来」**：不得在 **`preloadMaterialSlots`**（或任何高频预加载路径）里在 **槽位列表为空** 时，根据 `material-projects/all-by-node` 等旧 **`target_node_id`** 关联 **批量 `POST` 建槽**。用户删除槽位后必须保持为空，直到用户或迁移脚本显式再建。

## 何时应从「产品」角度完善 MCP / 素材编辑器

从第一性原理看，素材编辑器 + MCP 的目标是：**把设计意图稳定、可复用地映射到可导出资产与操作日志**。当出现下面一类信号时，应优先 **产品化**（编辑器、schema、MCP、导出一致），而不是长期用脚本或手写资源代偿：

1. **共性**：同一 **类** 视觉/交互在多处重复出现，且用现有 primitive **组合成本过高**（条数多、难改、易分叉）。  
2. **意图与表达落差大**：人描述的是 **少量连续或结构化参数**（沿某维度的变化、锚点、场），落库却变成 **大量离散图元**——说明 **抽象层级不对**。  
3. **MCP 测真链路**：若只有绕过 MCP 才能交付，说明链路上缺能力，应进 backlog，用 **一等公民参数与操作类型** 补全，否则测试无效。  
4. **可逆与协作**：需要版本、撤销、协同、槽位绑定的，必须落在 **正式素材/设计操作** 上，而不是一次性文件。  
5. **成本结构**：设计师与 Agent 反复手工对齐仍不稳时，应 **抬抽象**（参数化对象、模板、检查器），而不是继续堆个案。

**落地节奏**：先写清 gap（意图 vs 当前表达能力），再拆 MCP、编辑器 UI、导出/预览，**避免只改一端**。

## 第一性原理（抽象，不写具体方案进本文件）

很多「装饰性、非矩形」需求，本质是：**在某种载体几何上，对一个或多个外观量随位置变化**——载体可以是曲线、曲面、闭合轮廓或区域；外观量可以是 **不透明度、线宽、色相、模糊** 等及其组合。设计师常通过 **少量锚点 / 色标 / 预设** 控制这些场，而不是逐段手搓图元。

**复用**来自识别 **同一类问题**（沿路径/沿角度的场 + 载体模板），在编辑器里提供 **与意图同维的控件与数据模型**；具体某一种 UI（例如某一屏的光圈）的命名、字段表、验收标准，属于 **产品 spec / ADR / issue**，**不要写进 `agents.md`**，以免把「红线与元规则」和「某次讨论的结论」绑死在一起。

**发散方向**（仅提示思考维度，不定案）：场是一维还是二维；是否需要 **与现有渐变/矢量工具心智对齐**；导出与画布 **是否同一套参数**；预设与完全自定义的边界；与页面结构 **分层**（背景场 vs 内容树）等——均在专项文档里展开。
