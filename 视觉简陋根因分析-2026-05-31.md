# 视觉简陋根因分析 v2 — 二元分类（技能问题 vs 功能障碍）

> v1 写得云里雾里。v2 重写：直接给清楚每条**是技能问题（改 SKILL）还是功能障碍（改代码）**，每条给具体落点。
>
> 前提验证：登录页 schema `d84c140e-0437-4c80-a786-c1f389bcbb02`（实测自 2026-05-31 19:41 截图）。

---

## 0. 一段话核心结论

`.claude/skills/design-planner/SKILL.md §1` 角色定位写的是：
> **"资深企业级 UI/视觉设计师。你不是写 CSS 的，你是给整个产品定调、定层级、定氛围、定品牌的设计师。"**

但 §2 / §3.1 / §5.4 / Phase 1 任务清单 / methodology 实际把它**降级**成了「写绘制规格的对账员」：

| 应该怎么做（角色定位写的） | SKILL 实际怎么教（具体段落） |
|---|---|
| 像设计师一样画完整的页面 | §2 末段 + §5.4：素材最终绘制 = executor 写 `materialProjectId`；design 严禁触碰 |
| 逐组件逐节点精雕 | Phase 1 任务清单 D-X-styles 一棒扫所有节点 styles，没有"主角先深雕→配角再雕"分级 |
| 做出有品牌识别度的页面 | methodology/02 把 budget 教成"视觉资源是稀缺品，必须省着用"——工具节点 allowedTools = 极少 |
| 视觉态完备 | methodology/06 的 visualStates 矩阵只有 DOM 事件态（hover/pressed/focus/disabled），没有业务态（active/checked/selected） |

**AI 严格按当前 SKILL 跑，跑出来就是这个简陋画面**——这不是 AI 偷懒，是 SKILL 自相矛盾。

**分类**：

- **A 类 = 技能问题（85%丑度）** → 改 `.claude/skills/design-planner/` 文档，零代码改动
- **B 类 = 功能性障碍（15%丑度）** → 改 `apps/design-mcp/` + `features/design-engine/` 代码

下面 §1（A 类）/ §2（B 类）每条都给：现象 → 具体段落证据 → 怎么改。

## 1. A 类 — 技能问题（改 SKILL，零代码改动）

### A1. 「素材绘制」职责给错了棒次（最大根因）

#### 现象
登录页 BrandLogo 是 type=img 空 src → 截图显示虚线占位框 + "Logo" 文字。design-planner 把它标 done 了。

#### 段落证据
- `SKILL.md §2` 第 41 行：
  > 下游 executor 只负责：照 materialSpec 画 PNG → 应用素材 → 截图核对 → 终验。**executor 不做任何设计决策**——所以你这一步必须把规格写到位。

- `SKILL.md §5.4` 阶段边界红线（第 363 行）：
  > `node.materialProjectId` | ⛔ executor 写（素材上传产物）

- Phase 1 任务清单 `D-X-materials` 标题：
  > 素材规格 materialSpec（kind / 风格 4 维度 / colorStrategy / layers / qualityChecklist）

#### 矛盾点
- §1 角色 = "资深企业级设计师"
- §2 / §5.4 = "你只写规格，画图归 executor"

**真实世界的资深设计师交付物 = 完整画好的素材**。SKILL 把这一步切给 executor，但 executor 又被 §2 定义为"不做任何设计决策"——结果素材成了无主之地：design 阶段没画、executor 阶段又没视觉判断力。**这是当前 demo Logo 是占位框、Tab 没 icon、Checkbox 是黑方块的直接原因**。

#### 怎么改
**a) 重写 SKILL.md §2 流水线分工**：

```diff
- product → theme → interaction → design-planner → design-executor
- design-planner 写规格；executor 照规格画素材 + 截图终验
+ product → theme → interaction → design-planner（含素材绘制）→ design-executor（仅终验 + 截图）
+ design-planner 是设计师本人——逐节点精雕 styles + visualStates + 调 material-painter 画完所有素材
+ design-executor 退化为「QA 摄影师」——跑 generate_snapshots、对比 design 期望、报告差异
```

**b) 拆 D-X-materials 任务为两个**：

```diff
- D-X-materials  「素材规格 materialSpec（写规格）」
+ D-X-material-spec  「写 materialSpec 规格」  ← 现有任务
+ D-X-material-paint 「按 spec 调 material-painter 画素材 + applyMaterialDesign 应用」
   expectedArtifacts: [
     { kind: 'eachItem', path: 'rootNode.descendants[type=img|kind=brand|kind=icon]',
       check: { kind: 'nonEmpty', path: '$.materialProjectId' } }
   ]
```

**c) 修订 §5.4 阶段边界**：把 `node.materialProjectId` 从「⛔ executor 写」改为「✅ design 写，executor 只读 + 终验」。

**d) Phase 2 工作流**加一条：调用 material-painter 子技能（material-painter SKILL.md 已存在，工具链完整）。

#### 影响
单条修复消除：Logo 占位框、Tab/Checkbox 缺 icon、所有装饰图缺图。**截图丑度 ~25% 在这条**。

### A2. budget 教反了（教成"省着用"，不是"专业统筹"）

#### 现象
登录页 PolicyCheckbox `weight=1` `allowedTools=["边框","勾选"]` → AI 写出来 = 18×18 native input + accentColor —— 看见的就是黑方块。FooterLinks `weight=1` `allowedTools=["字色"]` → 灰色字两个，几乎看不见。

#### 段落证据
`references/methodology/02-visual-budget.md` 第 3 行：
> 这是**企业级 UI 设计**与"个人设计师补样式"的核心分水岭——**视觉资源是稀缺品，必须像项目预算一样统筹分配**。

`02-visual-budget.md §2` 权重表：
> | 3-4 | 工具-输入 | PhoneInput / Switch / Checkbox | 边框 + 聚焦光 | 极少 |
> | 2-3 | 工具-导航 | TabBar / 链接 / 返回按钮 | 文字色 + 微图标 | 极少 |

**问题**：把 budget 当"省着用的预算"而不是"专业统筹"。"weight 1-3" 的 allowedTools 写得太抠 —— 真实大厂登录页的 checkbox 都有自绘边框 + 选中态底色 + 勾 + 焦点光晕（4 个手段），不是只有 accentColor。

#### 怎么改
**重写 `02-visual-budget.md` 第 1-2 节**，把"省着用"改成"统筹分配 + 最低识别阈值"：

```diff
- 视觉资源是稀缺品，必须像项目预算一样统筹分配
+ budget 是「视觉权重」不是「装饰量上限」。
+ 主角组件高密度装饰让它跳出来，工具组件低密度装饰但仍必须达到「最低识别阈值」。
+ 每个 role 都有最低识别阈值（见 §5），低于阈值 = 设计未完成。
```

新加 `§5 最低识别阈值表`：

```yaml
roleMinSignals:
  工具-勾选(checkbox/radio):
    必须: ≥3 个手段
    清单: [自绘边框, 选中态底色, 勾(SVG path 或 unicode), 焦点光晕]
    禁止: 只用 accentColor
  工具-切换(tab/segment):
    必须: ≥2 个手段区分 active vs inactive
    清单: [字色, 字重, 下划线/底色, 移动指示条]
    禁止: 只静态 borderBottom
  工具-辅助(secondary CTA):
    必须: ≥2 个手段表达可点击
    清单: [边框 OR 底色, 箭头/外链 icon, hover 反馈]
    禁止: 只字色
  主角-品牌(Logo/Hero):
    必须: ≥1 个 ≥80px 的视觉锚（图/字/形状）+ 主色 OR 主色衍生
  氛围-装饰:
    必须: 实测渲染对比度 ≥ 0.05 vs 背景（不能写完是不可见的）
  ... (按 §2 的角色表逐项给阈值)
```

**配套**：D-X-budget 任务的 expectedArtifacts 加一条 `kind: minSignalDelta` 检查（B2 提供工具，§2.B2）。

#### 影响
**这条单独改约消除 50% 截图丑度**。是 7 项里 ROI 最高的。

### A3. visualState 矩阵漏「业务态」（只看 DOM 事件态）

#### 现象
登录页 ModeToggle 两 tab `CodeModeBtn / PasswordModeBtn` 各有 4 个 state（hover/pressed/focus/disabled），**就是没 active**。截图里两 tab 字色完全一样，看不出哪个被选中。但 interaction 阶段早就写了 `state.view.activeMode: "code"|"password"`——只是 design 阶段没把它映射到视觉。

#### 段落证据
`references/methodology/06-visualstates-completeness.md` （应有的内容，让我们查实际）：

实际 D-X-states 任务清单：
> visualStates 完备态（按 variant × state 矩阵 + childrenStates / activeWhen）

但矩阵的"state"列举只有 DOM 事件态（hover/pressed/focus/disabled/loading）——**没有业务态来源约束**。

interaction-designer 阶段已经在 schema 里写了 `state.view.activeMode/policyAccepted/passwordVisible/submitting/lockedUntil/codeCountdown` 等业务字段，design 阶段应该扫这些字段、给每个字段对应的视觉态——但 SKILL 没教这个映射动作。

#### 怎么改
**重写 `methodology/06-visualstates-completeness.md`** 加一节：

```markdown
## 业务态来源（与 DOM 事件态平级）

DOM 事件态：hover / pressed / focus / disabled / loading
业务态来源：扫 screen.stateInit.view + project.stateInit.view 所有 boolean/enum 字段

每个 boolean/enum 字段必须问：
1. 谁的视觉应该反映这个字段变化？（节点 ID 列表）
2. 这些节点的对应 visualState 名是什么？（active / checked / selected / busy / locked / countdown ...）
3. activeWhen 表达式怎么写？（{{ state.view.X === 'Y' }}）

D-X-states 任务的 expectedArtifacts 必须含：
- 每个非临时性 state.view 字段都对应到至少一个节点的 visualState
- 该 visualState 必须有 ≥1 个 distinct style override（不能空建）
```

**SKILL 配套**：interaction-designer SKILL 出场契约加一个新字段 `screen.meta.interaction.stateVisualMap`，写明每个 state.view 字段应触发哪些节点哪些 visualState。design-planner Phase 1 必读这个字段。

#### 影响
消除：Tab 没 active 视觉、Checkbox 没 checked 区分、表单没 submitting 反馈、PolicyText 没 error 高亮、PasswordToggleEye 没 visible 切换。**截图丑度 ~15%**。

### A4. D-X-styles 是「平铺扫」不是「逐节点深雕」

#### 现象
当前任务清单只有一个 D-X-styles：「全量样式落库（每节点 style/update 一次到位）」。AI 倾向于一遍扫完所有节点 → 每个节点拿到 5-10 行 styles → 没机会反复推敲。

你说"逐个页面逐个组件逐个节点细节逐步像一个设计师一样去思考"——SKILL 没有这个流程。

#### 段落证据
Phase 1 第 192 行：
```
{ id: "D-X-styles", title: "全量样式落库（每节点 style/update 一次到位 + 全 $token: 引用）",
  expectedArtifacts: [{ kind: 'nonEmpty', path: 'rootNode.styles' }] }
```

**问题**：一个任务管整屏几十节点，expectedArtifacts 只看 rootNode.styles 非空——**写完 1 个节点都算 done**。

#### 怎么改
**Phase 1 把 D-X-styles 拆成 4 个深雕任务**：

```diff
- D-X-styles  「全量样式落库」
+ D-X-styles-hero       「主角节点深雕（主 CTA + 品牌 Logo + 大标题）—— 每节点 ≥ 8 个 style 属性 + ≥1 个出彩手段（渐变/微动效/光晕/纹理）」
+ D-X-styles-form       「表单组件深雕（input/checkbox/select/button 类）—— 每节点必含自绘外观（不依赖 native 默认）+ 完整 visualStates」
+ D-X-styles-decoration 「装饰节点深雕（背景氛围/角落溢出/分割装饰/品牌点缀）—— 每节点必有可见 contrast，禁止字符串内嵌 token」
+ D-X-styles-rest       「剩余节点（容器/小工具/footer）—— 每节点必达 §A2 最低识别阈值」
+ 四个任务必须按顺序做，前一个 done 才能开下一个（dependsOn）
```

每任务 expectedArtifacts 加 `eachNode` 检查，确保覆盖率 + 深度。

#### 影响
迫使 AI 真的逐节点思考，不再一遍扫完了事。

### A5. 缺「Web 真实能力 / 陷阱清单」reference

#### 现象
PolicyCheckbox 写成 `<input type=checkbox>` + `accentColor:primary` —— AI 不知道 native checkbox 的未选中态浏览器自渲染、accentColor 只染勾不染框。

#### 段落证据
`references/` 下的清单（§12 列了 22 个文件）—— **没有任何一个讲"哪些 HTML 元素不能深度自定义"**。

#### 怎么改
**新增 `references/web-rendering-pitfalls.md`**（机器可读 + 替换方案）：

```yaml
nativeUncustomizable:
  - tag: input[type=checkbox]
    canStyle: [width, height, margin, accentColor(只染选中勾)]
    cannotStyle: [borderRadius, backgroundColor 未选中态, customCheckMark]
    workaround: |
      <label>
        <input type=checkbox style="display:none" bind="..."/>
        <span class=checkbox-visual>
          [styles: 18x18 / border-2 / radius-sm / 默认透明]
          [visualState checked: backgroundColor primary + ::after 自绘勾]
          [visualState focus: outline 2px primaryLight]
        </span>
        <span>文本</span>
      </label>

  - tag: input[type=radio]
    workaround: 同上 + ::after 圆点
  - tag: select
    cannotStyle: [箭头, options 列表]
    workaround: appearance:none + 自绘 chevron icon + 自定义 options 浮层（用 overlays）
  - tag: input[type=file]
    cannotStyle: [按钮文案/尺寸]
    workaround: <label for=hidden-input> 自绘按钮
  - tag: input[type=range/date/color/time]
    cannotStyle: [跨浏览器外观]
    workaround: 用第三方 / 自绘 + state.view 状态托管
```

**新增 `references/composite-patterns.md`**（业务复合控件清单）：
- Tab / Segment：active 强制视觉
- Stepper：当前步 / 已完成 / 未到达三态
- Accordion：展开 / 收起态
- Pagination：当前页 / hover / disabled
- Breadcrumb：当前 / 链接 / 分隔符
- Toast / Snackbar：四类语义色 + 进出动效
- Drawer / Sheet / Modal：进出动效 + backdrop
- Empty / Skeleton / Error：视觉 = 业务而非装饰

**SKILL §12 加这两个 reference + Phase 2 必读**。

#### 影响
消除 native 控件类的设计认知失误。

### A6. 缺「设计自审 → 重做」回路

#### 现象
当前 D-X-styles 标 done = schema 完整。**AI 永远没有"看一眼自己写的画面"的环节**。

#### 段落证据
Phase 2 第 250-275 行：
```
1. query/next_pending_task → T
2. update_plan_task → doing
3. read_file（template + methodology + schema-spec）
4. query/screen_schema → 看 interaction 衍生节点是否齐
5. 写 md
6. 落 schema（MCP 调用）
7. ★ 沉淀到 schema 的结论
8. update_plan_task → done
```

**第 4 步看 schema、第 5/6 步写**——没有任何步骤要求"调 generate_snapshots 看自己写的实际画面"。

#### 怎么改
**Phase 2 加 3 个步骤**：

```diff
6. 落 schema（MCP 调用）
+ 6a. ★ generate_snapshots 自截当前屏 + read 截图
+ 6b. ★ 在 md 末尾加「自审段」：识别度评分（5 维度：品牌 / 优先级 / 状态 / 情绪 / 完整度），每维度 ≥4/5 才能 done
+ 6c. ★ 不达标 → 不调 update_plan_task done，回 6 重做
7. ★ 沉淀到 schema 的结论
```

**配套** Phase 0 入场门禁加：检查 D-X-styles-* / D-X-states 等任务的 md 是否含「自审段」。

#### 影响
封堵"卷子答完没看画"的反馈缺口。是 v1 文档根因 ⑥ 的具体落点。

## 2. B 类 — 功能性障碍（必须改代码）

### B1. MCP `validateStyles` 漏拒「字符串内嵌 token」

#### 现象
登录页 BgBlobTopRight `background: "radial-gradient(circle at 50% 50%, $token:colors.primaryLight 0%, transparent 70%)"` 写进了 schema，渲染器原样输出，浏览器看到字面量字符串 → CSS parse 失败 → 装饰光斑完全消失。

#### 段落证据
`apps/design-mcp/src/tools/helpers/validateStyles.ts` 当前只检查"独立属性是不是合法 token"。

#### 怎么改
```ts
// validateStyles.ts 加：
function detectStringEmbeddedToken(prop: string, value: unknown): TokenIssue | null {
  if (typeof value !== 'string') return null;
  const matches = value.match(/\$token:[\w.\-]+/g);
  if (!matches) return null;
  // 整串 = 单个 token 是合法（resolver 处理）；其它情况都是字符串内嵌
  if (matches.length === 1 && value.trim() === matches[0]) return null;
  return {
    kind: 'string-embedded-token',
    prop, value, foundTokens: matches,
    message: `字符串内嵌 token "${matches[0]}" 不会被解析。建议拆字段：background-color: $token:... + background-image: "radial-gradient(..., var(--bg-color), ...)"；或在 themeConfig 里把整段 gradient 注册为 token`
  };
}
```

#### 工作量 / ROI
半天。消除一类 AI 反复踩的坑。

### B2. 缺「画面对账」MCP 工具集

#### 现象
A2/A3/A4/A6 的 expectedArtifacts 都需要"看渲染后的画面"才能机器对账：
- A2 `minSignalDelta` 要看节点视觉差异
- A3 stateVisualMap 覆盖要看 visualState 是否有 distinct override
- A4 styles-hero 深雕要看节点是不是 ≥8 个属性 ≥1 个出彩手段
- A6 自审段要看实际渲染截图

当前 MCP 没有这套查询。

#### 怎么改
新增 4 个 MCP query（与治理方案 P0-5 `canvas_render_status` 合并实现）：

```typescript
query/canvas_render_status { screenId } → [
  { nodeId, kind: 'broken-image' | 'token-fail' | 'string-embedded-token-fail' | 'overflow', severity, message }
]

query/canvas_overlap_audit { screenId } → [
  { nodeId, kind: 'overlap', target, overlapBox, severity }
]

query/canvas_recognition_audit { screenId, role? } → [
  { nodeId, role, signalCount, threshold, missing: [...], severity }
]

query/visual_state_distinctness { screenId } → [
  { nodeId, stateName, distinctOverrideCount, severity: stateName==='default' ? 'info' : 'error' }
]
```

实现路径：用 jsdom + features/design-engine 的 SchemaRenderer 跑无头渲染 → 取 getBoundingClientRect / getComputedStyle → 对账规则。

#### 工作量 / ROI
3-5 天。是 A 类 7 条改造的"机器对账底座"——没这个工具，A 类 expectedArtifacts 没法落地。

### B3. material 后端绘制服务的 3 个 invariant bug

#### 现象
