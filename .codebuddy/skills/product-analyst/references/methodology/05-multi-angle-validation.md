# 多角度验证（4 维度）

每个**关键决策**必须从至少 3 个角度过一遍。结论必须落到 md 的"推理过程"段——schema 只装最终结论。

## 4 个验证维度

| 维度 | 核心问题 |
|------|---------|
| **用户角度** | 用户真的需要吗？使用频率？替代方案？|
| **商业角度** | 对留存 / 转化 / 营收的贡献？ROI？|
| **技术角度** | 实现成本？技术风险？架构影响？|
| **竞品角度** | 竞品怎么做的？差异化在哪？取什么值？|

## 标准模板（md 中使用）

```markdown
### 多角度验证：<决策点>

| 维度 | 结论 |
|------|------|
| 用户角度 | 频次 / 痛点 / 替代方案分析 ... |
| 商业角度 | 留存 / 转化 / ROI 预期 ... |
| 技术角度 | 实现成本 / 技术风险 ... |
| 竞品角度 | 主要竞品 + 取值依据 ... |
```

## 实战例子

```markdown
### 多角度验证："密码错 5 次锁 30 分钟"

| 维度 | 结论 |
|------|------|
| 用户角度 | 5 次容错对正常用户够；30 分钟惩罚不至于无法找回 |
| 商业角度 | 防止暴力破解 / 减少账号被盗的客诉 |
| 技术角度 | failureCount / lockedUntil 两个 view 变量即可实现，复杂度低 |
| 竞品角度 | 微信 5 次 / 钉钉 5 次 / 飞书 3 次（取中位数 5） |

### 替代方案与否决

- 方案 A：渐进锁定（1 次 1 分钟、3 次 5 分钟、5 次 30 分钟）
  → 否决：复杂度收益比低，不值得为这个概率事件做精细化
- 方案 B：人脸识别解锁
  → 否决：本期不接活体检测，技术成本与产品价值不匹配
```

## 适用场景

每个以下决策都建议跑 4 维度验证：
- 业务规则的具体阈值（5 次？10 次？30 分钟？60 分钟？）
- 模块切分（这个功能放本期还是下期？）
- 信息架构（用 Tab 还是抽屉？）
- 风格方向（青春治愈 vs 商务专业）
- API 设计取舍（一次拉全 vs 分页拉）

## 不适用场景

- 没有显著选择空间的决策（如"按钮要可点击"——这不是决策）
- 工程实现细节（应该让工程师拍板）
