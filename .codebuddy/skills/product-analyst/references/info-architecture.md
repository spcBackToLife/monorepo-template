# 信息架构设计方法

## 信息架构的本质

信息架构 = **功能如何组织为用户可理解的结构**

不是"有多少个页面"，而是"用户如何找到想要的功能"。

## 导航模式选择

### 常见导航模式

| 模式 | 适用场景 | 典型应用 | 特征 |
|------|---------|---------|------|
| **底部 Tab** | 3-5 个平级核心功能 | 微信/抖音/淘宝 | 功能平等、快速切换 |
| **侧边 Drawer** | 功能多但有明确主线 | Gmail/Reddit | 主内容为主、功能藏侧边 |
| **顶部 Tab** | 同一维度的分类切换 | 新闻分类/微博热搜 | 内容分类而非功能分类 |
| **Stack 导航** | 深层级的线性流程 | 设置/表单向导 | 深度优先、逐步深入 |
| **混合模式** | 复杂应用 | 大众点评/支付宝 | Tab + Stack + Modal |

### 选择决策树

```
Q1: 核心功能有几个？
  ≤ 5个且平级 → 底部 Tab
  > 5个 → Q2

Q2: 是否有明确的"主线功能"？
  是（如 Feed 流为主）→ 底部 Tab(3-4个) + 更多入侧边
  否（功能并列）→ 宫格首页 or Drawer

Q3: 是否有深层级流程？
  是 → Stack 嵌套在 Tab 内
  否 → 扁平结构
```

## 页面层级设计

### 层级原则

```
L0: Tab 直达页（1次点击可达）     → 最高频功能
L1: 二级页面（2次点击可达）       → 中频功能  
L2: 三级页面（3次点击可达）       → 低频功能
L3+: 更深层级                     → 极低频（设置/帮助）
```

**80/20 原则**：80% 的用户操作应在 L0-L1 完成。

### 页面分类

| 类型 | 说明 | 示例 |
|------|------|------|
| 列表页 | 展示数据集合 | Feed 流、消息列表 |
| 详情页 | 单条数据的完整展示 | 帖子详情、个人主页 |
| 表单页 | 用户输入/编辑数据 | 注册、发布、编辑 |
| 结果页 | 操作完成后的反馈 | 注册成功、支付完成 |
| 筛选页 | 缩小数据范围 | 搜索结果、分类筛选 |
| 设置页 | 系统/个人偏好 | 通知设置、隐私设置 |

## 信息架构产出格式

### 页面树

```
App
├── [Tab] 首页
│   ├── Feed 列表
│   ├── → 帖子详情
│   │   ├── → 评论列表
│   │   └── → 用户主页
│   └── → 搜索
│       └── → 搜索结果
├── [Tab] 发现
│   ├── 社团列表
│   ├── → 社团详情
│   │   └── → 加入/活动列表
│   └── 活动列表
│       └── → 活动详情
├── [Tab] 发布 (Modal)
│   └── 发布编辑器
├── [Tab] 消息
│   ├── 通知列表
│   ├── → 私信对话
│   └── → 系统通知详情
├── [Tab] 我的
│   ├── 个人主页
│   ├── → 编辑资料
│   ├── → 我的收藏
│   ├── → 设置

    └── 注册页（多步）
```

### 导航流转图

标注每个页面间的跳转方式：

```
首页 ←→ 帖子详情     (push/pop)
首页 → 搜索          (push)
任意页 → 发布        (modal present)
帖子详情 → 用户主页   (push)
用户主页 → 私信       (push)
```

## 验证方法

设计完信息架构后，用以下方式验证：

1. **任务测试**：给定常见任务（如"发布一条动态"），数到达目标需要几步
2. **反向验证**：从任何页面能否快速回到首页？（≤2步）
3. **新用户测试**：首次打开 App 能否在 10 秒内理解核心功能？
4. **极端测试**：功能列表翻倍后，当前架构是否还能承载？
