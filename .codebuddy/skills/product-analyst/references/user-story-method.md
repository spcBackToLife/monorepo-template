# 用户故事设计方法论

## 用户故事格式

### 标准格式

```
作为 [角色]，我希望 [功能/行为]，以便 [获得的价值/目的]
```

### 分类体系

| 类型 | 说明 | 示例 |
|------|------|------|
| 核心故事 | 模块存在的理由，缺少则模块无意义 | "作为用户，我希望能发布动态" |
| 扩展故事 | 提升体验但不致命 | "作为用户，我希望能编辑已发布的动态" |
| 异常故事 | 错误/边界/异常场景的处理 | "作为用户，如果网络断开，我希望草稿不丢失" |
| 管理故事 | 后台/运营/管理员视角 | "作为管理员，我希望能审核违规内容" |

### 编写原则

1. **INVEST 原则**：
   - **I**ndependent — 故事间尽量独立
   - **N**egotiable — 细节可协商
   - **V**aluable — 对用户有价值
   - **E**stimable — 可估算工作量
   - **S**mall — 足够小可在一个迭代完成
   - **T**estable — 有明确验收条件

2. **穷举不遗漏**：
   - 先列核心故事（模块的 raison d'être）
   - 再想异常情况（用户会犯什么错？）
   - 再想边界情况（极端数据？并发？权限？）
   - 最后想管理视角（运营怎么管？数据怎么看？）

3. **验收标准**（每个故事必须附带）：
   ```
   Given [前置条件]
   When [用户操作]
   Then [预期结果]
   ```

### 常见遗漏清单

每个模块分析后用此清单检查是否遗漏：

- [ ] 首次使用 vs 老用户回访的区别
- [ ] 无数据/空状态怎么展示
- [ ] 数据过多时的分页/加载更多
- [ ] 权限不足时的处理
- [ ] 网络异常/超时的处理
- [ ] 并发操作冲突的处理
- [ ] 多设备同步的处理
- [ ] 用户删除/注销后的数据处理
- [ ] 内容违规/敏感词的处理
- [ ] 国际化/多语言考虑
- [ ] 无障碍访问考虑
