# Step C: <模块名> — 业务规则（4 类齐全）★ 核心

> 详细方法见 `methodology/04-five-step-method.md` Step C。
>
> **R-PRODUCT-01**：rules ≥ 4 条，4 类各至少 1 条。
> **R-PRODUCT-03**：业务状态机字段+枚举值必须显式写出。

## 推理过程

### 1. 候选规则池（穷举）

先尽可能穷举候选规则（哪怕最后会被砍），按 4 类列出：

#### 数据规则候选
- [ ] 字段 X 长度 N 位
- [ ] 字段 Y 格式 /regex/
- [ ] 字段 Z 必填 / 默认值
- ...

#### 业务规则候选
- [ ] 流程约束：A 必须在 B 之前
- [ ] 状态转换：状态 X → 触发条件 → 状态 Y
- [ ] 权限：角色 R 才能做 X
- [ ] 计费：满足条件触发 X
- ...

#### 安全规则候选
- [ ] 防滥用：限流 / 防刷
- [ ] 合规：实名 / 年龄 / 内容审核
- [ ] 隐私：哪些数据不能展示给谁
- ...

#### 边界 Case 候选
- [ ] 网络断
- [ ] 服务挂
- [ ] 数据空 / 超长 / 异常字符
- [ ] 并发冲突 / 重复提交
- [ ] 设备能力差异（键盘遮挡 / 离线缓存 / 后台中断）
- ...

### 2. 多角度验证（关键阈值）

> 模板见 `methodology/05-multi-angle-validation.md`。

对每个**有阈值的规则**做 4 维度验证：

**例：决策"密码错 N 次锁 M 分钟"的阈值**

| 维度 | 取 N=3, M=10 | 取 N=5, M=30 | 取 N=10, M=60 |
|------|------|------|------|
| 用户角度 | 严格但易误锁 | 平衡 | 宽松但暴破风险高 |
| 商业角度 | 客诉多 | 平衡 | 安全事故多 |
| 技术角度 | 实现成本相同 | 相同 | 相同 |
| 竞品角度 | 飞书 | 微信 / 钉钉 | 少见 |

**最终选择 N=5, M=30**——理由：[...]

### 3. 替代方案与否决

每条被砍掉的候选规则，写明否决理由：
- 否决"渐进锁定"：[原因]
- 否决"人脸识别解锁"：[原因]
- 否决"邮箱注册"：[原因]
- ...

### 4. 业务状态机字段（如本屏承载状态对象）★ R-PRODUCT-03

如果本屏承载有状态业务对象（订单 / 任务 / 工单 / 审批 / 账户 / 会话），**必须**显式写清：

```
状态字段：order.status
枚举值：{ pending_payment, awaiting_shipment, shipping, completed, cancelled, refunding }
状态转换：pending_payment → awaiting_shipment（支付成功）
        awaiting_shipment → shipping（仓库发货）
        ...
```

否则 interaction 阶段不知道有几个状态视图要建。
