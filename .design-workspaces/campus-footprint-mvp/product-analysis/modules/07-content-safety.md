# 内容安全体系 — 详细设计

---

## 一、安全策略概览

校园社交产品面临的内容风险：色情/暴力/广告/诈骗/隐私泄露/霸凌。
采用「AI前置 + 人工兜底 + 用户自治」三层防线。

---

## 二、三层防线

### 第一层：AI 自动审核（发布时）

```
审核节点：内容提交 → AI审核 → 通过/拒绝/人工复审
审核内容：
├── 文字：敏感词/涉政/色情/广告/诈骗
├── 图片：色情/暴力/广告水印/人脸隐私
└── 视频：截帧审核（V2.0+）

处理策略：
├── 安全（≥95%置信度）→ 立即通过
├── 疑似（50-95%）→ 先发后审（标记待复审）
└── 高危（<50%安全）→ 拦截 + 通知用户
```

### 第二层：人工审核（异步）

- 疑似内容24小时内人工复审
- 被举报内容优先人工审核
- 审核通过/违规/误判三种结果

### 第三层：用户自治

- 举报按钮（每条动态/每个用户/每条消息）
- 举报理由：色情/广告/骚扰/虚假/其他
- 同一内容被3人举报 → 自动下架待审
- 屏蔽：屏蔽用户后互不可见

---

## 三、处罚体系

### 信用分机制

```
初始分：100分
├── 轻度违规（广告/不当言论）：-10分 + 内容删除
├── 中度违规（骚扰/虚假信息）：-20分 + 功能限制24h
├── 重度违规（色情/诈骗/霸凌）：-50分 + 封禁7天
└── 极重（涉法）：永久封禁 + 保留证据

信用分影响：
├── 80-100分：正常使用
├── 60-79分：发布需审核通过后展示（先审后发）
├── 40-59分：禁止捞人+禁止发布7天
└── <40分：永久封禁
```

### 处罚通知

- 违规时：App内通知+说明原因+申诉入口
- 封禁时：登录时弹窗提示+倒计时+申诉入口
- 申诉：24小时内回复

## 四、特殊安全场景

| 场景 | 策略 |
|------|------|
| 位置动态含真实门牌号/教室号 | AI检测+模糊化提示 |
| 定向动态骚扰 | 被定向者可「屏蔽该人的定向动态」 |
| 捞人骚扰 | 打招呼3次被忽略→24h不能再向该用户发 |
| 私信骚扰 | 敏感词过滤+被举报处罚 |
| 冒充他人 | 头像+昵称相似度检测+举报处理 |
| 批量注册 | 设备指纹+IP限流+行为检测 |

## 五、数据模型

```
ContentReport（举报）
├── id: string
├── reporter_id: string
├── target_type: enum (MOMENT | USER | MESSAGE)
├── target_id: string
├── reason: enum (PORN | AD | HARASS | FAKE | OTHER)
├── description: string?
├── status: enum (PENDING | CONFIRMED | DISMISSED)
└── created_at: timestamp

UserCreditScore（信用分）
├── user_id: string (PK)
├── score: number (0-100)
├── history: [{ event, delta, reason, timestamp }]
└── restrictions: string[] (当前限制列表)

ContentAuditLog（审核记录）
├── id: string
├── content_type: enum (MOMENT | AVATAR | MESSAGE)
├── content_id: string
├── ai_result: { safe_score, labels[] }
├── human_result: enum (PASS | REJECT | PENDING)?
├── action: enum (PASS | DELETE | RESTRICT)
└── created_at: timestamp
```

### 核心接口

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | /api/reports | 提交举报 |
| GET | /api/users/me/credit | 查看我的信用分 |
| POST | /api/appeals | 提交申诉 |
| POST | /api/audit/callback | 审核结果回调（内部） |

## 六、MVP 阶段简化方案

全量安全体系成本高，MVP 阶段采用：

- AI审核：接入第三方服务（网易易盾/阿里绿网）
- 人工审核：暂时由运营人员兼任
- 信用分：简化为3档（正常/警告/封禁）
- 举报：有举报入口，运营手动处理

---

## 七、模块间依赖

```
内容安全
├── 依赖：所有内容产生模块（动态/消息/资料）
├── 被依赖：无
└── 关联：用户体系（封禁）、运营后台
```
