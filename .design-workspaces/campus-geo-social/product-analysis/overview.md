# 校园地理社交产品 - 全局框架分析（Phase 1）

> 本文档目的：基于用户原始描述，快速建立产品全貌。**不深入细节**，仅做大方向对齐。
> 用户原始诉求来源：`校园社交产品.md`

---

## 一、产品定位

### 1.1 一句话定位

> **「以地理位置为锚、以时间为轴、以游戏化交互为催化剂的校园社交App」**——把每一寸校园角落变成可被记忆、可被发现、可被惊喜触发的社交触点。

### 1.2 核心价值（与微信/小红书/Soul 等差异化）

| 维度 | 主流社交 | 本产品 |
|------|----------|--------|
| **内容锚点** | 关注关系 / 话题 | **地理坐标 + 时间窗口** |
| **可见性规则** | 公开 or 好友 | **谁经过这个点 + 什么时候经过 才看得到** |
| **交友破冰** | 主动搜索/推荐 | **捞网游戏化，不确定性 + 付费稀缺感** |
| **内容生命周期** | 发布即消失（短）/ 永久（朋友圈） | **可埋藏多年，旧地重游触景生情** |
| **场景边界** | 全网 | **以校园为单位，可切换"游览"其他校园** |

### 1.3 产品别名候选（待用户决定）

- 候选 A：**「捞光」**（捞 = 捞网，光 = 时光胶囊）
- 候选 B：**「这里有人」**
- 候选 C：**「回响」**
- 候选 D：保留中文工程名 `campus-geo-social`，启动后再定品牌名

## 二、目标用户与核心场景

### 2.1 主要用户（Primary）

**在校大学生（18~24 岁，本科/研究生）**

特征画像：
- 强烈的**社交破冰需求**（新生认识同学、跨院系结识、暧昧期试探）
- 对**趣味性/游戏化**敏感（不喜欢直白搜索式交友）
- 大量**校园生活分享欲**（动态、Vlog、自习/食堂打卡）
- 在**固定地理范围**内反复活动（宿舍/食堂/图书馆/操场/教学楼这几个高频点）
- **毕业焦虑感** & **念旧情绪**（对"留下痕迹/将来回忆"有共鸣）

### 2.2 次要用户（Secondary）

- **已毕业的校友**（多年后旧地重游、看自己当年的胶囊、同学组队回忆）
- **跨校观察者**（保研/考研/换校/远恋探校，想了解其他校园氛围）
- **新生**（开学前预热、了解未来学校的生活气息）

### 2.3 核心使用场景（5 个高频场景）

| 编号 | 场景描述 | 触发情绪 | 关键功能 |
|:----:|---------|---------|---------|
| S1 | 在食堂排队/自习中段，刷一下"附近发了什么" | **打发碎片时间** | 位置动态 feed |
| S2 | 看到喜欢的人，想留个惊喜——"明天她到操场时弹一段视频" | **暧昧、心动** | 定时定点投递 |
| S3 | 在宿舍楼下，"我想认识同楼的同学/隔壁院系的同学" | **孤独、好奇** | 捞网交友 |
| S4 | 毕业 N 年后回校，"这里我当年发了什么？" | **怀旧、感慨** | 时间胶囊回忆触发 |
| S5 | 在 A 校读书，听说 B 校樱花开了，"切去 B 校的地图看看" | **猎奇、远观** | 跨校切换 |

## 三、功能模块全景

按领域识别出 **12 个模块**，标注 P0/P1/P2 优先级（用户立场是「4 大玩法全做 MVP」，下方优先级仅作工作量分层，不裁剪范围）。

```
[核心玩法层]
├── M1  位置动态系统          ★P0 - 立项之本，没它产品不存在
├── M2  捞网交友系统          ★P0 - 产品差异化第二支柱
├── M3  时空胶囊系统          ★P0 - 定时定点 + 长期回忆，差异化第三支柱
└── M4  跨校漫游系统          ★P0 - 用户明确诉求，差异化第四支柱

[支撑层]
├── M5  用户与认证体系        ★P0 - 学生身份认证是这类产品红线
├── M6  社交关系链            ★P0 - 好友/关注/私聊（捞到人之后要能继续聊）
├── M7  积分与道具经济        ★P0 - 捞网付费玩法依赖（用户明确提到付费）
├── M8  通知与触达            ★P0 - 定时投递/捞到新人/回忆唤醒都依赖通知
└── M9  内容审核与安全        ★P0 - 定向投递有骚扰风险，必须前置

[完整性层]
├── M10 个人主页与内容管理    ★P1 - 我的胶囊/我的足迹/我发布的
├── M11 设置与隐私            ★P1 - 位置隐私是重中之重
└── M12 搜索与发现            ★P2 - 用户/校园/地点搜索（首版可弱化）
```

### 3.1 模块依赖关系

```
M5 用户认证 ─┬→ M1 位置动态 ─┬→ M3 时空胶囊
            │                 └→ M4 跨校漫游
            ├→ M2 捞网 ────────→ M6 社交关系链
            └→ M7 积分经济 ────↗
                                
M9 审核 横切所有内容生成模块（M1/M3）
M8 通知 横切所有触发场景（M1/M3 触发投递、M2 捞到结果）
```

### 3.2 风险预警（建议在 PRD 完成前就关注）

| 风险类别 | 描述 | 建议应对 |
|---------|------|---------|
| **法律合规** | 学生身份认证、未成年保护、付费虚拟币（涉及增值电信/网络游戏许可？） | M5/M7 设计时必须预留运营资质余量 |
| **隐私敏感** | 位置数据是高敏感数据，"被人发现我在哪"会引发恐慌 | M11 必须支持**隐身模式**、**位置精度模糊化**、**反跟踪** |
| **骚扰边界** | "定时定点给某人投递"如果被滥用 = 跟踪/骚扰工具 | M3 必须有**接收方一键关闭某发送者**、**冷却期**、**举报通道** |
| **冷启动** | 位置社交是密度敏感产品，单校园用户少则毫无价值 | 建议**单校种子运营策略**（一个校园打透再扩） |
| **GPS 精度** | 室内/楼层/移动中精度差，"经过这里"判定容易误触 | M1 必须设计**自定义半径**（比如 50m / 200m / 500m 档位） |
| **道德边界** | 跨校观察容易演变成"窥探"，需要保护被观察校园学生隐私 | M4 必须区分**公开广场内容** vs **个人动态**，跨校只能看广场 |

## 四、MVP 范围

> **用户立场**：4 大玩法全做 MVP。我尊重这个立场，下方按"必须达到的最小可玩闭环"来拆。

### 4.1 V1.0 必须闭环的功能

```
┌─────────────────────────────────────────────────────────┐
│  注册 → 学生认证 → 选择主校园                            │
│       ↓                                                  │
│  地图首页 ←── 位置动态 feed                              │
│       │                                                  │
│       ├─→ 发动态（图文/视频 + 地点 + 可见性 + 触发条件）│
│       │      └─ 公开 / 定时定点 / 时间胶囊（埋）        │
│       │                                                  │
│       ├─→ 捞网（在某地点开网，消耗道具，捞出附近用户）  │
│       │      └─ 看到匿名/半匿名信息卡 → 加好友/打招呼   │
│       │                                                  │
│       ├─→ 切换校园（本校 / 浏览其他校园广场）           │
│       │                                                  │
│       └─→ 我的（足迹地图/我的胶囊/已捞到的人/钱包）     │
└─────────────────────────────────────────────────────────┘
```

### 4.2 V1.0 不做的（明确划出去，避免范围蔓延）

- 视频拍摄滤镜/剪辑（用系统相册或调用第三方）
- 直播
- 群聊（私聊够用即可）
- 兴趣社团/校园官方账号
- 校园活动报名/票务
- 长内容（图文笔记按"动态"上限处理，不是小红书）
- 外卖/任务悬赏类校园工具

### 4.3 工作量与里程碑预估（粗估，待 PRD 完成精算）

| 阶段 | 内容 | 预计深度分析模块数 |
|------|------|-------------------|
| Phase 2.A | 核心玩法层（M1~M4） | 4 个，最复杂 |
| Phase 2.B | 支撑层（M5~M9） | 5 个，中等复杂 |
| Phase 2.C | 完整性层（M10~M12） | 3 个，相对简单 |
| Phase 3 | PRD 汇总 + 信息架构 + Registry 骨架 | 1 套 |

## 六、需要用户确认的关键问题

> 进入 Phase 2 之前，以下问题影响后续所有模块设计，请逐项答复：

### Q1（产品边界）
**「定时定点投递」的"被投递人"如何识别？**
- A) 必须是好友（安全，但暧昧期可能还没加好友→功能受限）
- B) 知道对方手机号/学号即可（接近现实暧昧场景，但有骚扰风险）
- C) 锁定一个用户头像/昵称（需要先在 App 里发现对方）
- D) 你来选，或描述你的想法

### Q2（认证强度）
**学生身份认证有多严？**
- A) 强：必须学信网/学生证 OCR + 人脸核身（最干净，但门槛高、冷启动慢）
- B) 中：邮箱后缀 + 学生证拍照人工审核（折中）
- C) 弱：手机号 + 自填学校（开放，但容易混入校外人员）
- D) 你来选

### Q3（捞网经济）
**捞网消耗哪种货币？**
- A) 完全付费（人民币/虚拟币充值）
- B) 任务/签到获得 + 付费补充（混合，类似游戏体力）
- C) 完全免费但有冷却时间（不依赖经济系统）
- D) 你来选

### Q4（跨校漫游边界）
**跨校能看什么、不能看什么？**
- A) 只看广场（公开动态聚合，看不到具体用户）
- B) 能看到用户但不能直接互动（需对方同意）
- C) 完全互通，跟本校体验一致（隐私风险大）
- D) 你来选

### Q5（时间胶囊解锁规则）
**时间胶囊的"解锁条件"是什么？**
- A) 只到达地点即可（任何人路过都能看 → 公共回忆墙）
- B) 必须是发布者本人 + 到达地点（私人回忆）
- C) 发布者指定一组人 + 到达地点（亲密小圈子）
- D) 多种规则用户自选（最灵活，但 UI 复杂）

### Q6（视觉风格倾向）
**整体视觉风格基调？**（影响后续主题生成）
- A) **赛博/未来感**（深色 + 霓虹/电子地图风，强调"科技感的位置触发"）
- B) **青春治愈**（明亮配色 + 手绘元素，强调"校园温暖回忆"）
- C) **复古胶片**（米黄/暗红 + 颗粒纹理，强调"时光胶囊"质感）
- D) **极简文艺**（黑白灰 + 一抹强调色，强调克制与高级感）
- E) 你描述一个参考品（比如 "类似 Soul 但更温暖"）
