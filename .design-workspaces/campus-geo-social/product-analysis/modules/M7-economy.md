# M7 积分与道具经济

> **优先级**：P0 - 支撑 M2 捞网付费玩法
> **决策依据**：D3 混合经济（任务/签到 + 付费补充）
> **关联模块**：M2 捞网（消耗）、M5 认证（注册赠送）、M8 通知（任务提醒）、M10 个人主页（钱包入口）

---

## Step A: 用户故事

### 核心故事
- 作为**用户**，我希望通过日常任务（签到/发动态/邀请朋友）获得积分，可以兑换"网"等基础道具，不必花钱也能体验产品
- 作为**用户**，我希望对于稀缺/高价值道具（炸弹/特别招呼），可以通过钻石（付费购买）获得，加快体验
- 作为**用户**，我希望钱包页清晰展示两种货币（积分 / 钻石）和道具数量，知道自己有什么、能干什么
- 作为**用户**，我希望能查看历史消费记录，知道每个钻石/积分花到哪了
- 作为**用户**，我希望充值流程顺畅，支持微信/支付宝主流方式

### 扩展故事
- 作为**用户**，我希望首次充值有优惠（首充礼包），后续也有定期活动促销
- 作为**用户**，我希望能赠送钻石/道具给好友（社交化经济）
- 作为**用户**，我希望任务有"挑战" 维度（连续 7 天签到加倍奖励）
- 作为**用户**，我希望被捞到/被点赞这种"被动行为"也能赚一些积分（鼓励产出优质内容）

### 异常故事
- 作为**用户**，充值订单已扣款但未到账 → 自动退款 / 客服人工补发
- 作为**用户**，怀疑账号被盗刷 → 一键冻结钱包 + 申诉
- 作为**用户**，注销账号但还有未消耗钻石 → 强制使用完 / 主动放弃 / 申请退款

## Step B: 核心流程

### B1. 任务获得积分流程

```
用户进入主页 → 系统检测今日任务进度 → "我的任务"页签可见
→ 完成任务（如发动态/捞网/收到招呼回复）→ 客户端上报事件 → 服务端核验
→ 任务进度更新 → 完成后弹窗"获得 +50 积分" → 进入钱包
```

### B2. 充值钻石流程

```
1. 钱包/购买道具 → 选择钻石套餐 → 确认协议 → 选择支付方式
→ 2. 唤起微信/支付宝 SDK → 用户支付 → 三方回调
→ 3. 服务端校验 → 落库订单 → 钻石入账 → 客户端轮询 / 推送通知
→ 4. 显示"充值成功 +N 钻石"

异常：
├── 支付中断（用户取消）   → 订单作废
├── 三方回调失败          → 客户端发起"补单"查询，服务端兜底
├── 重复支付              → 业务幂等保护，第二次自动退款
└── 风控拦截              → 客服介入
```

### B3. 道具兑换/购买流程

```
1. 道具商店页 → 选择道具 → 显示价格（积分版 / 钻石版） 
→ 2. 余额校验 → 余额不足时引导（任务获得积分 OR 充值钻石）
→ 3. 确认购买 → 扣款 + 道具入账 → 显示"道具已加入背包"
```

### B4. 退款流程（合规要求）

```
用户申请退款 → 提交原因 → 客服审核：
├── 7 天内未使用     → 全额退款（仅充值，赠送钻石不退）
├── 已使用部分       → 不予退款（虚拟商品已消耗）
└── 系统故障导致     → 全额退款 + 补偿
```

## Step C: 业务规则与边界 Case

### C1. 双账本设计

| 维度 | 积分（Coin） | 钻石（Gem） |
|------|--------------|-------------|
| 来源 | 任务/签到/活动赠送 | **付费购买** + 部分活动赠送 |
| 可购道具 | 基础道具（网/招呼） | 全部道具（含炸弹/特殊） |
| 是否可退款 | 否 | 充值部分可退（7 天内未使用） |
| 是否可转赠 | 否 | 部分道具可送好友（不送钻石本身） |
| 现金价值 | 0（明确无） | 有（按充值比例） |
| 上限 | 99,999 | 999,999 |
| 过期机制 | 每年 12-31 23:59 清零年初前未消耗的 50%（防囤积） | 永不过期 |

### C2. 道具清单（MVP）

| 道具 | 用途 | 积分价 | 钻石价 | 备注 |
|------|------|--------|--------|------|
| 网 | M2 普通捞网 | 10 | 1 | 主力消耗品 |
| 招呼 | M2 给捞到的人发招呼 | 30 | 3 | 每日免费 3 个 |
| 补网 | M2 同地点 1h 内 50% 折扣再开 | 50 | 5 | 50% 概率任务获得 |
| 小炸弹 | M2 一次捞 3 人 | - | 30 | **仅钻石** |
| 大炸弹 | M2 一次捞 5 人 | - | 60 | **仅钻石** |
| 时光戳 | M3 胶囊解锁加速（缩短 1 个月） | - | 100 | **仅钻石**，V1.x 上线 |
| 隐身券 | M11 暂时隐身 24h（不可被捞） | 200 | 20 | 防骚扰道具 |

### C3. 任务体系（积分来源）

#### 每日任务
| 任务 | 奖励 |
|------|------|
| 每日签到 | +5 积分（连签 7 天 +30 加成，连签 30 天 +200） |
| 发布 1 条动态 | +20 积分 |
| 完成 1 次开网 | +5 积分（仅普通网） |
| 收到 1 次招呼回复（被认可） | +10 积分 |
| 给好友的动态点赞/评论 1 次 | +2 积分（每日上限 5 次） |

#### 新手任务（一次性）
| 任务 | 奖励 |
|------|------|
| 完成强认证 | +100 积分 |
| 上传头像 | +20 积分 |
| 发布第一条动态 | +50 积分 |
| 第一次开网 | +30 积分 |
| 邀请 1 位好友注册成功 | +200 积分 |

#### 成就任务（长期）
| 任务 | 奖励 |
|------|------|
| 累计 30 天活跃登录 | +500 积分 + "勤奋徽章" |
| 累计发布 100 条动态 | +1000 积分 + "记录者徽章" |
| 累计认识 50 位新朋友 | +1000 积分 + "社交家徽章" |
| 埋下首个胶囊 | +50 积分 |

### C4. 充值套餐

| 套餐 | 价格（人民币） | 钻石数 | 赠送 |
|------|---------------|--------|------|
| 6 元档 | ¥6 | 60 | - |
| 30 元档 | ¥30 | 300 | +30（首充翻倍） |
| 88 元档 | ¥88 | 880 | +120 |
| 198 元档 | ¥198 | 1980 | +280 |
| 648 元档 | ¥648 | 6480 | +1000 |

### C5. 业务规则

#### 防欺诈
- 单设备每日充值上限 ¥1000（预防被盗刷）
- 单账号每月充值上限 ¥5000（防误用）
- 充值前必须实名（M5 强认证已覆盖）
- 异常账号（频繁充退/异地切换）触发风控冻结

#### 合规
- 充值虚拟币明确不可提现
- 充值协议必须勾选阅读
- 提供消费记录查询（≥ 12 个月）
- 退款政策清晰公示
- 未成年人保护（虽然大学生 ≥ 18，但若实名信息显示未成年，触发监护人通道）

#### 经济平衡
- 系统设置整体兑换比：1 钻石 ≈ 10 积分（防止积分通货膨胀）
- 钻石不可兑换为积分（单向）
- 积分有年清零机制（见上表）

### C6. 边界 Case

| 场景 | 处理 |
|------|------|
| 用户支付时网络中断 | 客户端轮询订单状态 + 服务端兜底，30 分钟内必然有结果 |
| 用户重复点击充值按钮 | 业务幂等：同 user_id + 时间戳 5 分钟内拒重复请求 |
| 苹果端要走 IAP | iOS 端走 Apple IAP，Android 走微信/支付宝；钻石数量同步但价格按各平台规则 |
| 任务进度同步丢失 | 客户端上报 + 服务端事件源（如发动态时携带任务上下文） |
| 用户兑换道具瞬间挂掉 | 事务性扣款 + 道具入账，失败回滚 |
| 退款已发放后用户用积分买了道具 | 不影响（积分非充值来源） |

## Step D: 数据模型概要

### 涉及实体

```typescript
Wallet {                        // 用户钱包（双账本）
  user_id: bigint,              // 主键
  coin_balance: int,            // 积分余额
  gem_balance: int,             // 钻石余额
  updated_at: timestamp,
}

CoinTransaction {               // 积分流水
  id: bigint,
  user_id: bigint,
  delta: int,                   // 正负
  balance_after: int,
  source_type: enum,            // task | sign_in | invite | exchange | system_grant | annual_clean
  source_id: string,
  description: string,
  created_at: timestamp,
}

GemTransaction {                // 钻石流水
  id: bigint,
  user_id: bigint,
  delta: int,
  balance_after: int,
  source_type: enum,            // recharge | exchange | gift_send | gift_receive | refund
  source_id: string,
  description: string,
  created_at: timestamp,
}

Item {                          // 道具背包
  user_id: bigint,
  item_code: string,            // 'net' | 'greet' | 'refish' | 'bomb_small' | ...
  quantity: int,
  updated_at: timestamp,
}

ItemTransaction {               // 道具流水
  id: bigint,
  user_id: bigint,
  item_code: string,
  delta: int,
  source_type: enum,            // buy | use | gift | system_grant | refund
  source_id: string,
  created_at: timestamp,
}

RechargeOrder {                 // 充值订单
  id: bigint,
  user_id: bigint,
  package_id: string,
  amount_cny: float,
  gem_quantity: int,
  bonus_gem: int,
  pay_channel: enum,            // wechat | alipay | apple_iap | google_play
  external_order_id: string,
  status: enum,                 // pending | success | failed | refunded
  created_at: timestamp,
  paid_at: timestamp,
  refunded_at: timestamp | null,
}

Task {                          // 任务定义
  id: string,
  type: enum,                   // daily | onboarding | achievement
  title: string,
  description: string,
  reward: { type: 'coin' | 'gem' | 'item', amount: int, item_code?: string },
  trigger_event: string,        // 'sign_in' | 'post_moment' | 'cast_net' | ...
  conditions: object,           // 任务完成条件 JSON
  active: bool,
}

UserTaskProgress {              // 用户任务进度
  user_id: bigint,
  task_id: string,
  progress: int,
  target: int,
  completed_at: timestamp | null,
  reward_claimed: bool,
  reset_at: timestamp,           // 每日任务有 reset
}
```

### 涉及接口

```
# 钱包查询

GET    /me/wallet/coin/transactions     积分流水
GET    /me/wallet/gem/transactions      钻石流水
GET    /me/items                        道具背包

# 任务
GET    /me/tasks?type=daily             今日任务列表
POST   /me/tasks/{id}/claim             领取任务奖励

# 充值
GET    /recharge/packages               充值套餐列表
POST   /recharge/orders                 创建充值订单
GET    /recharge/orders/{id}            查询订单状态
POST   /recharge/notify                 三方支付回调（服务端内部）
POST   /recharge/orders/{id}/refund     申请退款

# 道具购买/赠送
POST   /shop/buy                        购买道具（用积分或钻石）
POST   /shop/gift                       赠送道具给好友
```

## Step E: 交互要点

### 钱包页（M10 入口集成）
- **顶部双账本卡片**：左积分（暖色系奶油黄）+ 右钻石（蓝紫色 + 闪光），数量大字号，下方"充值/赚取"小按钮
- **道具背包格子**：6×N 网格，类似游戏背包，长按查看详情
- **快捷区**：常用任务入口（"今日签到"、"邀请好友"）+ 优惠活动 banner

### 任务页
- **任务进度可视化**：每个任务一行，进度条 + "去完成"或"领取"按钮
- **完成后的反馈**：金色光效从任务图标射出，币飞向钱包

### 充值页
- **大卡片式套餐**：每个套餐独立大卡片，明显标"首充翻倍" / "限时优惠"
- **协议勾选**：默认未勾选，必须用户主动勾上
- **支付方式**：iOS 仅 IAP；Android 微信/支付宝
- **支付中**：全屏 loading + 取消按钮
- **支付成功**：钻石飞入钱包动画 + 提示

### 风格基调（D6 治愈）
- 积分图标：奶油色硬币 + 微笑
- 钻石图标：粉/蓝渐变水晶 + 微闪
- 不用刺眼红色"立即充值" → 用温和的"添加钻石"措辞

## 风险标记

| 风险 | 缓解 |
|------|------|
| 合规：充值需要 ICP 增值/网络游戏许可 | 立项前后端走牌照流程；MVP 先以"虚拟道具"包装而非"游戏代币" |
| 苹果税 30% 影响利润 | iOS 端定价策略需独立（如同价位道具数量略少） |
| 未成年伪造身份充值 | 强认证 + 大额充值二次校验 + 监护人通道 |
| 通货膨胀（积分/钻石过剩） | 年度积分清零机制 + 道具数量上限 |
| 任务被脚本刷量 | 服务端事件验证（设备指纹、行为指纹）+ 异常账号风控 |
| 用户充值后申诉退款率高 | 清晰的退款政策 + 7 天保护期 + 客服 SLA |
