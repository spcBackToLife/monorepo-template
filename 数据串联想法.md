我觉得要保持数据从上到下透传的完整性，他们一定是一个完整的整体数据源，把他们做为一个整体来处理，就是一个大的 Schema。
- 产品需求分析出整体的所有逻辑，从页面到功能点等等细节，本来也是一个结构化的数据，只是我们现在用 markdown 表达了。
这个结构化就按照：app - 页面 - 功能点 - 组件 等维度组成一个schema 树，这是我举的例子非常不完善，你应该仔细思考怎么结构化？原来分析的 markdown 的文档作为每个节点上的内容？这样就有结构的组织起来了。
- 然后下一步做交互分析的时候就可以基于这个大json 直接就生成任务列表，有序逐步的一个个分析交互细节
- 从而将这个扩展成更大完善的一个 schema，包括了产品描述，交互描述，是一个结构化的组织，所有的内容细节还是保持在原来的 md 里
- 然后下一步 UI 分析的时候，又是上面的大 schema 通过脚本生成任务列表，逐个页面、组件、素材等等单个任务列表就逐个分析所有的细节 UI 设计，视觉设计等等
- 然后这样又可以完善成一个更大的 schema，包括了产品描述，交互描述，UI 设计，视觉设计信息。
- 结合这个schema 肯定可以给 design-executor 生成非常详细的任务列表，就可以逐个的去执行，通过 page-builder 和 material-painter 来通过 mcp 进行设计。

这样通过这个 schema 保证了所有细节的完整性，又把每一步的内容合理的组织起来了，所有的元素多个维度的信息都在一起，
- 无论在什么时候都不会遗忘和漏掉
- 我们在最下游实现某个模块的具体的时候，它的产品表达、交互设计、视觉设计是什么，都在这个节点上清清楚楚
- 如果我们要核查组件少了没、素材少了没，这个schema 跑一个脚本一下就检查出来了

结合上述描述，帮我写一个完整的方案出来，看如何保证这一连串的信息准确性？怎么调整这些技能，要注意 skill-creator 规范，但是不许删掉核心技能内容！！



在帮我审阅下这个 schema 是否完整：
1. 我产品的交互细节是不是都有了，各种状态，比如：成功、失败了都怎么办，相关组件是什么？
2. 我每个组件的样式、素材、结构、产品交互逻辑等等细节，在实现的时候是不是都知道？
3. 页面的 api 设计、整体前端的状态管理怎么描述的，有吗？是不是也有独立文档，这里应该也得有索引？
4. 业务逻辑的阐述，是不是也得有文档，因为可能之前业务逻辑都在产品描述和交互描述里，但是落到后面实现的时候，这个业务逻辑怎么表达，比如：什么时候展示什么，拿什么，判断策略是什么等等之类的？
5. 一个组件的多个状态，悬浮、点击、聚焦等等，这个 schema 是否已经包含了，在组件实现的时候不会遗漏呢？
