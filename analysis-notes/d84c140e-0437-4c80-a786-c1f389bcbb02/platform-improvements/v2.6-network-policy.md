# Platform Improvement v2.6 — NetworkPolicy 一等公民

> 时间：2026-05-31
> 触发：interaction-designer 修 00-login 5 条漏译时，第 4/5 条 `endpoint.timeout` 暴露平台缺字段（boundaries.md D-B1 写了 timeout 阈值，但 ApiEndpoint 类型/zod schema/MCP 工具/运行时 EffectExecutor 全都不支持）。这不是 AI 漏译，是平台 backlog。
> 决策方式：按 v2.6 "做对"原则一次性建完整 NetworkPolicy 抽象，不补"最小闭环"。

## 1. 问题根因（第一性原理）

| 层 | timeout 字段是否存在 | 后果 |
|---|---|---|
| `ApiEndpoint` interface | ❌ 无 | AI 写不进 |
| `ApiEndpointSchema` zod | ❌ 无（strip 未知字段）| 即使硬塞会被 strip |
| `DataSourceSetEndpointOp` ops | ❌ 无 | service 端传不过去 |
| MCP `set_endpoint` EndpointSchema | ❌ 无 | MCP 工具入口拒收 |
| `EffectExecutor.run()` | ❌ 不读 timeout | 即使写进去也不消费 |
| MockDriver | ✅ 间接（delay + isTimeout）| 仅 mock 行为正确，真接口无效 |
| 错误码 `TIMEOUT` | ✅ 已存在但上层不知道 | events 模板/AI 都写 NETWORK_ERROR，runtime mock 实际返 TIMEOUT |

事实链：
1. AI 写 boundaries 写出 timeout 阈值 → 业务诉求正确识别
2. AI 落 datasources 发现工具不支持 → 标 D-DS6 "暂不动"
3. v2.5 复盘列为"漏译" → 但根因是平台缺字段
4. 第三次撞上（本次）→ 没有平台改造工具，AI 只能 skipped

## 2. 完整方案

### 2.1 schema 设计

```ts
export type ErrorCode = 'TIMEOUT' | 'NETWORK_ERROR' | 'SERVER_ERROR';

export interface NetworkPolicy {
  timeout?: number;           // 整请求超时（ms）；触发 onError code='TIMEOUT'
  retryCount?: number;        // 重试次数（不含首次）；默认 0
  retryDelay?: number;        // 指数退避基数（ms）；间隔 = base × 2^attempt
  retryOn?: ErrorCode[];      // 默认 ['TIMEOUT','NETWORK_ERROR']
}

export interface ApiEndpoint {
  // ... 原有字段
  networkPolicy?: NetworkPolicy;
}
```

子结构选型理由：
- 表达"网络层关注点"的语义边界，与业务字段（method/path/body）解耦
- 后续加 retry/circuitBreaker 字段不用再改 ApiEndpoint 顶层
- 符合 AGENTS.md §四"DRY 与抽象"+§九"永远只保留最新版本"

### 2.2 错误码标准化

历史问题：MockDriver 已用 `code: 'TIMEOUT'`，但 boundaries.md / events.md / errors.md 全写 NETWORK_ERROR。**TIMEOUT 从 NETWORK_ERROR 分出来**，按"用户该怎么处理"分类：
- `TIMEOUT`：链路慢 → 用户该重试 / 换时段
- `NETWORK_ERROR`：物理断网 / DNS / connection refused → 用户该开网络
- `SERVER_ERROR`：5xx → 服务方问题，需上报

HttpDriver / MockDriver 统一：
- 5xx → 归一化 SERVER_ERROR
- 4xx → 保留原数字 code（业务错由 responseBody.code 由上层 logic.switch 消费）
- AbortError（reason=timeout）→ TIMEOUT
- AbortError（reason=cancel）→ status='idle'
- 其他异常 → NETWORK_ERROR

### 2.4 兼容性策略（无双版本）

- 历史 dataSource.endpoint 都没 networkPolicy → undefined → runtime 走"无超时无重试"分支 = 与现状等价
- 无 v1 兼容分支：MockDriver `isTimeout` 保留（这是 v1 唯一表达方式，与 networkPolicy.timeout 是两种独立机制）
- 错误码 TIMEOUT 拆分：现有 SubmitBtn/GetCodeBtn 的 onError 没 TIMEOUT case 会走 default 兜底（已有 Toast"出了点问题"）→ 不会崩
- 后续阶段（design / executor）**无需感知** networkPolicy

## 4. 经验沉淀

1. **平台 backlog 不是 demo 项目可以"绕过去"的事**——AI 多次发现同一缺陷而没机制升级为 issue → 平台改造永远拖延
2. **第一性原理不是"够用就行"，是"做对"**——同一改造一次扩齐 timeout/retry/retryOn 抽象，比补丁 N 次改 N 个字段更省
3. **错误码混乱要早治**——runtime 已用 TIMEOUT 但上层全写 NETWORK_ERROR 这种"已有但未对齐"的隐藏问题比"未实现"更隐蔽
4. **schema-runtime-mcp-编辑器-文档"五层闭环**——少一层就有"声明但无人读"的死字段，AGENTS.md §九"无双版本"+§三"禁止臆想"的具体体现
5. **AbortController.abort(reason)** 是 Web API 现代姿势 —— 区分主动 cancel vs 自动 timeout 用 reason 一字段就够，比维护额外状态 flag 更清晰
