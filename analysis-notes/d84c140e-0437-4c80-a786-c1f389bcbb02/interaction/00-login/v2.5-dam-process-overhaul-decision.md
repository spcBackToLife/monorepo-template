# 流程改造决策（v2.5）：DAM 翻译契约 + minimal-debug styles

> 时间：2026-05-31
> 触发场景：interaction-designer 完成 00-login 屏所有任务、integrity 0 错后，用户预览发现：
>   1. **输入"138"失焦看不到红字提示**（PhoneError styles 空，渲染塌陷）
>   2. **点 SubmitBtn 静默无反馈**（D-E4 决策"onBlur+onSubmit 双档"中 onSubmit 档漏译）
> 结论：v2.4 的产物指纹机制（anyNodeHasEvents + R-EVENTS-02/03）能拦"事后大类遗漏"，但拦不住"事前细粒度漏译"——必须给 interaction-designer 流程加翻译追踪机制 + 派生节点最小可见性。

## 1. 问题根因（第一性原理）

### 1.1 "输入框 onSubmit 校验漏译"为什么会发生？

现场还原：
- `errors.md` 决策 D-E4 写"onBlur+onSubmit 两档"
- `events.md` 落库时只写了 onBlur 那档；onSubmit 档漏译
- 但 `events.md` 自审"覆盖完整"，`I-M1-events` 也标 done 了

根因：
- `I-X-events` 任务粒度过大（一锅炖 18+ 个节点的 events）
- AI 在这个粒度下没办法系统性交叉对账"errors/operations/boundaries 的每条决策都被翻译到 schema 了吗"
- 决策→产物映射只活在 AI 写 events.md 那一瞬的"脑子里"，写完丢失，没人能查
- v2.4 的 expectedArtifacts (`anyNodeHasEvents`) 只能保证"屏内有交互"，看不出"D-E4(3) SubmitBtn.click 前置校验"这种粒度的语义遗漏

### 1.2 "输入 138 看不到红字"为什么会发生？

现场还原：
- PhoneInput.blur 校验逻辑正确，写入 `view.errors.phone`
- PhoneError 节点 textContent + visibleWhen 表达式正确
- 但 PhoneError.styles 是 `{}`——按"interaction 不写 styles"边界规则保留给 design 阶段
- 渲染器拿空 styles 给文字 → 默认黑色小字 / 高度可能塌陷

根因：
- "interaction 阶段 → 验证逻辑"和"design 阶段 → 视觉"的硬分隔断裂了**可验证性闭环**
- 用户在 interaction 阶段做完所有逻辑落库后，**没有任何方式预览到自己做的对不对**——因为派生显示节点全是默认无 style 黑色小字
- AI 自审"逻辑全 ✓"，但用户实操"看不到任何反馈" → **事前漏译只能等到 design 阶段才被发现**

## 2. 解决方案

### 2.1 翻译契约（DAM, Decision-to-Artifact Mapping）

**核心**：把"决策→产物"映射从"AI 脑子里"显式化为"md 中的强制结构化段落"，service 端可机器对账。

#### 上游分析 md 强制 ★ 翻译契约 段（7 类模板：statemachine/operations/loading/errors/boundaries/state-vars/datasources）

每份 md 末尾必须含：

```markdown
## ★ 翻译契约（Decision-to-Artifact Mapping）

| 决策 ID | 决策内容 | 应翻译为 schema 产物 | 落库任务 | nodeId | 期望指纹 |
|---------|---------|---------------------|---------|--------|---------|
| D-E4 | onBlur+onSubmit 双档 | (1) PhoneInput.blur 校验<br>(2) CredentialInput.blur 校验<br>(3) **SubmitBtn.click 前置校验** | I-X-events | (3) SubmitBtn | `nodeHasEvent { trigger:'click' }` |
| D-E5 | 不做 onChange debounce | （无 schema 产物，决策记录即足）| — | — | — |
```

**所有决策都列**——没产物的标"无 schema 产物"，证明 AI 没漏想这条。

#### 落库 md 强制 ☐ 翻译清单 todo 段（events / view-* / overlays 等）

md 头部必须先汇总所有上游 md 的"翻译契约"段：

```markdown
## ☐ 翻译清单 todo

- [ ] D-E4(1) PhoneInput.blur 校验 → `nodeHasEvent { trigger:'blur' }`
- [ ] D-E4(2) CredentialInput.blur 校验
- [ ] **D-E4(3) SubmitBtn.click 前置校验** ★ ← 历史漏译复盘
- [ ] ...

→ 全部 [x] 才能写"沉淀"段并标 done
```

#### 配套新 ArtifactCheck.kind = 'nodeHasEvent'

```ts
{ kind: 'nodeHasEvent';
  nodeId: string;          // 节点真实 ID
  trigger?: EventTrigger;  // 可选限定 trigger
  min?: number;            // 默认 1
  path?: string }          // 默认 'rootNode' 子树搜索起点
```

实现：在 path 起的子树深度优先查 `id===nodeId` 的节点，校验其 events 满足条件（actions 非空数 ≥ min；若指定 trigger 则该 trigger 至少 1 个有非空 actions）。

落库任务 md 末尾把 todo 的产物指纹汇总进 `expectedArtifacts`，service 端在 `update_plan_task done` 时机器对账：

```jsonc
expectedArtifacts: [
  { kind: 'anyNodeHasEvents', path: 'rootNode', min: 1 },                      // 屏级粗
  { kind: 'nodeHasEvent', nodeId: '<SubmitBtn-id>', trigger: 'click' },        // D-E4(3)
  { kind: 'nodeHasEvent', nodeId: '<PhoneInput-id>', trigger: 'blur' },        // D-E4(1)
  // ...
]
```

### 2.2 派生展示节点 minimal-debug styles 白名单

新开 7 属性窄白名单，让 interaction 阶段的派生展示节点（PhoneError / Spinner / 倒计时位 等）写最小可见 styles：

| 属性 | 推荐值 |
|------|------|
| color | `#ef4444` 错 / `#16a34a` 成功 / `#6b7280` 提示 |
| fontSize | 12-13px |
| lineHeight | 1.4-1.5 |
| marginTop / marginBottom | 4-6px |
| minHeight | 16-20px（防空内容塌陷）|
| padding | 2-4px |

仅限 6 类节点角色：`inline-error` / `inline-hint` / `inline-success` / `countdown-text` / `spinner` / `toast-text`。

不是"interaction 做了一半视觉"，而是**让逻辑层产物可被立即预览验证**。design 阶段会用主题 token 完整覆盖（`color: '$token:colors.error'` 等），不冲突。

## 4. 验证

### 4.1 nodeHasEvent kind 落地
- 给 00-login 的 `I-M1-events` 加 6 条精确指纹（SubmitBtn.click / PhoneInput.blur / CredentialInput.blur / GetCodeBtn.click / Root.screenEnter / Root.screenExit）→ 全部通过
- 故意构造 nodeId="nd_NOT_EXIST_TEST" → R-PLAN-01 精确报错 "子树 'rootNode' 中未找到 id='nd_NOT_EXIST_TEST' 的节点"
- 改回正确 → 0 错

### 4.2 因果闭环

| 原"事前漏译"场景 | 改造后由谁拦 |
|---|---|
| 上游决策没列翻译契约段 | 落库 md 写 todo 时显式发现 → 退回上游补 |
| 翻译契约列了，落库 md 没拼到 todo | SKILL §5.1 红线 + 任务 review |
| todo 列了 [ ]，落库时漏一条 | 头部 todo 全 [x] 才能写"沉淀"段 |
| todo 全 [x]，但 schema 实际不满足 | expectedArtifacts(`nodeHasEvent`) 在 update_plan_task done 时机器对账，service 拒 |
| 派生节点逻辑落了但用户预览看不到 | minimal-debug styles 必须写，让用户立即能看到错误反馈 |

每层都是结构化的、可机器或人类对账的。

## 6. 经验沉淀

1. **任务粒度决定漏译概率**：`I-X-events` 一锅炖 18+ 节点必然漏；翻译契约段 + todo 清单把它从"自由发挥"改成"逐条勾"
2. **决策→产物映射必须显式化**：只活在 AI 脑子里的映射写完就丢，必须用结构化段落锁住
3. **可验证性闭环不能断在阶段切分上**：interaction 落完不能要求"等到 design 才能预览"——派生展示节点必须有 minimal-debug styles 立即可见
4. **机器对账 > 人类自审**：精确产物指纹（nodeHasEvent { nodeId, trigger }）是杜绝"AI 自报式造假"的最终手段
5. **AGENTS.md §3 第一性原理再次验证**：从"事后检测怎么不漏" → "事前怎么不漏"的视角倒推，才能挖出根因（任务粒度 + 映射追踪缺失）；不要在错误抽象层（关键词正则、白名单扩容、补丁式修复）打补丁
