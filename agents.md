# Agent 约束（本仓库）

## 红线：素材与画布能力必须通过 MCP 验证

凡涉及 **素材编辑器 / 画布几何 / 导出应用到设计节点** 的工作，**必须**走 **design-mcp**（如 `canvas`、`material_project`、`material_slot`、`execute_operations_batch` 等工具）以及 **design-api 上正式的素材/设计操作接口**，以便与真实产品链路、操作日志、协作行为一致。

**禁止「绕过 MCP」**，包括但不限于：

- 为求快直接向 `apps/design-api/uploads/`（或任意静态资源目录）**手写 / 脚本生成** SVG、PNG 等，再手动把 URL 填进 schema，而**不**经过 MCP 驱动的素材工程与导出流程；
- 用一次性 `curl`/脚本只打 HTTP、却**不**通过 MCP 工具编排，来冒充「已用素材能力测过」；
- 任何等价于**跳过素材画布与 MCP 工具链**、使测试结果无法代表 MCP 与编辑器真实行为的捷径。

若 MCP 能力不足或阻塞，应 **先说明限制**，再讨论改 MCP、改 API 或拆任务；**不得**用文件落盘代替 MCP 完成本属素材/MCP 测试范围的需求。

## 何时应从「产品」角度完善 MCP / 素材编辑器（而非继续堆一次性方案）

从第一性原理看，素材编辑器 + MCP 的目标是：**把设计意图稳定、可复用地映射到可导出资产与操作日志**。当出现下面一类信号时，应优先走 **产品化能力**（改编辑器、改 schema、扩 MCP），而不是让 Agent 或工程师长期用脚本/手写文件代偿：

1. **共性**：同一类 UI（例如「带缺口的渐变光圈」「沿路径的霓虹描边」「多段不等宽描边」）在多个项目/屏幕反复出现，且每次都要 **大量 primitive 堆叠** 或 **超出当前 path/rect 表达力**。  
2. **意图 vs 表达落差大**：设计师用自然语言或设计稿描述的是 **少数参数**（缺口角度、色环、粗细、模糊半径），落库却要拆成 **几十条** `me:addObject` / 手写 SVG——说明 **抽象层级不对**。  
3. **MCP 测的是「真能力」**：若绕过 MCP 才能交付，说明 **MCP 或编辑器在链路上缺一块**，应把缺口记为 backlog，用 **新 action、新预设、新检查器字段** 补上，否则测试无效。  
4. **可逆与协作**：凡需要 **版本、撤销、协同编辑、槽位绑定** 的资产，能力应落在 **素材操作类型** 或 **官方导出路径** 上，而不是一次性文件。  
5. **成本结构**：若 **设计师** 在 UI 里拖半天 + **Agent** 写几百行 path 仍难对齐参考，而 **一个参数化组件** 能覆盖 80% 变体，就应做 **编辑器内建 + MCP 一等公民参数**。

**落地节奏建议**：先写清「设计意图 → 当前 schema/MCP 表达」的 gap 说明（1 页以内），再拆为 **MCP 扩展**（新 `canvas` action 或 `me:*`）、**编辑器 UI**、**导出/渲染一致性** 三条线之一或组合，避免只改一端导致 MCP 与前端行为分叉。

## 场景方案：「非闭合渐变光圈 / 语音光晕环」（第一性原理 → 产品化设计）

### 设计意图（抽象层）

- 几何：**圆环（或近似圆环）描边**，可选 **圆周缺口**（如 12 点方向小开口）。  
- 颜色：**沿圆周变化**（类 conic / 分段色相），而非一条全局 `linear-gradient` 斜切在 border 上。  
- 质感：**外发光 / 厚度不均 / 端点圆角** 等属于 **效果参数**，不应靠手工调几十个线段模拟。  
- 交付：导出 **SVG 或 PNG**，绑定 **参考框尺寸**，可 **apply 到 div 背景** 或与子内容（文案、波形素材）**分层叠放**。

### 当前痛点

- 仅用 `rect`/`path` 拼：缺口 + 平滑色相 + 发光，**操作数多、难迭代、MCP 冗长**，且与「设计师调两个滑块」的效率目标不符。  
- 纯 CSS 双背景 `linear-gradient` + `border-box`：**无法**沿圆周均匀铺色，也 **不易** 做可控缺口，且与「素材工程」模型脱节（你要求走 MCP）。  
- 手写 SVG 文件：**违反本仓库红线**，且不进操作日志。

### 推荐方案（分层，可分期）

**A. 产品原子能力（推荐主路径）——「参数化描边环」**

在素材编辑器中增加一类对象（或 inspector 扩展），例如 **`strokeRing` / `angularRing`**，核心参数：

| 参数 | 含义 |
|------|------|
| `diameter` / `strokeWidth` | 外径与环粗细 |
| `gapAngle` / `gapRotation` | 缺口张角与朝向（默认朝上） |
| `colorStops` | 沿弧长的色标（内部映射为 conic 或分段 stroke） |
| `blur` / `glowIntensity` | 外发光（导出时写入 filter 或 raster 化） |
| `lineCap` | 端点圆角 |

**MCP**：暴露单一操作，例如 `canvas` 的 `add_angular_ring` / `me:addAngularRing`，一次写入 schema + 可撤销；**导出**：引擎侧统一转 SVG path + `stroke`/`filter` 或栅格化，保证与设计稿一致。

**B. 中台能力——「沿路径的渐变描边」**

若 A 仍过窄：在引擎层支持 **矢量 path 的 stroke 使用沿路径渐变**（SVG2 / 近似实现），则缺口环退化为 **一条 Arc path + 渐变 stroke + blur**，仍可由 MCP 用 **一条操作** 创建，设计师也可在检查器里编辑 path 与渐变。

**C. 模板 / 预设库（增效）**

内置 **「语音 UI 光晕」「渐变缺口环」** 等预设；MCP 支持 `apply_material_preset(id, overrides)`，设计师改参数即可，Agent 只调 overrides。

**D. 与现有页面结构的关系**

- **环**：单独素材层 → `backgroundImage` / 槽位绑定到大圆 `div`。  
- **内区文案与波形**：继续用子节点 + 现有文本与波形素材，**层级清晰**，避免一个 div 同时扛「复杂背景 + flex 子树」导致样式难维护。

### 本期若尚未实现 A/B 时

在能力未上线前：**仍须用 MCP** 在画布内用 **有限条 path 或 line 段** 近似（并接受迭代成本），同时在 issue/backlog 里挂 **「参数化描边环」** 需求，避免再用文件落盘绕过。
