## 📝 重要约束（不要忘）

1. **demo 不重要，重要的是测试发现的问题** — 当前登录页 demo 里的视觉细节都是"症状"，治理的是"根因"
2. **事前治理优先，事后检测其次** — 别把"加截图任务/加文档"当治理；那是后置检测
3. **代码是契约，文档同步对齐** — SKILL md 与 resolveTokens.ts 的语法支持必须 100% 一致；任何一边漂移都退回上游补
4. **新会话续接时**：第一时间读这份文件，看 P0 剩余项 → P1 顺序做下去；不要跳着做 P2 的结构性改造
5. **★ workspace 包改了源码必须 build dist + 清 vite 缓存** —— `features/design-engine` `package.json` `main` 指向 `dist/index.js`，design-front 引用的是 build 产物。改 src 后必须：
   ```
   cd features/design-engine && pnpm build
   rm -rf apps/design_front/node_modules/.vite node_modules/.vite
   # 然后重启 vite + 浏览器硬刷 (Cmd+Shift+R)
   ```
   2026/05/31 P0-7 端到端验证时栽过坑：dist 没 build，vite 用 5/31 早上的旧版预构建包，截图依然全部 token 不解析。我误以为"P0-1 修了 resolveTokens 单测过了就生效"——但运行时跑的是 dist。
   → 这就是 **P2-1 渲染产物自检** 应该堵的洞：让 design-mcp 写完后立刻 headless 渲染对账，能在第一时间发现"schema 对了但运行时没接上"。

6. **★★ 架构断路诊断顺序：从渲染器入参反推，不要信任"接口签名上有就一定有传"** ——
   2026/05/31 P0-7 二次端到端验证时再次栽：第一性分析发现**真正的断路点**是 `SchemaRenderer / PreviewRenderer` 调用 `resolveNodeStyles` 时**根本没传第 5 个参数 themeConfig**。整条 token 解析链字段定义齐全、单测全过、build 干净，但**渲染器从来没把 ThemeConfig 注入下来**——所以 33/33 单测都是在"测试套手动传 themeConfig 进去"，**生产链路 themeConfig 永远 undefined → resolveTokensInStyles 第一行 `if (!themeConfig) return styles` 直接原样返回**。
   - 教训：诊断 token 解析问题不要只看"resolver 单测过没过"、"schema 字段对没对"、"build 接没接上"——必须**看运行时 resolver 实际收到的 themeConfig 是不是 null**。
   - 修复：建 `ThemeConfigContext`，design-front Canvas 在 SchemaRenderer/PreviewRenderer 外层 wrap Provider 注入 `editorStore.project?.themeConfig`，组件内 useContext 读传给 resolveNodeStyles 第 5 参。
   - **下次再遇"schema 对、resolver 单测过、视觉还是没出"立刻去看渲染器调 resolveNodeStyles 那行的实参列表**。
