**C3 的设计核心思路：**

```
Schema 五维度 → 代码五层映射

  结构维（组件树）   →  组件文件层（components/ + pages/）
  样式维（CSS）     →  样式文件层（*.module.css / tailwind classes）
  交互维（事件）     →  逻辑层（event handlers / hooks/）
  状态维（状态模型） →  状态管理层（stores/ / context/）
  数据维（数据绑定） →  数据层（api/ / types/ / mock/）
```

**代码拆分规则设计：**

```
拆分决策树：
  节点是组件模板实例？
    ├── 是（reference）→ 独立组件文件 + import
    └── 否（普通节点）→ 保持内联
  
  节点有事件绑定？
    ├── 多个事件 → 提取自定义 Hook
    └── 单个简单事件 → 内联 handler
  
  全局状态使用？
    ├── 有 → 生成 store 文件 + 组件中 useStore
    └── 无 → 省略状态管理层
```

## 七、风险与依赖

### 7.1 关键风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 代码导出质量难以达到「企业级」 | G1 目标打折 | 渐进式：先支持干净的 React 代码，再支持项目结构化导出 |
| 需求文档生成的自然语言质量 | G4 目标打折 | 可结合 LLM 润色，Schema → 结构化数据 → LLM 生成自然语言 |
| 事件链复杂度导致 Schema 膨胀 | G1 性能问题 | 限制事件链深度，提供「逻辑太复杂请用代码」的边界引导 |
| 全景截图的性能与稳定性 | G4 验收能力 | 后台异步执行，分批生成 |

### 7.2 关键依赖

```
Phase C 依赖 Phase B：
  代码导出需要状态/数据/事件系统就绪
  需求文档需要交互链路、数据绑定完整

Phase D 依赖 Phase C：
  设计 Token 需要样式系统成熟
  AI 增强需要操作系统稳定

Phase B 依赖 Phase A：
  状态/数据需要画布和属性面板作为载体
```

## 八、验收标准：四大目标的量化检验

### G1 验收：设计即代码

```
场景：一个包含登录页、首页、个人中心的 App
验收：
  1. 设计师在编辑器中完成三个页面的 UI + 状态 + 交互设计
  2. 点击「导出代码」
  3. 生成的 React 项目可直接 npm install && npm start
  4. 页面结构、样式、路由跳转、状态切换与设计稿一致
  5. 代码具备：组件拆分、类型定义、样式文件分离、状态管理
  6. 无需手动修改任何代码即可运行出设计效果
```

### G2 验收：AI 增量操作

```
场景：通过 AI 从零构建一个表单页面
验收：
  1. 用户对 AI 说「创建一个登录表单」→ AI 一步步添加元素
  2. 用户说「把按钮改成圆角蓝色」→ AI 只改按钮样式
  3. 用户说「撤销刚才的颜色修改」→ 只撤销颜色，保留圆角
  4. 全程画布实时更新，用户看到每步变化
  5. 任意步骤可撤销/重做
```

### G3 验收：设计师友好

```
场景：设计一个多状态的用户卡片组件
验收：
  1. 创建一个 UserCard，定义三种业务状态：normal / vip / banned
  2. 在不同状态下配置不同的样式和内容
  3. 创建三个数据集：正常用户 / VIP用户 / 空数据
  4. 在编辑器中切换状态和数据集，实时预览六种组合
  5. 保存为组件资产，在其他页面复用
  6. 预览模式中真实交互可用
```

### G4 验收：产品友好

```
场景：产品经理基于设计稿生成需求文档
验收：
  1. 设计师完成一个完整 App 的设计
  2. 产品点击「生成需求文档」
  3. 生成的文档包含：
     - 页面信息架构
     - 每个页面的交互说明表
     - 页面跳转流程图
     - 数据接口需求清单
     - 各状态/数据场景的 UI 截图
  4. 文档中的交互链路与设计稿 100% 一致
  5. 开发人员仅凭文档即可理解全部需求
```

## 九、总结

```
Phase A (W1-W8)：让编辑器「能用」      → 基础编辑 + AI 基础操作
Phase B (W9-W14)：让编辑器「好用」     → 状态 + 数据 + 预览（核心差异化）
Phase C (W15-W22)：让编辑器「有产出」  → 企业级代码 + 需求文档（目标兑现）
Phase D (W23-W30)：让编辑器「成生态」  → 协作 + Token + AI增强（规模化）
```

> 核心策略：**先完成 11 个子系统的基础实现（Phase A-B），再补齐产出能力（Phase C），最后构建生态（Phase D）。**
> 
> 产出能力（代码导出 + 需求文档）是四大目标兑现的关键里程碑，需要在 Phase C 重点投入。
