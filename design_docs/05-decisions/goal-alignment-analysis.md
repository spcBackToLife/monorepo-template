## 一、最根本目标回顾

我们的四大目标：

| # | 目标 | 一句话概括 |
|---|------|-----------|
| G1 | **设计即代码** | 无需分类/分析，Schema 天然转化为企业级代码（代码组织、逻辑抽象、完整交互） |
| G2 | **AI 增量操作** | AI 通过 MCP 一步步绘制/调整 UI，可灵活撤销任意步骤，非全量重生成 |
| G3 | **设计师友好** | 组件资产沉淀 + 基于交互的设计 + 数据 Mock 驱动多状态呈现 |
| G4 | **产品友好** | 设计即需求表达，轻松转化需求文档，交互链路清晰，无需 AI 深度分析 |

## 二、当前架构与目标的天然匹配

### 2.1 架构基因的优势

当前架构从第一性原理推导出的核心基因，天然地支撑了大部分目标：

```
Schema = 结构(组件树) + 样式(CSS) + 交互(事件) + 状态 + 数据
                    ↓
        这五个维度恰好是代码的五个核心关注点
                    ↓
    Schema 不是「图纸」，而是「可执行的 UI 源数据」
```

**与传统设计工具的根本区别（已实现的架构优势）：**

| 维度 | 传统工具（Figma） | 我们的架构 | 对目标的意义 |
|------|------------------|-----------|-------------|
| 编辑对象 | 像素/矢量 | 结构化 Schema | G1: 无需「识别」组件，天然有语义 |
| 操作方式 | 像素操作 | Operation-based | G2: 每步操作可追踪、可撤销 |
| 状态表达 | 每状态一帧 | 三层状态模型 | G3: 一份稿覆盖多状态 |
| 数据驱动 | 无 | 数据集 + 表达式绑定 | G3/G4: 数据变化即可验证 UI |
| 交互定义 | 连线 + 原型 | 声明式事件系统 | G1/G4: 交互即代码、即文档 |
| 输出 | 静态图片 | Schema → 多框架代码 | G1: 直接产出可运行代码 |

### 2.2 匹配度总览

```
                满足度评估（基于当前产品设计完成度）

  G1 设计即代码     ████████░░░░  65%   ← 基础链路通，但代码工程化有缺口
  G2 AI 增量操作    ██████████░░  85%   ← 架构天然支持，细节待完善
  G3 设计师友好     █████████░░░  75%   ← 核心能力已设计，体验细节待打磨
  G4 产品友好       █████░░░░░░░  40%   ← 最大缺口：需求文档生成能力未设计
```

## 三、逐目标深度分析

### 3.1 G1：设计即代码 —— 天然转化企业级代码

#### 已具备的能力

| 能力 | 支撑子系统 | 现状 |
|------|-----------|------|
| 组件树 = 代码组件树 | 01-canvas + 08-layer-tree | ✅ Schema 树直接对应 React 组件树 |
| CSS 1:1 映射 | 03-property-panel | ✅ 样式用 CSS 属性名，零翻译成本 |
| 事件直接导出 | 09-interaction-binding | ⚠️ 基础事件可导出，事件链/条件为远期 |
| Props 类型契约 | 06-component-props | ✅ propDefinitions 直接对应 TypeScript Props |
| 数据绑定表达式 | 05-data-driven | ✅ `{{data.xxx}}` 可映射到真实数据管线 |
| 多框架 Codegen | design-codegen | ⚠️ React 已有基础，其他框架待实现 |

#### 核心缺口

**缺口 1：代码组织与模块拆分策略**

当前 Schema → 代码是「整棵树平铺到一个文件」，缺乏企业级代码组织：

```
当前：
  Schema Tree → 单个大组件文件（所有节点在一起）

期望：
  Schema Tree → 项目结构
    ├── components/          ← 按组件模板拆分
    │   ├── NavBar/
    │   │   ├── index.tsx
    │   │   ├── styles.module.css
    │   │   └── types.ts
    │   └── LoginForm/
    ├── pages/               ← 按 Screen 拆分
    │   ├── Home/
    │   └── Login/
    ├── stores/              ← 全局状态 → 状态管理
    │   └── globalState.ts
    └── hooks/               ← 交互逻辑 → 自定义 Hook
        └── useNavigation.ts
```

**需要补充的设计：**
- 组件模板实例 → 独立组件文件的拆分规则
- Screen → Page 文件的映射规则
- 全局状态 → 状态管理代码（useState / zustand / context）的映射
- 事件处理 → 自定义 Hook 的抽取策略
- 数据绑定 → API 调用层的映射

**缺口 2：逻辑抽象层**

当前事件系统是「行为列表」（navigate / setState / openUrl），缺乏业务逻辑抽象：

```
已有：onClick → navigate("/home")
缺少：onClick → validateForm() → if(valid) → submitData() → navigate("/home")
               ↑ 表单验证       ↑ 条件分支    ↑ 数据提交     ↑ 页面跳转
```

事件链（V2）和条件事件是远期设计，但对「企业级代码」至关重要。

**缺口 3：代码质量元数据**

Schema 中缺少影响代码质量的元数据：
- 组件命名规范（当前节点 name 是显示名，非代码变量名）
- 导出代码的文件命名偏好
- 样式方案偏好（CSS Modules / Tailwind / styled-components）
- 状态管理方案偏好（useState / zustand / redux）

### 3.2 G2：AI 增量操作 —— 非全量重生成

#### 已具备的能力

| 能力 | 支撑子系统 | 现状 |
|------|-----------|------|
| MCP Server | 11-collaboration | ✅ 已实现，工具丰富 |
| Operation-based | design-operations | ✅ 每步操作原子化 |
| 撤销/重做 | 全局 | ✅ 操作日志正反向执行 |
| 双通道同步 | 11-collaboration | ✅ 手动 + AI 共享操作定义 |
| 批量操作 | batch tool | ✅ 多个操作原子化执行 |
| 实时反馈 | WebSocket | ✅ AI 操作实时推送到画布 |

#### 核心优势

这是当前架构**最强的匹配点**。Operation-based 架构的四合一特性：

```
design-operations 的操作 = 编辑器的命令 = MCP 的 Tool Call = 版本管理的事件日志
```

天然满足「一步步画 UI」的需求。与 v0/bolt 等「每次全量生成」的工具形成根本差异。

#### 细节缺口

| 缺口 | 说明 | 优先级 |
|------|------|--------|
| **选择性撤销** | 当前只能顺序撤销，不能跳过某步单独撤销 | P2（Phase 6 of collaboration） |
| **AI 状态感知** | AI 对当前画布的「视觉理解」有限，依赖 Schema 文本描述 | P2 |
| **操作建议** | AI 不知道「下一步应该做什么」，缺乏设计上下文推理 | P3 |
| **操作分组** | 多步操作缺乏「逻辑分组」概念（如「创建导航栏」= 5 步操作应视为一组） | P2 |

### 3.3 G3：设计师友好 —— 组件资产 + 交互设计 + 数据 Mock

#### 已具备的能力

| 能力 | 支撑子系统 | 现状 |
|------|-----------|------|
| 两层资产体系 | 07-asset-management | ✅ 骨架组件 + 风格组件已设计 |
| 三级资产来源 | 07-asset-management | ✅ 我的/团队/官方 |
| 三层状态模型 | 04-state-system | ✅ 交互/业务/全局 |
| 数据集管理 | 05-data-driven | ✅ 多数据集切换、Mock 数据 |
| 全景截图矩阵 | 05-data-driven | ⚠️ 已设计，实现为远期 |
| 预览模式 | 10-preview-mode | ✅ 真实交互预览 |
| 组件 Props 契约 | 06-component-props | ✅ 标准化可配置能力 |

#### 核心优势

**「基于交互的设计」是本系统的最大创新点之一：**

```
传统（Figma）：
  登录页默认状态    → 一个 Frame
  登录页加载中状态  → 又一个 Frame
  登录页错误状态    → 又一个 Frame
  然后在三个 Frame 之间画箭头解释交互

我们：
  登录页 = 一份 Schema
    ├── 默认状态：看到表单
    ├── loading 状态：按钮变为加载中（业务状态切换）
    ├── error 状态：显示错误信息（业务状态切换）
    └── 点击登录按钮 → setState("loading") → 验证 → 跳转
  
  设计师在编辑器中切换状态，实时看到 UI 变化
  在预览模式中，真实体验完整交互流程
```

#### 细节缺口

| 缺口 | 说明 | 优先级 |
|------|------|--------|
| **素材设计器** | 背景/图标拼装能力（Phase 6） | P3 |
| **设计 Token / 主题** | 缺乏全局设计变量（颜色、字号、间距规范） | P2 |
| **动画/过渡** | 状态切换动画不完整（Phase 5） | P2 |
| **版本管理** | 资产版本、diff、回滚 | P3 |
| **响应式设计** | 布局流模式（Flex 内拖拽排序）仍为未实现 | P1 |
| **团队协作** | 多人实时编辑（远期 Phase 7） | P3 |

### 3.4 G4：产品友好 —— 设计即需求表达

#### 已具备的能力

| 能力 | 支撑子系统 | 现状 |
|------|-----------|------|
| 完整交互链路 | 09-interaction-binding | ⚠️ 基础事件已有，链路可视化待做 |
| 页面跳转关系 | Screen.transitions | ✅ 数据结构支持 |
| 数据集管理 | 05-data-driven | ✅ 各场景数据已有 |
| 全景截图 | 05-data-driven | ⚠️ 已设计，实现为远期 |

#### 核心缺口 —— 这是四大目标中差距最大的

**缺口 1：需求文档自动生成**

当前完全没有「Schema → 需求文档」的设计。这是一个全新能力，需要：

```
Schema 数据已包含的信息：
  ├── 页面结构与层级 → 可生成「页面信息架构」
  ├── 事件绑定 → 可生成「交互说明」
  ├── 页面跳转 → 可生成「页面流程图」
  ├── 全局状态 → 可生成「状态说明」
  ├── 数据绑定 → 可生成「数据接口需求」
  └── 全景截图 → 可生成「验收标准（各状态 UI 截图）」

需要新增的能力：
  ├── 需求导出引擎（Schema → Markdown/Word/Confluence）
  ├── 交互流程图自动生成（页面跳转关系可视化）
  ├── 接口需求提取（数据绑定表达式 → API 字段清单）
  └── 验收矩阵生成（数据集 × 状态 × 视口 = 完整截图集）
```

**缺口 2：交互链路可视化**

导航关系图在 09-interaction-binding 中标注为远期，但对产品来说至关重要：
- 页面间跳转关系的全局视图
- 每个页面内的交互事件汇总
- 状态流转图

**缺口 3：产品级标注与描述**

Schema 中缺少需求级元数据：
- 功能描述（每个页面/组件的产品说明）
- 验收标准（每个交互的期望行为描述）
- 优先级标注
- 注释节点（已在 Schema 扩展中提到，但完整设计未展开）

## 五、结论与策略

### 5.1 整体结论

> **当前架构从第一性原理出发，Schema 五维模型 + Operation-based 操作 + MCP 协议的组合，天然且强有力地支撑了 G1-G4 的底层需求。这是一个正确的架构方向。**
>
> **但从「底层能力」到「目标完全实现」之间，还存在「最后一公里」的产品与工程设计缺口，尤其是 G4（产品友好）和 G1（企业级代码质量）方向。**

### 5.2 能力缺口分层

```
┌─────────────────────────────────────────────────────────────┐
│ 第三层：产出层（缺口最大）                                     │
│   需求文档生成、工程化代码导出、交互流程图、验收矩阵              │
│                          ↑                                   │
├─────────────────────────────────────────────────────────────┤
│ 第二层：增强层（部分已设计，待完善）                             │
│   事件链、布局流、设计Token、动画、AI操作分组                    │
│                          ↑                                   │
├─────────────────────────────────────────────────────────────┤
│ 第一层：基础层（已完整设计，实现中）                             │
│   画布、工具栏、属性面板、状态、数据、Props、资产、树、预览、协作   │
└─────────────────────────────────────────────────────────────┘
```

### 5.3 策略建议

1. **继续推进第一层基础能力的实现**——当前 11 个子系统的设计已经非常完整，需要按既有路线图落地
2. **优先补齐第二层中影响「设计即产物」核心体验的能力**——特别是事件链、布局流模式
3. **启动第三层「产出层」的产品设计**——这是从「工具」到「平台」的跃迁，也是最大的差异化壁垒
4. **G4（产品友好）需要专项设计**——当前 11 个子系统中没有专门服务产品经理的子系统，建议新增

## 六、建议新增子系统

基于缺口分析，建议在编辑器总纲中补充以下方向：

### 12-codegen-pro（工程化代码导出）

解决 G1 中「企业级代码」的最后一公里：
- 项目结构模板（Next.js / CRA / Vite 等）
- 组件文件拆分策略
- 样式方案选择（CSS Modules / Tailwind / styled-components）
- 状态管理代码映射
- 路由代码生成
- API 层代码框架

### 13-requirement-export（需求文档导出）

解决 G4 的核心缺口：
- Schema → 结构化需求文档（Markdown / Word / Confluence）
- 页面信息架构自动提取
- 交互链路说明自动生成
- 数据接口需求清单
- 验收截图矩阵
- 交互流程图（可视化页面跳转关系）

### 14-design-tokens（设计 Token 系统）

解决 G3 中设计一致性的需求：
- 全局颜色/字号/间距/圆角变量
- 主题切换（亮/暗）
- Token 在样式编辑中的联动
- 导出时映射到 CSS Variables / Tailwind Config
