# 编辑器设计
结合第一性原理，帮我设计下整个渲染区域：
1. 渲染区域肯定是基于 schema 渲染出来的，在选中节点后，右侧可以对节点做各种样式设计、交互设计、状态设计
2. 渲染区的元素都应该是相对于父级的相对位置，在拖动的时候可以精准放下，类似于 figma 的体验
3. 画的每一个元素比如：span、div 或者组件等，天然就具备一些 props 无论是文字还是样式，尤其是沉淀组件的时候需要标准化暴露 props，表示这个组件有哪些是可以配置的、调整的，不允许调整的不暴露，这样的话，真正在渲染区域渲染就需要传递这些 props 得有一个地方配置？
另外，如果组件在不同状态下，props 传递的应该不一样，也应该支持，比如：一个任务在进行中、已完成等等，页面状态或者组件会有差别？这里就有另一个问题，就是对于设计页面来说有全局状态，也有组件状态，全局就是整个页面通用比如任务状态，多个组件都会受到影响而有不一样的 props，局部就是某个组件的不同状态，比如：按钮的 loading 状态，输入框的错误提示状态，正常状态等等，另外还具备一些通用的状态如：悬浮、点击、正常等等
4. 如：design_docs/images/底部工具栏.png，我们也应该设计一个这样的工具栏，包括：选中元素的那个箭头，可以选择不同元素放到画布里，尤其是方块 div，可以像 figma 点击一样画，然后放到对应的位置，注意所有的操作都得先选中父级容器才可添加

整体编辑器的目标就是：
1. 我可以基于 schema 渲染的东西看到页面
2. 我可以对渲染的页面做非常详细高效的交互，比如：调整某个元素的细节等等
3. 我们这个渲染器和其他的不一样的是，它有数据管理，状态管理（全局、局部等），就像真实的组件一样，这样对于数据的维护、状态的调整管理会比 figma 更灵活更便捷，更高效的看到各种状态下的 UI 表现，也可以结合这些数据定义不同状态下 UI 展示成什么样子，随时可以从全局和局部切换状态来看样式细节和针对性的样式设计。这是一个突破性的创新

这个编辑器应该有很多详细的产品设计内容：
- 整体的编辑器如何交互
- 组件、资产怎么管理
- 单个组件的状态交互如何管理
- 单个组件的静态资源如何管理和编辑，比如：svg 插入图片等
- 整体页面的状态如何管理和操作，包括全局状态等等
- 如何设计完成后就像一个真实的软件一样可以交互
- 工具栏如何设计
- 右侧配置面板如何设计，如：样式、交互、状态、数据等等

# 最根本目标
我想设计即产物：
- 无需对数据做组件分类，交互分析等等，天然就可以轻松转化成企业级代码，包括：代码组织、逻辑抽象、完整交互
- AI 可以通过 mcp 来根据用户需求绘制和调整 UI，不是从页面级别的从0-1 生成，而是通过操作一步步可以把 UI 画出来，可灵活调整和撤销任意步骤任意模块，不像现在的设计软件，每次对话调整都是重新生成。
- 同时对设计师友好，具备组件资产沉淀，基于交互的设计，即不是画一堆 frame 反复解释如何交互，设计的过程中就是正常的软件交互，结合数据mock能力还可以快捷的设计各种状态下的呈现形态和逻辑
- 对产品友好，设计即需求表达，根据设计数据，轻易的可以转化成需求文档，产品所有的交互链路清清楚楚，无需大模型做任何深度分析。
