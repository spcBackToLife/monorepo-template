## 一、产品定位

### 1.1 一句话定位

**不是设计工具——是可视化的产品构建器。**

设计出来的不是一张张图，而是一个**数据流动、状态切换、交互可用**的完整产品原型，
天然可以导出为企业级代码，天然可以转化为需求文档。

### 1.2 四大根本目标

```
G1 设计即代码
   设计稿天然具备完整的组件结构、状态逻辑、交互行为、数据绑定，
   无需额外分析即可导出企业级代码（组件拆分 + 状态管理 + 路由 + API 层）。

G2 AI 增量操作
   AI 通过 MCP 一步步构建和调整 UI（不是从 0 到 1 整页生成），
   每一步可撤销、可调整、可组合——像人一样画，不像生成器一样吐。

G3 设计师友好
   具备组件资产沉淀，基于交互的设计（不是画 N 个 frame 反复解释如何交互），
   设计过程中就是正常的软件交互，结合数据 Mock 能力快速设计各种状态下的表现。

G4 产品友好
   设计即需求表达。根据设计数据，轻松转化为需求文档，
   所有交互链路、状态流转、数据依赖清清楚楚，无需大模型做深度分析。
```

## 二、与传统工具的本质差异

### 2.1 核心对比

| 维度 | Figma / Sketch | 我们 |
|------|---------------|------|
| **设计对象** | 一张张独立的画面（Frame） | 一个完整的产品（页面通过导航连接，数据贯穿全局） |
| **元素本质** | 无语义的矩形/椭圆/文本 | 有语义的 HTML 元素和组件实例 |
| **状态表达** | 每个状态画一份，手动标注差异 | 同一元素定义多状态，状态切换实时预览 |
| **数据** | 无（或手动填充占位文本） | 数据集 + 数据绑定，数据变化驱动 UI 变化 |
| **交互定义** | Prototype 模式连线 + 静态注释 | 事件直接绑定在元素上，设计过程就是交互定义过程 |
| **页面关系** | 独立 Frame，手动画箭头 | 页面通过交互事件（点击跳转）自然形成导航图 |
| **输出** | 静态图片 / 切图 / 标注 | 可运行的代码 + 需求文档 |
| **全局一致性** | 靠人工检查 | 全局状态 + 数据驱动，一处改处处变 |

### 2.2 根本差异：从「画页面」到「建产品」

```
Figma 的心智模型：
  创建画板 → 画 UI → 标注交互（连线、注释）→ 导出切图 → 开发者重新理解并编码
  
  页面之间是孤立的。状态靠画多份。数据靠手动填。交互靠标注解释。

我们的心智模型：
  创建项目 → 设计页面（实时渲染真实 DOM）
           → 定义交互（点击按钮 = 跳转到新页面 = 页面自然创建）
           → 绑定数据（Mock 数据集 + 表达式绑定 = 数据驱动 UI）
           → 切换状态（全局/业务/交互状态 = 一个元素展示多种形态）
           → 预览（就像真实 App 一样操作）
           → 导出（代码 / 需求文档）

  页面通过交互连接。状态在一份 Schema 内多维表达。
  数据驱动 UI 变化。设计过程 = 产品定义过程。
```

### 2.3 这意味着什么？

```
对编辑器的核心要求不是「画得好看」，而是：

1. 让用户在设计过程中自然地定义出产品的完整行为
2. 数据、状态、交互不是事后补充，而是设计的一等公民
3. 设计完成时，产品的结构、行为、数据接口已经完整定义
4. 代码导出不是「翻译图片」，而是「序列化已定义的产品」
5. 需求文档不是「描述图片」，而是「格式化已定义的产品」
```

## 三、核心用户流程

### 3.1 产品构建流程（不是页面设计流程）

```
┌─────────────────────────────────────────────────────┐
│  第一步：创建项目                                      │
│  选择端（PC/Mobile）、选择初始视口                     │
│  → 进入编辑器，看到空白的第一个页面                     │
└──────────┬──────────────────────────────────────────┘
           │
           ▼
┌─────────────────────────────────────────────────────┐
│  第二步：构建页面 UI                                   │
│  在画布上添加元素 → 调整样式 → 设置属性                 │
│  画布实时渲染真实 DOM（所见即所得）                     │
│  AI 也可以通过 MCP 同步操作                            │
└──────────┬──────────────────────────────────────────┘
           │
           ▼
┌─────────────────────────────────────────────────────┐
│  第三步：定义交互 → 连接页面                            │
│  在按钮上绑定 click → navigate: /signup                │
│  → 系统自动创建 signup 页面 → 两页面形成导航关系         │
│  继续设计 signup 页面的 UI → 绑定更多交互 → 页面图展开  │
└──────────┬──────────────────────────────────────────┘
           │
           ▼
┌─────────────────────────────────────────────────────┐
│  第四步：接入数据                                      │
│  创建数据集（正常数据/空数据/异常数据/VIP 数据）         │
│  属性绑定数据表达式 {{ data.user.name }}                │
│  切换数据集 → UI 实时变化 → 设计各场景下的表现           │
└──────────┬──────────────────────────────────────────┘
           │
           ▼
┌─────────────────────────────────────────────────────┐
│  第五步：定义状态变体                                   │
│  全局状态：theme=dark, role=admin                     │
│  业务状态：loading / success / error                   │
│  交互状态：hover / active / disabled                   │
│  每个状态下编辑完整表现（样式+属性+子元素+行为）         │
└──────────┬──────────────────────────────────────────┘
           │
           ▼
┌─────────────────────────────────────────────────────┐
│  第六步：预览验证                                      │
│  进入预览模式 → 像真实 App 一样操作                     │
│  点击按钮跳转页面、状态切换、数据更新                   │
│  所有交互行为真实执行                                   │
└──────────┬──────────────────────────────────────────┘
           │
           ▼
┌─────────────────────────────────────────────────────┐
│  第七步：导出产出                                      │
│  导出代码：React 项目（组件拆分 + 路由 + 状态管理）     │
│  导出需求文档：交互说明 + 状态流转 + 数据接口清单       │
│  导出截图矩阵：状态 × 数据 × 视口 的全景截图           │
└─────────────────────────────────────────────────────┘
```

### 3.2 数据/状态/交互在设计流程中的位置

```
传统工具：          我们：

画 UI ──────────    画 UI ──┬── 同时定义交互（事件绑定）
      │                    ├── 同时绑定数据（表达式）
      │                    ├── 同时定义状态（覆盖集）
      ▼                    │
标注交互 ──────     设计过程 = 产品定义过程
      │                    │
      ▼                    ▼
开发重新理解 ──     直接导出代码 / 需求文档
```

**关键区别：数据/状态/交互不是设计完成后的「补充步骤」，
而是渗透在整个设计过程中的一等公民。**

## 四、核心产品特性

### 特性 1：产品级设计，不是页面级设计

- 一个设计稿 = 一个产品 = 多个页面通过导航连接
- 页面的产生方式：在某个元素上绑定「点击跳转到新页面」→ 新页面自然创建
- 天然形成页面跳转关系图（不需要手动画箭头）
- 全局状态和数据跨页面共享

### 特性 2：数据驱动的设计

- 创建数据集（正常/空/异常/边界等 Mock 数据）
- 属性值通过表达式绑定数据 `{{ data.user.name }}`
- 切换数据集 → 整个 UI 实时变化
- 列表自动根据数据集长度渲染 N 行
- **设计师在设计阶段就能看到所有数据场景下的 UI 表现**

### 特性 3：状态驱动的设计

- 四层状态模型：交互态 / 业务态 / 数据场景 / 全局状态
- 每个状态下可以覆盖：样式、属性、子元素可见性、交互行为
- 一份 Schema 表达所有状态变体（不画 N 份）
- **设计一个组件 = 定义它在所有状态下的完整行为**

### 特性 4：交互即设计

- 事件直接绑定在元素上（click → navigate / setState / openUrl）
- 不是事后用箭头连线，而是设计过程中自然定义
- 预览模式下交互真实可用
- **设计完成时，所有交互链路已完整定义**

### 特性 5：双层元素体系 + 组件资产飞轮

- 原子元素：div / span / button / img / input 等，完全样式自由度
- 组件资产：之前设计的 Schema 片段，作为可复用资产沉淀
- 三级来源：我的组件 → 团队组件 → 官方组件
- **飞轮效应：每次设计行为都在生产可复用资产**

### 特性 6：AI 增量操作

- AI 调用标准化 Operation API，不是生成像素图
- 增量修改：指哪改哪，每步可撤销
- 与手动操作完全对等——用同一套 Operation 系统
- **AI 和人类协作编辑同一份 Schema**

### 特性 7：设计即代码，设计即文档

- Schema 使用 CSS 规范描述样式，1:1 对应真实代码
- 导出 React/Vue/Flutter 项目（含组件拆分、路由、状态管理、API 层）
- 导出需求文档（交互说明、状态流转、数据接口清单、验收截图）
- **不需要二次翻译——设计就是最终产出的源头**

## 五、设计稿数据模型（概念层）

```
DesignProject                          ← 一个产品
├── id, name, createdAt, updatedAt
├── platform: "pc" | "mobile"
├── defaultDevice: { name, width, height }
├── globalStates: GlobalStateVar[]     ← 全局状态变量（theme, role, locale...）
├── screens: Screen[]                  ← 产品的所有页面
│   └── Screen
│       ├── id, name
│       ├── rootNode: ComponentNode    ← 组件树（结构 + 样式 + 状态 + 交互）
│       ├── dataSets: DataSet[]        ← 数据集（Mock 数据）
│       └── transitions: Transition[]  ← 页面跳转关系（由交互事件自然形成）
├── componentAssets: ComponentTemplate[] ← 组件资产（可复用 Schema 片段）
│   └── ComponentTemplate
│       ├── id, name, description, category, tags
│       ├── schema: ComponentNode
│       ├── propDefinitions: PropDef[] ← 标准化暴露的可配置属性
│       └── scope: "project" | "team" | "global"
├── viewportPresets: Viewport[]        ← 视口预设（快速切换设备）
└── globalStyles: { ... }
```

> 完整 TypeScript 类型定义见技术方案文档中 `@globallink/design-schema` 的接口设计。
> 组件资产模型的产品设计见 [组件资产体系](./component-assets.md)。
