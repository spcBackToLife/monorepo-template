## 一、第一性原理：为什么需要全景视图？

### 1.1 V1 的痛点反思

```
V1 全景视图的两个核心问题：

  问题 1: 全景不是独立页面
    V1 做法：全景直接替换画布区域，左右面板和工具栏仍在
    → 空间局促，信息密度高但展示面积小
    → 不像一个独立的"走查模式"，而是一个"画布的另一种渲染"
    → 用户反馈："全景应该是个单独的页面，纯粹插入中间的交互体验很差"

  问题 2: 编辑态看不到状态预览
    选中登录按钮后，右侧面板显示 默认/悬停/按下/聚焦/禁用 和自定义状态
    但只能一次看一个状态——点一个 tab 看一眼，再点另一个 tab 看一眼
    无法同时对比，选中 Root 节点时更看不到自定义状态的实际表现

  本质原因：V1 试图用一个功能（全景模式）解决两种不同的需求
    · 需求 A: "我想快速看看各状态长什么样"  → 轻量级，不想离开编辑态
    · 需求 B: "我想仔细走查对比所有状态"    → 沉浸式，需要全屏空间
```

### 1.2 两种认知需求 → 两个 Feature

```
从第一性原理出发：

  设计师在编辑组件时的认知流程是：
    构建 → 微调 → 验证（不断循环）

  "快速看看" = 验证的低成本前置检查
    · 不打断编辑流
    · 不离开当前上下文
    · 只需要一个缩略图级别的视觉确认
    → 解决方案: 右侧面板的"状态预览缩略图条"

  "仔细走查" = 验证的高质量最终检查
    · 需要全屏空间来并排对比
    · 需要看到组件在真实尺寸下的表现
    · 需要分类展示（交互态 vs 自定义态）
    · 可能需要分享给团队 review
    → 解决方案: 独立路由的"全景页面"

  V1 的错误：把这两种需求揉进一个"替换画布"的模式里
    → 对"快速看看"来说太重（需要进入全景模式）
    → 对"仔细走查"来说又不够（空间受限、不能分享URL）

  V2 的核心设计决策：
    拆成两个互补的 Feature:
    ┌──────────────────────┐  ┌──────────────────────────┐
    │ 状态预览缩略图条      │  │ 全景独立页面              │
    │                      │  │                          │
    │ · 在右侧面板中       │  │ · 独立路由 /panorama      │
    │ · 横向缩略图排列     │  │ · 全屏沉浸展示           │
    │ · 不离开编辑态       │  │ · URL 可分享/可收藏       │
    │ · 点击切换编辑状态   │  │ · 点击跳回编辑+应用状态   │
    │ · 快速浏览           │  │ · 深度走查对比            │
    └──────────────────────┘  └──────────────────────────┘
```

### 1.3 核心洞察

```
全景视图不只是"查看所有状态"的工具——
它是设计质量保障的核心环节。

设计流程的三个阶段对应三种视图：
  ┌──────────────────────────────────────────────────────────┐
  │                                                          │
  │  编辑模式 (Edit)                                         │
  │  · 单画布，单状态切片                                    │
  │  · 所有编辑工具可用                                      │
  │  · 适合：精细调整某一个状态下的 UI                       │
  │  · 辅助: 右侧面板缩略图条提供快速状态预览               │
  │                                                          │
  │  预览模式 (Preview)                                      │
  │  · 单画布，可交互                                        │
  │  · 事件真实执行，状态动态切换                             │
  │  · 适合：验证交互流程是否正确                             │
  │                                                          │
  │  全景模式 (Panorama) — 独立路由页面                      │
  │  · 多画布网格，所有状态平铺                               │
  │  · 全屏沉浸，分类展示                                    │
  │  · 适合：走查所有状态是否设计完备                         │
  │  · URL 可分享，团队协作 review                           │
  │                                                          │
  └──────────────────────────────────────────────────────────┘
```

## 二、全景模式的两个层级

### 2.1 组件全景（Component Panorama）

```
选中一个组件 → 进入全景页 → 看到该组件在所有状态下的表现

适用场景：
  · 一个按钮的 default / hover / pressed / disabled / loading / success
  · 一个输入框的 default / focus / filled / error / disabled
  · 一个卡片的 collapsed / expanded

展示方式（分类排列）：

  ── 交互状态 ──────────────────────────────────────
  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐
  │      │  │      │  │      │  │      │  │      │
  │ 默认 │  │ 悬浮 │  │ 按下 │  │ 聚焦 │  │ 禁用 │
  │      │  │      │  │      │  │      │  │      │
  └──────┘  └──────┘  └──────┘  └──────┘  └──────┘

  ── 自定义状态 ────────────────────────────────────
  ┌──────┐  ┌──────┐
  │      │  │      │
  │加载中│  │ 成功 │
  │      │  │      │
  └──────┘  └──────┘

  特点：
  · 每个格子 = 该组件在对应状态下的 SchemaRenderer 渲染
  · 格子下方标注状态名
  · 交互态和自定义态用分隔线和标题区分
  · 格子大小 = 组件实际尺寸（component mode scale = 1.0）
  · 点击某个格子 → 跳回编辑模式并自动切到该状态
  · 可通过筛选器只看交互态/自定义态/全部
```

### 2.2 页面全景（Page Panorama）

```
不选中组件（或选中 Root） → 进入全景页 → 看到该页面在所有领域态下的表现

维度：
  · 领域态变量值（如 pagePhase: loading / loaded / error）
  · 数据场景（如 正常用户 / 空数据 / VIP用户）
  · 视口尺寸（如 iPhone SE / iPhone 15 Pro / iPad）（远期）

展示方式：
  ┌──────────────────────────────────────────────┐
  │                                              │
  │  pagePhase = loading                         │
  │  ┌──────────────────┐                        │
  │  │                  │                        │
  │  │   正在加载...    │                        │
  │  │                  │                        │
  │  └──────────────────┘                        │
  │  加载中                                      │
  │                                              │
  │  pagePhase = loaded                          │
  │  ┌──────────────────┐                        │
  │  │  [头像] 张三     │                        │
  │  │  前端工程师      │                        │
  │  └──────────────────┘                        │
  │  已加载                                      │
  │                                              │
  │  pagePhase = error                           │
  │  ┌──────────────────┐                        │
  │  │    加载失败      │                        │
  │  │  [重新加载]      │                        │
  │  └──────────────────┘                        │
  │  加载失败                                    │
  └──────────────────────────────────────────────┘

  特点：
  · 每个格子 = 一个完整页面渲染（缩放到适配，默认 scale=0.35）
  · 格子下方标注领域态的值
  · 多领域态变量时使用笛卡尔积矩阵（行/列标签）
  · 支持鼠标滚轮缩放（调整格子大小）
  · 点击格子 → 跳回编辑模式，自动切到对应领域态组合
```

## 三、产品交互设计

### 3.1 Feature A: 状态预览缩略图条（编辑态内）

```
位置: 右侧属性面板 → StateContextBar 下方
触发: 选中任意组件时自动显示（可折叠）

┌─────────────────────────────────────────────────┐
│  默认  │ 悬停  │  按下  │  聚焦  │ 禁用         │  ← 现有交互态 Tab
├─────────────────────────────────────────────────┤
│  自定义  [login-success] [login-failure] [+]     │  ← 现有自定义状态行
├─────────────────────────────────────────────────┤
│  ▾ 状态预览                          ↗ 全景对比  │  ← 新增：可折叠标题
│ ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐   │
│ │ 默认 │ │ 悬停 │ │ 按下 │ │ 聚焦 │ │ 禁用 │   │  ← 缩略图 80x60px
│ └──────┘ └──────┘ └──────┘ └──────┘ └──────┘   │     横向可滚动
│ ┌──────────┐ ┌──────────┐                        │
│ │ success  │ │ failure  │                        │  ← 自定义态缩略图
│ └──────────┘ └──────────┘                        │
├─────────────────────────────────────────────────┤
│  属性面板内容...                                  │

交互行为：
  · 缩略图 = SchemaRenderer 在 scale=0.1 下的实时渲染
  · 当前激活的状态 → 缩略图有蓝色高亮边框
  · 点击缩略图 → 切换编辑到该状态（等同于点击上方 Tab）
  · Hover 缩略图 → 显示 Tooltip 放大预览（2倍大小）
  · 点击"↗ 全景对比" → 跳转全景路由页
  · 折叠/展开状态持久化到 localStorage

性能保障：
  · 缩略图 pointerEvents: none，不影响交互
  · Schema 变更后 300ms debounce 重渲染
  · 最多展示 8 个缩略图，超出显示"+N 更多"
  · React.memo + schema 版本号避免无效重渲染
```

### 3.2 Feature B: 全景独立路由页

```
入口：
  入口 1: 顶部工具栏 → [全景] 按钮
    · 有选中组件 → 跳转 /editor/:id/panorama?node=<nodeId>（组件全景）
    · 无选中组件 → 跳转 /editor/:id/panorama（页面全景）
  入口 2: 快捷键 Cmd+Shift+P / Ctrl+Shift+P
  入口 3: 缩略图条 → "↗ 全景对比" 链接
  入口 4: 右键组件 → "查看全部状态"

路由设计：
  /editor/:id/panorama          → 页面全景
  /editor/:id/panorama?node=xxx → 组件全景

  URL 可分享: 团队成员打开同一 URL 看到同样的全景视图
  浏览器前进/后退: 天然可用，不需要额外处理
  收藏夹: 可以收藏某个组件的全景页快速访问

退出方式：
  · 点击"← 返回编辑器"按钮
  · 按 Esc 键
  · 浏览器后退按钮
  · 点击格子 → 应用状态后自动返回编辑器
```

### 3.3 全景页控制条

```
┌─────────────────────────────────────────────────────────────────────┐
│ [← 返回编辑器]  │  🔲 组件全景: LoginButton  │  7 个状态           │
│                 │  [全部] [交互态] [自定义态]  │  缩放: ━━━○━━ 80%  │
└─────────────────────────────────────────────────────────────────────┘

[全部/交互态/自定义态]: Segmented 筛选器
  · 全部: 显示所有状态，用分隔线区分交互态和自定义态
  · 交互态: 只显示 default/hover/pressed/focus/disabled
  · 自定义态: 只显示用户自定义的业务状态

缩放滑块:
  · 组件模式默认 100%（实际大小）
  · 页面模式默认 35%（缩小到适配）
  · 范围 20% ~ 150%
```

## 四、与 Figma Variants 的对比

```
┌───────────────────┬────────────────────────┬────────────────────────┐
│ 维度               │ Figma Variants          │ 我们的全景模式 V2       │
├───────────────────┼────────────────────────┼────────────────────────┤
│ 数据来源           │ 手动画 N 个副本         │ 自动从 Schema 生成     │
│ 维护成本           │ 改一处改 N 处           │ 改一处全景自动更新     │
│ 完整性             │ 容易漏画某个状态        │ 系统确保覆盖所有状态   │
│ 可交互性           │ 静态图                  │ 实时 DOM，支持 hover   │
│ 数据维度           │ 无                      │ 数据场景 x 状态 x 视口 │
│ 编辑态预览         │ 无（需切到 Variant）    │ 缩略图条实时预览       │
│ 分享               │ 需截图                  │ URL 直接分享           │
│ 进入编辑           │ 直接编辑               │ 点击格子跳转到编辑态    │
│ 导出               │ 手动截图               │ 一键导出全景截图       │
└───────────────────┴────────────────────────┴────────────────────────┘

核心优势：零维护成本 + 编辑态内联预览。
  Figma 设计师要手动维护每个 Variant 的一致性。
  我们的全景视图是 Schema 的"投影"——改了 Schema，全景自动更新。
  且我们在编辑态右侧面板就能看到缩略图对比，不需要离开编辑。
```

## 五、用户场景示例

### 场景 1：按钮设计走查（缩略图 + 全景联合使用）

```
设计师完成了 LoginButton 的所有状态设计。

第一步: 缩略图快速验证
  选中按钮 → 右侧面板底部出现缩略图条
  一排小图: 默认(紫色) / 悬浮(亮紫) / 按下(深紫) / 禁用(灰色) / 加载中 / 成功
  → 快速扫一眼，大部分状态看起来OK

第二步: 全景深度走查
  发现禁用态缩略图颜色不太对 → 点"↗ 全景对比"进入全景页
  全屏看到 6 个状态并排，每个都是真实尺寸
  确认问题：禁用态的文字颜色对比度不够！
  点击"禁用"格子 → 跳回编辑模式，状态自动切到 disabled → 修改颜色
```

### 场景 2：页面多状态走查

```
设计师完成了主页的三种页面状态（loading/loaded/error）。
进入页面全景 → URL: /editor/abc123/panorama
全屏看到三个完整页面并排

发现问题：错误页没有返回按钮！
点击错误页格子 → 跳回编辑模式 → pagePhase 切到 error → 添加返回按钮
```

### 场景 3：团队 Review

```
设计师完成按钮的所有状态，想让同事 review。
进入按钮全景页 → 复制 URL: /editor/abc123/panorama?node=btn_001
```

## 六、设计决策记录

| 决策 | 选择 | 理由 |
|------|------|------|
| V2 vs V1 的核心区别？ | 拆成两个 Feature（缩略图条 + 全景路由页） | V1 用一个功能解决两种需求，导致两头不讨好 |
| 全景是独立路由还是 overlay？ | **独立嵌套路由** `/editor/:id/panorama` | URL 可分享/可收藏，浏览器前进后退自然工作 |
| 缩略图条放在哪里？ | 右侧面板 StateContextBar 下方 | 与状态切换器紧邻，操作连贯 |
| 缩略图用截图还是实时渲染？ | **实时 DOM**（SchemaRenderer scale=0.1） | 零维护，schema 变则缩略图变；截图有延迟和同步问题 |
| 全景格子可以编辑吗？ | 不可以（只读） | 多实例同时编辑会冲突；点击跳转编辑更安全 |
| 组件全景的格子大小？ | 实际尺寸（scale=1.0） | 组件小，1:1 展示更清晰 |
| 页面全景的格子大小？ | 缩放（默认 scale=0.35） | 页面大，需缩小才能并排 |
| 交互态和自定义态怎么排列？ | 分区展示，用标题和分隔线区分 | 两类状态语义不同，混排容易混淆 |
| 最大格子数？ | 50 个，超出给警告 | 性能和信息密度的平衡 |
| 缩略图条最多显示几个？ | 8 个，超出显示"+N 更多" | 右侧面板空间有限 |

## 七、边界情况

| 场景 | 预期行为 |
|------|---------|
| 组件没有自定义状态 | 缩略图条只显示 5 个交互态；全景页只显示交互状态区域 |
| 组件没有任何状态覆盖 | 所有缩略图/格子看起来一样，提示"该组件未定义状态差异" |
| 页面没有领域态变量 | 页面全景只显示单个"当前状态"格子，提示"添加领域态变量以查看更多" |
| 组合数 > 50 | 弹出警告"组合过多（N 个），建议减少维度"，仅显示前 50 个 |
| 全景页面 URL 中 node 不存在 | 显示"组件已被删除"，3 秒后自动返回编辑器 |
| Schema 在全景页打开期间被修改（AI 或协作者） | 所有格子实时更新（MobX 响应式），因为 EditorShell 和全景页共享 store |
| 缩略图条展开时有 20+ 状态 | 只显示前 8 个 + "+12 更多"badge，badge 可点击跳转全景 |
| 编辑器窗口很小（< 800px） | 缩略图条自动折叠，只显示"查看状态预览"按钮 |
| 全景页缩放到很小（< 20%） | 限制最小 20%，格子内文字可能模糊但布局仍清晰 |
| 快捷键冲突 | Cmd+Shift+P 在预览模式中无效（优先级低于预览模式的快捷键） |
