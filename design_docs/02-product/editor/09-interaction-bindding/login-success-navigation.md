# 登录成功后跳转主页 — 产品交互方案

> **需求**：登录按钮点击 → 发送请求 → 成功时 Toast 提示 → 跳转主页
>
> **现状**：已实现 Toast（通过 setState + autoRevert），缺少跳转动作

## 一、第一性原理分析：这个需求的本质是什么？

### 1.1 从现状推导

```
当前登录按钮的事件链（截图所示）：

  触发: 点击 (Click)
  动作链:
    1. 发送请求 (apiRequest)
       ├── ✅ 成功时执行:
       │   └── 设置状态 → Root → "login-succsess"  (3000ms 自动回退)
       │       → 画布上出现成功 Toast → 3 秒后消失
       └── ❌ 失败时执行:
           └── 设置状态 → Root → "login-failure"  (3000ms 自动回退)
               → 画布上出现错误 Toast → 3 秒后消失

缺失的部分：
  成功路径停留在了"看到 Toast"，没有后续的页面跳转。
  真实产品中，登录成功后应该自动跳转到主页。
```

### 1.2 本质推导：反馈与跳转的时序关系

```
Q: 为什么不能直接跳转，要先 Toast？
A: 因为"跳转"是破坏性操作——用户会失去当前页面的上下文。
   Toast 是跳转前的"预告"：
     → 让用户知道"登录成功了"
     → 给用户一个心理预期"即将离开这个页面"
     → 真实产品几乎都遵循这个模式（微信、支付宝、各种 SaaS）

Q: Toast 和跳转的时间间隔多长合适？
A: 从第一性原理推：
   · 太短（< 300ms）→ Toast 来不及被用户感知，等于没有反馈
   · 太长（> 2000ms）→ 用户等得不耐烦，感觉"怎么还不跳"
   · 最佳区间：500ms ~ 1500ms
   · 推荐值：1000ms（1秒）—— 足够看到 Toast，又不至于等太久

   注意：这里的 delay 不是 Toast 的 autoRevert 时间（3000ms），
   而是"看到 Toast 后多久开始跳转"。
   跳转后 Toast 自然消失（页面都变了）。

Q: 跳转时 Toast 还在吗？
A: 不重要——跳转本身就是最强的"成功信号"。
   页面切换动画（slideLeft/fadeIn）替代了 Toast 作为视觉反馈。
   所以不需要等 Toast 的 3000ms 结束再跳转。
```

### 1.3 核心洞察

```
登录成功后的完整交互链路应该是：

  请求成功
    → ① 切换到"登陆成功"状态（Toast 出现）—— 即时反馈
    → ② 等待 1 秒 —— 让用户看到反馈
    → ③ 跳转到主页（带转场动画）—— 目标达成

三个动作在成功子链路中顺序执行。

这完全符合现有事件系统的"事件链"能力：
  onSuccess: [setState, delay, navigate]
  无需发明任何新概念。
```

## 二、产品方案设计

### 2.1 目标事件链配置

```
修改后的登录按钮事件配置：

  触发: 点击 (Click)
  动作链:
    1. 发送请求 (apiRequest) → 用户登录
       ├── ✅ 成功时执行 (3 个动作):
       │   ① 设置状态 → Root → "login-succsess"  (3000ms 自动回退)
       │      → 画布切换到"登陆成功"状态，Toast 出现
       │   ② 延时等待 → 1000ms
       │      → 让用户看到 Toast 约 1 秒
       │   ③ 跳转页面 → "主页"  (动画: slideLeft)
       │      → 页面向左滑出，进入主页
       │
       └── ❌ 失败时执行 (1 个动作，不变):
           ① 设置状态 → Root → "login-failure"  (3000ms 自动回退)
              → 画布切换到"登陆失败"状态，错误 Toast 出现
              → 3 秒后自动回退，用户留在登录页可重试
```

### 2.2 成功路径的时间线

```
 t=0ms      用户点击登录按钮
   │
   │        发送 API 请求（Mock 延迟 800ms）
   │
 t=800ms    请求返回成功
   │
   ├──①──  setState(Root, "login-succsess", autoRevert: 3000ms)
   │        → Toast "登录成功" 从画布顶部出现
   │
 t=800ms    delay(1000ms) 开始计时
   │
   │        用户看到 Toast，理解"登录成功了"
   │
 t=1800ms   delay 结束
   │
   ├──③──  navigate("主页", animation: "slideLeft")
   │        → 页面向左滑出，主页从右侧进入
   │        → Toast 随旧页面一起离开（或被新页面覆盖）
   │
 t=2100ms   转场动画完成，用户到达主页
```

### 2.3 失败路径的时间线（不变）

```
 t=0ms      用户点击登录按钮
   │
   │        发送 API 请求（Mock 延迟 500ms）
   │
 t=500ms    请求返回失败
   │
   ├──①──  setState(Root, "login-failure", autoRevert: 3000ms)
   │        → 错误 Toast "用户名或密码错误" 出现
   │
 t=3500ms   autoRevert 触发
   │        → Root 回到 default 状态
   │        → 错误 Toast 消失
   │        → 用户可以修改输入后重试
```

### 2.4 为什么成功路径选 delay 1000ms 而非等 Toast 3000ms 结束

```
对比两种方案：

  方案 A: Toast 3000ms 结束后再跳转
  ─────────────────────────────────
  · Toast 出现 → 等 3 秒 → Toast 消失 → 再跳转
  · 用户体验：等太久了！登录成功了为什么还让我看 3 秒空白？
  · 感觉像卡住了
  · 总耗时：800ms(请求) + 3000ms(toast) + 300ms(转场) = 4.1s

  方案 B（推荐）: Toast 出现后 1 秒就跳转
  ──────────────────────────────────────────
  · Toast 出现 → 用户看到 → 1 秒后页面切换
  · 用户体验：流畅！看到成功提示，紧接着就到主页了
  · 感觉很"利落"
  · 总耗时：800ms(请求) + 1000ms(等待) + 300ms(转场) = 2.1s

  选择方案 B。

  注：Toast 的 autoRevert 3000ms 在跳转后不再有意义——
  页面已经切走了，旧页面的状态回退无人看到。
  但 autoRevert 应该保留，以防万一跳转失败时 Toast 仍能自动消失。
```

## 三、编辑器操作步骤（设计师视角）

### 3.1 前置条件

```
✅ 已存在"主页"屏幕（如果没有，需要先创建）
✅ 登录按钮已有 apiRequest 事件且成功链路中有 setState
✅ Toast UI 已在"login-succsess"状态的画布上绘制完毕
```

### 3.2 操作步骤

```
步骤 1: 选中登录按钮，打开行为面板

  右侧属性面板 → 行为 Tab
  找到已有的 "点击 (Click)" 事件 → 展开编辑

步骤 2: 在成功链路中添加"延时等待"动作

  ✅ 成功时执行 区域 → 点击 [+ 添加]
  行为类型选择: [延时等待 ▾]
  时长: 1000 ms

  此时成功链路:
    1. 设置状态 → Root → "login-succsess" (3000ms)
    2. 延时等待 → 1000ms                             ← 新增

步骤 3: 在成功链路中添加"跳转页面"动作

  ✅ 成功时执行 区域 → 再次点击 [+ 添加]
  行为类型选择: [跳转页面 ▾]
  目标页面: [主页 ▾]         ← 从项目屏幕列表中选择
  切换动画: [向左滑入 ▾]     ← 标准的前进跳转动画

  最终成功链路:
    1. 设置状态 → Root → "login-succsess" (3000ms)
    2. 延时等待 → 1000ms                             ← 新增
    3. 跳转页面 → 主页 (slideLeft)                   ← 新增

步骤 4: 保存事件配置

  点击 [保存] 按钮
```

## 四、预览验证

### 4.1 成功场景验证

```
前提: Mock 场景切换到"登录成功"

1. 进入预览模式 (Cmd+P)
2. 点击"登录"按钮
3. 观察效果：
   · 按钮发送请求（如有 loading 状态则显示加载中）
   · ~800ms 后：✅ 绿色 Toast "登录成功" 从顶部出现
   · ~1000ms 后：页面开始向左滑出
   · ~1300ms 后：主页完全展示
4. 确认主页内容正确渲染

验证要点：
  ☑ Toast 出现后有明显的停留时间（~1秒）让用户感知
  ☑ 页面转场动画流畅
  ☑ 主页内容完整显示
```

### 4.2 失败场景验证

```
前提: Mock 场景切换到"密码错误"

1. 在预览模式中点击"登录"按钮
2. 观察效果：
   · ~500ms 后：❌ 红色 Toast "用户名或密码错误" 出现
   · 页面保持不动（不跳转！）
   · ~3000ms 后：Toast 自动消失
3. 用户可以修改输入重新尝试

验证要点：
  ☑ 失败时不触发跳转
  ☑ Toast 正常显示并自动消失
  ☑ 页面仍停留在登录页
```

## 五、设计决策记录

| 决策 | 选择 | 理由 |
|------|------|------|
| Toast 和跳转的间隔？ | 1000ms (delay) | 500ms 太快看不清 Toast，2000ms 太慢感觉卡顿，1000ms 刚好 |
| 跳转动画选什么？ | slideLeft (向左滑入) | 符合"向前推进"的心智模型：登录→主页 = 前进 |
| 失败路径要跳转吗？ | 不跳转 | 失败后用户需要修改输入重试，跳走会增加操作成本 |
| Toast autoRevert 和 delay 冲突吗？ | 不冲突 | autoRevert 是 Toast 的安全回退机制；delay 控制的是跳转时机。跳转后 autoRevert 自然无效但无害 |
| 需要创建"主页"屏幕吗？ | 是，需提前创建 | navigate 需要有一个确定的 targetScreenId，没有目标页面则事件卡片会显示 ⚠️ 警告 |
| 是否需要新的 Action 类型？ | 否 | 完全复用现有的 setState + delay + navigate 三个 Action，事件链顺序执行即可 |

## 六、边界情况

| 场景 | 预期行为 |
|------|---------|
| 主页屏幕不存在 | 事件卡片显示 ⚠️ "目标页面不存在"，预览时跳转无效，Toast 正常显示 |
| 用户在 Toast 期间快速重复点击 | apiRequest 的 debounce 机制阻止重复请求 |
| 跳转后按返回 | 预览导航栈支持 [← 后退]，回到登录页 |
| 网络超时（Mock 场景） | 走 onFailure 路径，不跳转 |
| delay 期间用户手动切换页面 | 允许——delay 定时器在页面切换后可能失效或被清理（引擎需处理） |

## 七、与代码导出的映射

```typescript
// 导出的 React 代码（预期）
async function handleLoginClick() {
  try {
    const response = await fetch("/api/auth/login", {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({ username, password }),
    });
    const data = await response.json();

    if (response.ok) {
      // ① 显示成功 Toast（通过状态切换）
      setPageState("login-success");
      // ② 等待 1 秒
      await new Promise(resolve => setTimeout(resolve, 1000));
      // ③ 跳转到主页
      router.push("/home", { transition: "slideLeft" });
    } else {
      // ① 显示失败 Toast（通过状态切换）
      setPageState("login-failure");
      // 3 秒后自动回退
      setTimeout(() => setPageState("default"), 3000);
    }
  } catch (error) {
    setPageState("login-failure");
    setTimeout(() => setPageState("default"), 3000);
  }
}
```

## 八、总结

```
一句话方案：
  在登录按钮的 apiRequest 成功子链路中，
  在已有的 setState (Toast) 之后，追加 delay(1000ms) + navigate("主页")。

不需要任何新概念、新组件、新系统。
完全复用现有的事件链 + setState + delay + navigate 四个已有能力。
这就是"状态 + 事件链"架构的价值——
  新的交互链路 = 已有 Action 的新组合。

修改点：
  · Schema: 登录按钮 events[0].actions[0].onSuccess 数组从 1 项变为 3 项
  · 编辑器: 无需修改（事件链编辑 UI 已支持多动作）
  · 引擎: 无需修改（事件链顺序执行 + delay + navigate 均已实现）
  · 纯配置变更，零代码改动。
```
