## 一、第一性原理：组件树解决什么根本问题？

### 1.1 本质推导

```
Schema 是一棵树 —— ComponentNode 嵌套 ComponentNode
画布是这棵树的"视觉投影" —— 把树渲染为可见的 UI

但视觉投影有天然缺陷：
  · 重叠的元素看不清层级关系（谁在前谁在后？）
  · 嵌套的元素分不清父子关系（这个 div 是谁的子元素？）
  · 隐藏的元素看不见（被 overflow:hidden 裁切的、opacity:0 的）
  · 极小的元素点不到（1px 的分隔线、空的 flex 容器）

组件树 = Schema 树的"结构投影" —— 把树渲染为可导航的文本列表

画布回答"长什么样"，组件树回答"是什么结构"。
两者互补，缺一不可。
```

### 1.2 页面管理的本质

```
一个 DesignProject 包含多个 Screen（页面）
每个 Screen 有自己的 rootNode（组件树根节点）

页面列表的作用：
  · 在多个 Screen 之间切换当前编辑的页面
  · 管理页面的增删排序
  · 预览页面间的跳转关系（远期）

页面列表 + 组件树 = 项目结构的完整导航系统
```

### 1.3 设计原则

```
1. 1:1 映射 → 组件树的每个节点精确对应一个 ComponentNode，不合并、不隐藏
2. 双向联动 → 画布选中 ↔ 组件树选中 实时同步
3. 拖拽直觉 → 拖拽节点 = 移动元素的父子关系和顺序，与画布效果一致
4. 信息密度 → 一行展示：图标 + 类型 + 名称 + 文本预览，不需要展开就能识别
5. 大树可用 → 500 个节点仍流畅（虚拟滚动 + 懒展开）
```

## 三、组件树详细设计

### 3.1 节点行结构

每个树节点渲染为一行，包含以下元素：

```
一行节点的结构（高度 28px）：

  ┌──────────────────────────────────────────────────────────────┐
  │ [缩进] [▾] [🏷] [名称/文本预览]              [👁] [🔒]      │
  └──────────────────────────────────────────────────────────────┘
    ↑        ↑    ↑    ↑                           ↑     ↑
   层级    折叠  类型  节点标识                    可见  锁定
   缩进    箭头  图标  （hover 时显示右侧操作）

各部分说明：

  缩进：每层级 16px，从第 0 级开始
         层级 0 (rootNode): 0px 缩进
         层级 1: 16px
         层级 2: 32px
         ...

  折叠箭头 [▾/▸]：
    · 有子节点 → 显示 ▾ (展开) 或 ▸ (折叠)
    · 无子节点 → 不显示（占位留空，保持对齐）
    · 点击箭头 → 切换展开/折叠

  类型图标 [🏷]：
    · 16×16px，按 NodeType 显示不同图标
    · 详见 3.2 图标系统

  名称/文本预览：
    · 优先显示 node.name（如 "div.header"）
    · 如果没有 name → 显示 node.type（如 "div"）
    · 如果是文本元素 → 追加文本预览（灰色，截断至 15 字）
      例：📝 span "登录按钮文案..."
    · 如果是组件实例 → 显示 📦 + 模板名
      例：📦 LoginForm

  右侧操作 [👁] [🔒]：
    · 默认隐藏，hover 该行时才显示
    · 👁 → 切换可见性（visible: true/false）
    · 🔒 → 切换锁定（locked: true/false）
    · 隐藏的节点 → 名称显示为删除线 + 灰色
    · 锁定的节点 → 🔒 图标常驻显示（非 hover 也可见）
```

### 3.2 图标系统

```
按 NodeType 分配图标（16×16px，单色线条风格）：

容器类型：
  📦  div / section / main → 方盒图标
  📦  nav → 导航图标（汉堡菜单形）
  📦  header → 顶栏图标
  📦  footer → 底栏图标
  📦  ul / ol → 列表图标
  📦  li → 列表项图标

内容类型：
  📝  span / p / h1 / h2 / h3 → 文字图标（T 或 Aa）
  📷  img → 图片图标
  🔗  a → 链接图标

交互类型：
  🔘  button → 按钮图标
  📥  input / textarea → 输入框图标
  📋  select → 下拉图标
  🎬  video → 视频图标

特殊类型：
  📦  component:* → 组件实例图标（紫色菱形，区别于普通容器）
  📌  annotation → 注释图标（仅在组件树可见，画布覆盖层渲染）

图标颜色：
  · 默认：#666（灰色）
  · 选中：#0D99FF（蓝色）
  · 组件实例：#9747FF（紫色）
  · 注释：#FF9500（橙色）
```

### 3.3 选中与高亮状态

```
节点行的视觉状态：

  ┌──────────────────────────────────────┐
  │ 正常                                  │  背景：透明
  │ hover                                │  背景：#F5F5F5
  │ 选中                                  │  背景：#E8F4FD，左侧 2px 蓝色竖线
  │ 包含选中子节点                         │  左侧 2px 浅蓝色竖线（半透明）
  │ 拖拽 hover（可放置目标）               │  蓝色虚线边框
  │ 隐藏 (visible=false)                  │  文字灰色 + 删除线，图标半透明
  │ 锁定 (locked=true)                    │  🔒 图标常驻，其余正常
  └──────────────────────────────────────┘

选中联动：
  · 画布上选中元素 → 组件树对应行高亮 + 自动滚动到可见区域
  · 组件树中点击行 → 画布对应元素高亮 + 如果不在视口内则滚动画布
  · 多选 (Cmd+Click) → 组件树中多行同时高亮
  · 框选 → 组件树中框选范围内的节点全部高亮
```

### 3.4 展开/折叠策略

```
初始状态：
  · 根节点 (rootNode) → 展开
  · 第一层子节点 → 展开
  · 更深层级 → 折叠
  · 理由：展示整体结构概览，不过度暴露细节

自动展开规则：
  · 选中某节点 → 自动展开其所有祖先节点（确保选中节点可见）
  · 搜索命中 → 自动展开命中节点的路径
  · 拖拽进入折叠节点 → 悬停 500ms 后自动展开

用户操作：
  · 点击箭头 → 切换该节点的展开/折叠
  · Alt + 点击箭头 → 递归展开/折叠所有后代节点
  · 右键 → "全部展开" / "全部折叠"（作用于该节点的子树）

记忆：
  · 展开状态保存在 sessionStorage
  · 页面切换后恢复（同一会话内）
```

## 四、拖拽调整层级

这是组件树最复杂的交互——拖拽节点来改变元素的父子关系和兄弟顺序。

### 4.1 拖拽的三种意图

```
拖拽一个节点时，可能的意图有三种：

  1. 调整兄弟顺序（在同一父节点内上下移动）
     例：把 input.password 移到 input.username 的前面

  2. 移入另一个容器（改变父节点）
     例：把 button.submit 从 div.form 移到 div.footer

  3. 移出当前容器（提升层级）
     例：把 span "登录" 从 button.submit 中拖出到 div.form 中

系统需要通过拖拽位置来判断用户的意图。
```

### 4.2 放置指示器设计

```
拖拽过程中，目标位置用三种视觉指示器表达：

指示器 A：插入线（兄弟顺序调整）
  ┌──────────────────────────┐
  │  📦 div.header           │
  ╠══════════════════════════╣ ← 蓝色水平线 (2px)，带圆点指示
  │  📦 div.form             │    表示"插入到 header 和 form 之间"
  │  📦 div.footer           │
  └──────────────────────────┘

指示器 B：容器高亮（移入容器）
  ┌──────────────────────────┐
  │  📦 div.header           │
  │  ┌────────────────────┐  │
  │  │ 📦 div.form        │  │ ← 蓝色虚线边框包围整行
  │  └────────────────────┘  │    表示"移入 div.form 作为子节点"
  │  📦 div.footer           │
  └──────────────────────────┘

指示器 C：缩进插入线（放入特定层级）
  ┌──────────────────────────┐
  │  📦 div.form             │
  │    📥 input.username     │
  │    ═════════════════════ │ ← 插入线，但缩进对齐 div.form 的子层级
  │    📥 input.password     │    表示"插入为 div.form 的子节点"
  └──────────────────────────┘
```

### 4.3 命中区域判定

```
每个节点行（28px 高）被分为三个命中区域：

  ┌──────────────────────────────┐
  │ ▲ 上区域 (0-8px):            │ → 显示插入线（在此节点之前插入为兄弟）
  ├──────────────────────────────┤
  │ ■ 中区域 (8-20px):           │ → 显示容器高亮（移入此节点作为子节点）
  │   仅当目标是容器类型时生效    │    若非容器 → fallback 到上/下区域
  ├──────────────────────────────┤
  │ ▼ 下区域 (20-28px):          │ → 显示插入线（在此节点之后插入为兄弟）
  └──────────────────────────────┘

额外规则：
  · 拖到折叠节点的中区域 → 悬停 500ms 后自动展开
  · 拖到展开容器节点的下区域 → 插入为该容器的第一个子节点（不是兄弟）
  · 不能拖入自己的后代节点（循环引用）→ 禁止标志 🚫
  · 不能拖到根节点之外 → rootNode 始终为最顶层
```

### 4.4 拖拽完整流程

```
mousedown (在某节点行上)
     │
     ├── 记录拖拽源: dragSourceId = node.id
     ├── 延迟 150ms 后才进入拖拽模式（区分点击选中 vs 拖拽）
     │
     ▼
mousemove (进入拖拽模式后)
     │
     ├── 视觉反馈:
     │   · 源节点行变半透明 (opacity: 0.4)
     │   · 鼠标旁显示拖拽预览（源节点的缩略信息）
     │   · 目标位置显示放置指示器
     │
     ├── 命中判定:
     │   · 根据鼠标 Y 坐标确定悬停在哪一行
     │   · 根据行内 Y 偏移确定命中区域（上/中/下）
     │   · 根据鼠标 X 坐标判断缩进层级（是否要移出当前父容器）
     │
     ├── 合法性检查:
     │   · 目标不是源节点的后代 → ✓ 允许
     │   · 目标不是源节点自身 → ✓ 允许
     │   · 目标是容器类型（移入时）→ ✓ 允许
     │   · 不满足 → 🚫 禁止图标，不显示指示器
     │
     ├── 自动滚动:
     │   · 拖拽到组件树顶部/底部边缘 → 自动滚动
     │   · 滚动速度根据距离边缘的像素数线性增加
     │
     ▼
mouseup (松手)
     │
     ├── 如果位置合法:
     │   ├── 生成 moveElement Operation:
     │   │     {
     │   │       type: "moveElement",
     │   │       params: {
     │   │         elementId: dragSourceId,
     │   │         newParentId: targetParentId,
     │   │         position: insertIndex
     │   │       }
     │   │     }
     │   ├── 执行 Operation → Schema 更新 → 画布 + 组件树同步更新
     │   └── 支持 Cmd+Z 撤销
     │
     └── 如果位置非法或未移动:
         └── 取消拖拽，恢复原状

多选拖拽:
  · 多个节点选中时拖拽 → 所有选中节点一起移动
  · 生成 batch Operation（多个 moveElement）
  · 保持选中节点之间的相对顺序
  · 如果选中节点中包含祖先-后代关系 → 只移动祖先（后代自动跟随）
```

### 4.5 键盘调整顺序

```
选中节点后，可以用键盘调整顺序（不需要拖拽）：

  Cmd + ↑  → 在兄弟中上移一位（等同于拖到上一个兄弟之前）
  Cmd + ↓  → 在兄弟中下移一位
  ]        → 移到兄弟中的最后一位（最前层级）
  [        → 移到兄弟中的第一位（最后层级）

这些快捷键与画布的层级操作快捷键一致（见 02-toolbar 快捷键表）。
```

## 五、节点操作

### 5.1 重命名

```
触发方式:
  · 双击节点名称区域
  · 选中后按 F2
  · 右键 → "重命名"

交互:
  · 名称区域变为 inline 文本输入框
  · 自动全选当前名称
  · Enter / blur → 确认重命名
  · Escape → 取消

限制:
  · 最长 50 字符
  · 允许中文、英文、数字、- _ .
  · 空值 → 恢复为默认（显示 node.type）
```

### 5.2 显示/隐藏

```
操作:
  · 点击行右侧的 👁 图标
  · 右键 → "隐藏" / "显示"
  · 快捷键: Cmd+Shift+H (选中后)

行为:
  · 设置 node.visible = false
  · 画布: 该元素不渲染（DOM 层不输出，Canvas 覆盖层也不绘制选区）
  · 组件树: 节点行仍显示，但名称变灰色 + 删除线，👁 图标变为 👁‍🗨 (划线眼睛)
  · 隐藏的节点仍可在组件树中选中 → 在属性面板中编辑
  · 导出代码时: 隐藏节点不输出（或输出为注释）

注意:
  · 隐藏父节点 → 子节点也全部隐藏（但子节点自身的 visible 不变）
  · 恢复父节点可见 → 子节点恢复各自的 visible 状态
```

### 5.3 锁定/解锁

```
操作:
  · 点击行右侧的 🔒 图标
  · 右键 → "锁定" / "解锁"

行为:
  · 设置 node.locked = true
  · 画布: 元素正常渲染，但不可选中、不可拖拽、不可 resize
  · 组件树: 🔒 图标常驻显示（非 hover 也可见）
  · 属性面板: 选中锁定节点后，所有编辑控件置灰（只读）
  · 场景: 背景层、装饰元素等不想误触的元素

注意:
  · 锁定父节点 → 子节点也全部锁定（但子节点自身的 locked 不变）
  · 在组件树中仍可选中锁定节点（只是画布上点不到）
```

### 5.4 右键菜单

```
在组件树节点上右键:

┌────────────────────────────────────┐
│  选中       (此节点)               │  ← 如果右键的不是当前选中节点
│  ────────────────────────────────  │
│  重命名                     F2    │
│  复制                     Cmd+C   │
│  粘贴                     Cmd+V   │
│  复制副本                 Cmd+D   │
│  删除                    Delete   │
│  ────────────────────────────────  │
│  包裹为容器               Cmd+G   │
│  解除包裹            Cmd+Shift+G  │
│  ────────────────────────────────  │
│  显示/隐藏          Cmd+Shift+H   │
│  锁定/解锁                        │
│  ────────────────────────────────  │
│  保存为组件资产...                  │
│  分离组件实例                      │  ← 仅组件实例
│  ────────────────────────────────  │
│  移到最前                   ]     │
│  移到最后                   [     │
│  ────────────────────────────────  │
│  全部展开                          │
│  全部折叠                          │
└────────────────────────────────────┘

在空白区域（节点下方的空间）右键:

┌────────────────────────────────────┐
│  粘贴                     Cmd+V   │
│  ────────────────────────────────  │
│  全部展开                          │
```

## 六、搜索与过滤

### 6.1 搜索入口

```
组件树标题栏右侧的 [🔍] 按钮：

  点击 → 标题栏变为搜索输入框:

  ┌──────────────────────────────┐
  │  [🔍 搜索节点...         ] [×]│
  └──────────────────────────────┘

  · 实时搜索（每次输入后 200ms debounce）
  · [×] 清除搜索并恢复完整树
  · Escape 关闭搜索框
  · Cmd+F (画布聚焦时) → 打开组件树搜索
```

### 6.2 搜索匹配规则

```
搜索关键词匹配以下字段（OR 关系）：

  1. node.name → "header" 匹配 "div.header"
  2. node.type → "button" 匹配所有 button 元素
  3. 文本内容 → "登录" 匹配内含 "登录" 文本的 span/p/h1
  4. 组件模板名 → "Login" 匹配 "component:LoginForm"

不区分大小写。
支持中文。
```

### 6.3 搜索结果展示

```
搜索模式下的组件树:

方案：过滤 + 路径保留

  · 只显示匹配的节点 + 它们的祖先路径
  · 不匹配的兄弟节点隐藏
  · 匹配节点高亮搜索关键词（黄色背景）
  · 祖先路径节点不高亮但保留（提供上下文）

示例：搜索 "button"

  搜索前:                        搜索后:
  ▾ 📦 div.root                 ▾ 📦 div.root
    ▾ 📦 div.header               ▾ 📦 div.form

```

## 八、与画布的双向联动

### 8.1 画布 → 组件树

```
画布上的操作          →  组件树的响应

  选中元素            →  对应行高亮 + 自动滚动到可见区域
  Hover 元素          →  对应行浅色高亮（不滚动）
  取消选中            →  清除所有高亮
  多选                →  多行同时高亮
  添加元素            →  树中插入新行 + 自动展开父节点
  删除元素            →  树中移除对应行
  拖拽移动元素到新父容器 → 树中节点移动到新位置
  Undo / Redo         →  树完整刷新以反映 Schema 变更
```

### 8.2 组件树 → 画布

```
组件树上的操作        →  画布的响应

  点击节点行          →  画布选中该元素 + 蓝色边框 + resize handles
  Hover 节点行        →  画布上该元素浅蓝色 hover 边框
  双击节点行          →  与画布双击一致（进入深层选择 / 文本编辑）
  拖拽移动节点        →  Schema 更新 → 画布重渲染（元素在新位置）
  删除节点            →  画布移除该元素
  隐藏节点 (👁)      →  画布不渲染该元素
  锁定节点 (🔒)      →  画布该元素不可交互（但正常渲染）

滚动定位:
  · 组件树中选中的节点如果在画布视口之外 → 画布自动平移使其可见
  · 使用 scrollIntoView 式的平滑动画（300ms ease-out）
  · 不改变缩放级别，只平移
```

### 8.3 联动接口


## 九、性能优化

### 9.1 虚拟滚动

```
当组件树节点数 > 100 时，启用虚拟滚动:

  · 只渲染可见区域 + 上下各 10 行缓冲区的 DOM 节点
  · 滚动时动态回收/创建 DOM
  · 保持滚动条高度正确（总高度 = 节点数 × 28px）

实现方案:
  · 使用 react-window 或 react-virtuoso
  · 需要预计算每个可见节点的绝对位置（考虑折叠/展开状态）
  · 展开/折叠操作 → 重新计算总高度 + 更新可见范围

平铺化计算:
  · 将树结构平铺为一维数组（DFS 遍历，跳过折叠节点的后代）
  · flatNodes = [{ node, depth, isExpanded, isVisible }]
  · 只有 isVisible=true 的节点参与渲染
  · 展开/折叠 → 重新执行平铺计算（通常 < 5ms）
```

### 9.2 搜索性能

```
搜索优化（节点数 > 500 时重要）:

  · 维护一份可搜索索引: Map<nodeId, searchableText>
    searchableText = [node.name, node.type, textContent, templateName].join(" ")
  · Schema 变更时增量更新索引（只更新变化的节点）
  · 搜索时遍历索引 O(n)，n = 总节点数
  · 200ms debounce 避免逐字搜索
```

### 9.3 拖拽性能

```
拖拽期间的优化:

  · 拖拽预览使用 CSS transform（不引起 reflow）
  · 放置指示器用绝对定位 div（不改变树的 DOM 结构）
  · hitTest 使用缓存的节点位置（从平铺数组计算，不查 DOM）
  · 自动滚动用 requestAnimationFrame 节流
```

## 十一、边界情况与异常处理

| 场景 | 预期行为 |
|------|---------|
| 空页面（rootNode 无子节点）| 组件树只显示 rootNode 一行 + 提示"在画布上添加元素" |
| 极深嵌套（> 15 层）| 缩进 16px × 15 = 240px → 超出面板宽度时水平滚动，或缩进递减（深层级每级 8px） |
| 极多节点（> 500）| 虚拟滚动生效；搜索有 200ms debounce |
| 拖拽到自己的后代 | 🚫 禁止光标，松手无效果 |
| 拖拽到锁定节点内 | 允许（锁定只阻止画布交互，不阻止结构调整）|
| 选中的节点被 AI 删除 | 自动清除选中状态，组件树刷新 |
| 页面名重复 | 允许（用 id 区分，名称只是显示用） |
| 删除最后一个页面 | 禁止（删除按钮置灰，tooltip 提示"至少保留一个页面"）|
| 快速连续展开/折叠 | 平铺化计算做 16ms debounce（一帧内最多执行一次）|
| 组件树与画布选中不一致 | 不应出现——两者共享同一个 selectedNodeIds 状态源 |
| 拖拽中窗口失焦 | 取消拖拽，恢复原状 |

## 十三、核心设计决策记录

| 决策 | 选择 | 理由 |
|------|------|------|
| 组件树位置？ | 左侧面板上部 | Figma 的左侧 Layers 位置，用户心智模型已建立 |
| 页面列表位置？ | 左侧面板下部（与组件树同列） | 页面和组件树是结构导航的两个维度，放在一起便于切换 |
| 节点行高度？ | 28px | 信息密度与可点击面积的平衡；Figma 同为 28px |
| 每层缩进量？ | 16px | 足够区分层级但不会太快超出面板宽度 |
| 拖拽命中区域？ | 三区域（上/中/下各占 8/12/8px）| 同时支持"插入为兄弟"和"移入为子节点"两种意图 |
| 搜索方式？ | 过滤+路径保留（不是高亮跳转）| 过滤模式下信息更聚焦；路径保留提供上下文 |
| 虚拟滚动阈值？ | 100 节点 | 100 节点以下原生 DOM 性能足够；100 以上需要优化 |
| 页面与组件树的共享？ | 共享同一个选中状态 + 各自独立的 UI | 两者是同一棵 Schema 树的不同视角，选中状态必须一致 |
| 隐藏/锁定的继承？ | 父节点隐藏→子节点视觉隐藏但自身 visible 不变 | 与 CSS 的 display:none 继承行为一致，解锁父不会意外解锁子 |
