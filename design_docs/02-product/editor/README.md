> 本文档是编辑器的**顶层设计总纲**。
>
> 核心定位：这不是一个画图工具，而是一个**可视化的产品构建器**。
> 用户在这里构建的不是一张张独立的页面，而是一个数据流动、状态切换、交互可用的完整产品。

## 一、第一性原理：从根本目标推导编辑器设计

### 1.1 四大目标对编辑器的要求

```
G1 设计即代码  → 编辑器构建的 Schema 必须是代码的完整源头
                 结构、样式、状态、交互、数据——缺一不可
                 → 编辑器必须让这五个维度都能高效定义

G2 AI 增量操作 → 编辑器的所有操作都是标准化的 Operation
                 AI 和人类共享同一套操作语义
                 → 编辑器必须 Operation-first

G3 设计师友好  → 编辑器的交互体验不低于 Figma
                 但同时自然地引导用户定义状态、数据、交互
                 → 数据/状态/交互不是事后补充，而是设计过程的一部分

G4 产品友好   → 设计完成时，产品的所有行为已经完整定义
                 需求文档是 Schema 的格式化输出，不需要人工补充
                 → 编辑器必须让用户构建的是「完整产品」，不是「一张张图」
```

### 1.2 关键推导

```
从 G4 倒推：
  需求文档需要什么？ → 页面结构 + 交互说明 + 状态流转 + 数据接口
  这些信息从哪来？   → 全部来自 Schema
  Schema 包含什么？  → 结构 + 样式 + 交互 + 状态 + 数据
  谁来构建 Schema？  → 编辑器

  → 编辑器的职责 = 让用户在设计过程中自然地定义出 Schema 的全部五个维度

从 G3 推导：
  设计过程中就是正常的软件交互（不是画一堆 frame 反复解释如何交互）
  → 页面之间通过交互事件（click → navigate）自然连接
  → 数据通过数据集和表达式自然绑定
  → 状态通过上下文切换自然定义
  
  → 数据、状态、交互不是「附加 Tab」，而是渗透在整个编辑过程中的一等公民

从 G1 推导：
  代码 = 组件树 + 状态管理 + 事件处理 + API 调用 + 路由
  所有这些都必须在 Schema 中有完整表达
  → 编辑器必须让每一个设计动作都在构建代码可映射的 Schema
```

### 1.3 与 Figma 的根本区别

| 维度 | Figma | 我们 |
|------|-------|------|
| 设计对象 | 孤立的页面画面 | 通过导航 / 数据 / 状态连接的完整产品 |
| 元素本质 | 无语义的图形 | 有语义的 HTML 元素和组件实例 |
| 状态表达 | 每个状态画一份 | 同一元素 + 状态上下文 = 多种表现 |
| 数据 | 无 | 数据集 + 绑定表达式，数据驱动 UI |
| 交互 | Prototype 模式连线 | 事件直接绑定在元素上，设计 = 定义交互 |
| 页面关系 | 独立 Frame | 交互事件自然形成导航图 |
| 输出 | 静态图片 | 可运行的代码 + 需求文档 |

**一句话：Figma 是画图工具，我们是产品构建器。**

## 二、编辑器的三大核心概念

在进入具体布局之前，先定义三个贯穿编辑器设计的核心概念：

### 2.1 产品上下文（Product Context）

用户在编辑器中看到的 UI 表现，始终是「产品上下文」的函数：

```
当前 UI 表现 = f(
  当前页面,
  全局状态:   { theme: 'dark', role: 'admin', locale: 'zh' },
  数据场景:   { dataSet: '空数据' },
  业务状态:   { loginForm: 'error' },
  交互状态:   { submitButton: 'hover' },
  视口:       { device: 'iPhone 15 Pro', width: 393, height: 852 }
)
```

编辑器的核心职责之一是让用户**在这个多维上下文中自由导航和编辑**。
切换任何一个维度 → UI 实时变化 → 用户可以在新上下文中编辑差异。

### 2.2 连接的页面（Connected Pages）

页面不是孤立的画面，而是通过交互事件自然连接的产品节点：

```
登录页 ──click: 登录按钮──→ 首页
  │                        │
  │                    click: 设置
  │                        │
  └──click: 注册──→ 注册页  └──→ 设置页
```

- 页面通过「在元素上绑定 navigate 事件」自然创建和连接
- 系统自动维护页面间的导航关系图
- 全局状态和数据跨页面共享

### 2.3 实时渲染（Live Rendering）

画布上渲染的不是静态图片，而是真实 DOM：

- 数据绑定表达式实时求值并渲染
- 状态覆盖实时应用
- 列表元素根据数据集自动渲染 N 行
- 预览模式下交互真实可用

## 三、编辑器整体架构

### 3.1 布局

```
┌──────────────────────────────────────────────────────────────────────┐
│  顶部栏                                                               │
│  [产品名] [视口: iPhone 15 ▾] [缩放] [↩撤销 ↪重做] [▶预览] [导出 ▾]   │
├──────────┬───────────────────────────────────┬───────────────────────┤
│          │                                   │                       │
│  左侧    │          中央画布                  │    右侧编辑面板        │

│  导航    │                                   │  ┌─ 状态上下文栏 ────┐ │
│          │  ┌─────────────────────────┐      │  │ ● default ○ hover │ │
│  [页面]  │  │ Canvas 覆盖层           │      │  │ 数据: [正常 ▾]    │ │

│  [数据]  │  ├─────────────────────────┤      │                       │
│          │  │ React DOM 渲染层        │      │  ▸ 属性               │
│          │  │ Schema → 真实组件       │      │  ▸ 样式               │
│          │  │ 数据绑定实时求值         │      │  ▸ 子元素可见性        │
│          │  └─────────────────────────┘      │  ▸ 交互行为           │
│          │                                   │  ▸ 代码预览           │
│          │                                   │                       │
├──────────┴───────────────────────────────────┴───────────────────────┤
│  底部工具栏                                                           │
│  [↖选择] [✋平移] [# 容器] [□ 方块] [T 文本] [◇ 组件库] [💬 注释]    │
└──────────────────────────────────────────────────────────────────────┘
```

### 3.2 五个区域的重新定义

| 区域 | 旧定位 | 新定位 | 为什么变 |
|------|--------|--------|---------|
| **顶部栏** | 全局控制（视口/缩放/撤销） | **产品级控制**（产品名 + 视口 + 预览 + 导出） | 强调「产品」而非「页面」 |
| **左侧面板** | 图层树 + 页面列表 | **产品导航器**（页面/元素/数据三个视图） | 数据是产品级的一等公民，不应藏在右侧 Tab |
| **中央画布** | Schema 的可视化渲染 | **产品的实时预览**（数据驱动 + 状态感知） | 强调画布是「活的」产品，不是静态图 |
| **右侧面板** | 6 Tab 属性面板 | **状态驱动的统一编辑面板** | 状态是上下文，不是 Tab（详见[面板重组方案](./03-property-panel/redesign.md)）|
| **底部工具栏** | 绘制工具入口 | **构建工具**（选择/绘制/组件/注释） | 本质不变，强调是「构建」不是「画图」 |

## 四、左侧产品导航器

左侧面板从「图层树 + 页面列表」升级为**产品导航器**，
有三个视图，对应产品的三个导航维度：

### 4.1 页面视图

```
┌──────────────────────────┐
│  页面                     │
├──────────────────────────┤
│                          │
│  ● 登录页 ←(当前)        │
│    └→ 首页               │ ← 跳转关系自动显示
│    └→ 注册页             │
│  ○ 首页                  │
│    └→ 设置页             │
│    └→ 个人中心           │
│  ○ 注册页                │
│  ○ 设置页                │
│  ○ 个人中心              │
│                          │
│  [+ 新建页面]             │
│                          │
│  ── 导航关系图 ────────── │
│  (缩略版：页面节点 + 连线) │
│  [展开查看完整流程图]      │
│                          │
└──────────────────────────┘
```

**设计要点：**

- 页面列表不仅是名字列表，还自动显示**页面间的跳转关系**
- 跳转关系来自交互事件中的 `navigate` 动作（不需要手动连线）
- 底部可展开一个迷你导航关系图（页面节点 + 有向连线）
- 点击页面 → 画布切换到该页面
- 当前页面高亮显示

### 4.2 元素视图

```
┌──────────────────────────┐
│  元素                     │
├──────────────────────────┤
│                          │
│  ▸ Root (div)             │
│    ▸ Header (div)         │
│      ├ Logo (img)         │
│      └ NavBar (component) │
│    ▾ LoginForm (div)  ⚡🔗│ ← ⚡有事件 🔗有数据绑定
│      ├ Username (input)   │
│      ├ Password (input)   │
│      └ Submit (button) ⚡ │
│    ▸ Footer (div)         │
│                          │
│  [搜索元素...]             │
│                          │
└──────────────────────────┘
```

**设计要点：**

- 元素树节点旁边显示**行为指示器**：
  - ⚡ = 有事件绑定
  - 🔗 = 有数据绑定表达式
  - 🔵 = 有非 default 状态定义
  - 👁 = 有条件可见性
- 这些指示器让用户一眼看出哪些元素有「行为」，不只是视觉元素
- 与画布选中联动
- 支持拖拽调整层级

### 4.3 数据视图

```
┌──────────────────────────┐
│  数据                     │
├──────────────────────────┤
│                          │
│  ── 数据集 ──────────── │
│  ● 正常数据              │
│  ○ 空数据                │
│  ○ 异常数据              │
│  ○ VIP 用户数据          │
│  [+ 新建数据集]           │
│                          │
│  ── 全局状态 ──────────── │
│  theme: [light ▾]        │
│  role:  [user  ▾]        │
│  locale:[zh    ▾]        │
│  [+ 添加全局状态变量]     │
│                          │
│  ── 数据编辑 ──────────── │
│  [树形] [JSON]            │
│  { user: {               │
│      name: "张三",        │
│      role: "admin",      │
│      avatar: "..."       │
│    },                    │
│    items: [...]           │
│  }                       │
│                          │
└──────────────────────────┘
```

**设计要点：**

- 数据视图是**页面级**的，与选中哪个元素无关
- 顶部是数据集切换器，切换后画布实时更新
- 中部是全局状态变量管理，修改后所有响应元素实时更新
- 底部是当前数据集的 JSON 编辑器
- 数据**绑定**（某属性绑定某数据路径）保留在右侧编辑面板的属性区内联

**为什么数据在左侧而不是右侧？**

```
左侧 = 产品级导航（页面结构、元素结构、数据结构）
右侧 = 节点级编辑（选中一个元素后编辑它的属性、样式、状态、行为）

数据集 / 全局状态 = 产品级概念 → 属于左侧
数据绑定表达式 = 节点级操作 → 属于右侧属性区
```

## 五、中央画布

### 5.1 核心特性

画布不是静态画板，而是**产品在当前上下文下的实时预览**：

```
画布显示 = f(当前页面, 当前数据集, 当前全局状态, 当前视口)

切换数据集 → 画布实时更新（列表从有数据变成空数据提示）
切换全局状态 → 画布实时更新（主题从 light 变成 dark）
切换视口 → 画布实时更新（从 iPhone 变成 iPad）
```

### 5.2 双层架构（保持不变）

```
Canvas 覆盖层（EditorOverlay）
  选区框、拖拽手柄、对齐线、间距标注、绘制预览
  所有编辑交互在此层处理

React DOM 渲染层（SchemaRenderer）
  Schema → 真实 HTML 元素 / React 组件
  数据绑定实时求值
  状态覆盖实时应用
```

### 5.3 画布上的产品上下文可视化

画布顶部可选显示一个**上下文提示条**，告诉用户当前在什么上下文下编辑：

```
┌──────────────────────────────────────────────┐
│ 📊 数据:正常数据  🌙 theme:dark  👤 role:admin │ ← 上下文提示条
├──────────────────────────────────────────────┤
│                                              │
│         (画布内容)                            │
│                                              │
└──────────────────────────────────────────────┘
```

- 点击各指示器可快速切换
- 非 default 上下文时提示条显示醒目颜色
- 可隐藏此条（纯画布模式）

## 六、右侧编辑面板

右侧面板已重组为**状态驱动的统一编辑面板**，详见 [面板重组方案](./03-property-panel/redesign.md)。

核心设计：

```
元素信息条 → 状态上下文栏（固定顶部）→ 滚动编辑区：
  ▸ 属性（当前状态下的 props）
  ▸ 样式（当前状态下的 CSS，8 组折叠）
  ▸ 子元素可见性（当前状态下哪些子元素展示）
  ▸ 交互行为（当前状态下的事件绑定）
  ▸ 代码预览（折叠）
```

**关键洞察：状态是一切的上下文，不是一个 Tab。**

选中一个元素 → 切换到 loading 状态 →
编辑面板中所有区域都显示 loading 态的值 →
编辑任何值都自动写入 loading 态的覆盖。

## 七、数据/状态/交互如何渗透整个编辑器

这是本方案的核心——数据、状态、交互不是藏在某个 Tab 里的附加功能，
而是在编辑器的每个角落都可见、可操作的一等公民：

### 7.1 数据的渗透

| 位置 | 数据的存在形式 |
|------|---------------|
| **左侧数据视图** | 数据集管理、JSON 编辑、全局状态变量管理 |
| **右侧属性区** | 属性值旁的 🔗 图标 → 绑定数据表达式 `{{ data.xxx }}` |
| **画布** | 数据绑定值实时渲染；切换数据集 → 画布实时变化 |
| **画布上下文条** | 显示当前数据集名称，点击快速切换 |
| **元素树** | 🔗 图标标记有数据绑定的节点 |
| **预览模式** | 数据驱动 UI 变化真实可用 |

### 7.2 状态的渗透

| 位置 | 状态的存在形式 |
|------|---------------|
| **右侧状态上下文栏** | 四层状态切换器（交互/业务/数据/全局） |
| **右侧样式区** | 蓝色标记 🔵 = 当前状态覆盖了此值 |
| **右侧子元素区** | 当前状态下子元素的可见性开关 |
| **画布** | 实时预览当前状态组合的 UI 效果 |
| **画布上下文条** | 显示当前全局状态值 |
| **左侧数据视图** | 全局状态变量管理 |
| **元素树** | 🔵 图标标记有状态定义的节点 |

### 7.3 交互的渗透

| 位置 | 交互的存在形式 |
|------|---------------|
| **右侧交互行为区** | 事件卡片列表 + 添加事件 |
| **左侧页面视图** | 页面间跳转关系（来自 navigate 事件） |
| **画布** | ⚡ 闪电图标标记有事件的元素 |
| **元素树** | ⚡ 图标标记有事件绑定的节点 |
| **预览模式** | 所有事件真实执行 |

### 7.4 为什么这很重要？

```
传统设计工具：数据/状态/交互藏在附加功能里
  → 设计师忘记定义 → 开发者需要猜测 → 需求文档不完整

我们的编辑器：数据/状态/交互随处可见
  → 设计过程自然地定义完整行为
  → 代码导出和需求文档天然完整
  → 实现四大根本目标
```

## 八、预览模式与编辑模式

### 8.1 不是两个模式，而是一个连续体

```
传统工具：  编辑模式 ←切换→ 预览模式（完全不同的界面）

我们：     编辑模式 ←── 渐进过渡 ──→ 预览模式

编辑模式：画布上有覆盖层（选区/手柄/对齐线），事件不执行
预览模式：覆盖层隐藏，事件真实执行，像真实 App 一样操作

关键：即使在编辑模式，画布也是「活的」：
  · 数据绑定实时渲染
  · 状态切换实时预览
  · 列表根据数据自动展开
  · 唯一不执行的是交互事件（因为点击需要用于选中元素）
```

### 8.2 预览模式的增强

预览模式不只是「隐藏编辑层」，它是**产品验证环境**：

- 点击按钮 → 真实跳转页面
- 状态切换 → UI 真实变化
- 数据更新 → 列表真实刷新
- 预览控制条：切换数据集 / 全局状态 / 重置 / 后退

## 十、数据流

编辑器内有两条并行的数据流路径：

```
路径 A：手动操作（前端直接）
  用户操作画布/面板
       │
       ▼
  OperationExecutor.execute(op)    ← 标准化 Operation
       │
       ├─→ MobX Store 更新 → 画布重新渲染
       └─→ REST API → design-api 持久化

路径 B：AI 操作（通过 MCP）
  Cursor/Claude Code 发出 Tool Call
       │
       ▼
  MCP Server → design-api REST 接口
       │
       ├─→ 持久化
       └─→ WebSocket 推送给前端
              │
              ▼
        前端收到变更 → 应用操作 → 画布重新渲染
```

两条路径共享同一套 Operation 定义（`@globallink/design-operations`）。
