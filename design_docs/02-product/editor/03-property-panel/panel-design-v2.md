## 一、现状问题诊断（为什么「难用到不知道怎么用」）

### 1.1 截图诊断

当前右侧面板从上到下出现了以下内容：

```
┌─ 当前实现 ────────────────────────────────────┐
│  交互 [默认][悬停][按下][聚焦][禁用]            │  ← StateContextBar
│  组件态 [default（基础）▾]                      │  ← StateContextBar
│                                                │
│  div nd_6e291                [🔒][👁] nd_6e2  │  ← 元素信息条
│                                                │
│  ▸ 属性                                        │  ← 折叠
│  ▸ 样式                                        │  ← 折叠
│  ▾ 状态与领域·                                  │  ← 展开！
│     交互状态 [hover][active][focus][disabled]   │  ← ⚠️ 与顶部重复！
│     业务状态 · 暂无 · [+添加状态]               │  ← ⚠️ 与顶部"组件态"重复！
│     全局状态绑定 · 请先添加                      │  ← ⚠️ 用户不知道这是什么
│     全局状态变量 · 暂无 · [+添加变量]            │  ← ⚠️ 页面级概念塞在节点面板
│     组合预览（W6-042）                          │  ← ⚠️ 功能堆砌
│  ▸ 子元素可见性                                 │
│  ...                                           │
└────────────────────────────────────────────────┘
```

### 1.2 五个致命问题


| #   | 问题        | 症状                                     | 根因                                                |
| --- | --------- | -------------------------------------- | ------------------------------------------------- |
| 1   | **控件重复**  | 「交互状态」出现两次（顶部 + 状态区内部），功能完全一样          | StateContextBar 作为新增叠加在旧 StatesTab 上，没有替代         |
| 2   | **概念混淆**  | 顶部叫「组件态」，下面叫「业务状态」—— 是同一个东西的两个名字       | 术语不统一，技术命名泄漏到界面                                   |
| 3   | **技术黑话**  | 「状态与领域」「全局状态绑定」「领域」—— 用户不知道这些词什么意思     | 按技术模型（DomainState / EnvironmentState）命名界面，不是按用户心智 |
| 4   | **作用域混放** | 「全局状态变量」是页面级/项目级概念，塞在节点编辑面板里           | 没有区分「节点级编辑」和「页面级配置」的位置                            |
| 5   | **信息过载**  | 「状态与领域」section 内部有 5 个子区域，全部展开，每个都要注意力 | redesign.md 的信息架构没有被正确实现                          |


### 1.3 根因总结

> **redesign.md 提出的方案是对的**：「状态是上下文，不是 Tab」。
>
> 但实现时犯了一个根本错误：**只加了新的（StateContextBar），没删掉旧的（StatesTab 内的交互状态/业务状态）**，导致同一概念出现两次，比原来更混乱。

## 二、第一性原理：右侧面板到底解决什么问题

### 2.1 一句话定义

> 右侧面板 = **对当前选中元素，在明确的状态条件下，编辑它的完整表现。**

分解：

- **当前选中元素**：面板内容完全由选中节点驱动
- **明确的状态条件**：用户要知道自己在编辑 default 态还是 hover 态还是 loading 态
- **完整表现**：包括外观（CSS）、内容（文本/属性）、结构（子元素显隐）、行为（事件）

### 2.2 用户不是按技术维度思考的

用户在编辑一个按钮时，脑子里想的不是「我要去 CSS Tab 改颜色」，而是：


| 用户在想        | 对应的操作                                       |
| ----------- | ------------------------------------------- |
| 「这个按钮长什么样」  | 编辑外观（样式）                                    |
| 「按钮上写什么字」   | 编辑内容（文本/属性）                                 |
| 「鼠标放上去变什么样」 | 切换到 hover → 编辑外观差异                          |
| 「加载中是什么样」   | 创建 loading 状态 → 切换到 loading → 编辑内容+外观+子元素差异 |
| 「点了会发生什么」   | 配置事件                                        |
| 「暗色主题下什么样」  | 切换环境变量 → 编辑差异                               |


**关键洞察**：用户的工作流是**「先确定条件，再编辑表现」**，不是**「先选技术维度，再找状态」**。

### 2.3 状态的本质（三层模型）

从用户视角，影响元素表现的「条件」只有三层：

```
第一层：元素自身的交互状态（最近、最直接）
  ├── 鼠标悬停时 (hover)
  ├── 按下时 (active/pressed)
  ├── 获得焦点时 (focus)
  └── 禁用时 (disabled)
  触发方式：用户直接交互，自动触发
  作用域：仅影响该元素自身

第二层：元素的业务状态（组件级自定义）
  ├── 加载中 (loading)
  ├── 成功 (success)
  ├── 出错 (error)
  └── ... 自定义任意名称
  触发方式：应用逻辑、事件驱动
  作用域：影响该元素及其子树

第三层：全局条件（页面/项目级）
  ├── 数据场景：正常数据 / 空数据 / 异常数据
  ├── 领域状态：taskStatus=done / formStep=2
  └── 环境变量：theme=dark / locale=zh
  触发方式：外部条件变化
  作用域：影响多个元素
```

**设计原则：第一层和第二层是节点级的，属于右侧面板。第三层是全局的，不应该塞进右侧面板。**

## 三、用户画像与用户故事

### 3.1 两类核心用户


| 角色           | 占比  | 关注点                   |
| ------------ | --- | --------------------- |
| **界面搭建者**    | 80% | 布局、样式、文案、hover 效果     |
| **交互/状态定义者** | 20% | 业务状态、事件绑定、数据绑定、全局状态响应 |


### 3.2 用户故事（按优先级排序）

**P0 — 日常搭建（占 80% 使用时间，必须零学习成本）**

```
US-1: 选中元素 → 立即看到外观和内容 → 直接编辑
  作为搭建者，我选中一个 div，希望面板立刻展示它的样式和文本，
  我不需要切任何 Tab 就能修改颜色、间距、字体。

US-2: 给按钮加 hover 效果
  作为搭建者，我选中一个按钮，想让它 hover 时变色。
  我只需要在面板顶部点一下「悬停」，然后直接改颜色就行了，
  不需要找「状态 Tab」也不需要知道什么是「状态覆盖」。

US-3: 空元素不阻塞工作流
  作为搭建者，我选中一个 div，它没有特殊属性。
  面板不应该显示一大块「没有可编辑属性」来吓我，
  而应该直接展示我最可能用到的东西（外观）。
```

**P1 — 状态设计（占 15% 使用时间）**

```
US-4: 创建和编辑业务状态
  作为设计者，我要给一个表单定义 loading/success/error 三种状态。
  我需要：创建状态 → 切换到该状态 → 编辑差异（样式+内容+子元素显隐）。
  整个过程应该在同一个面板里完成，不需要在多个 Tab 间来回切换。

US-5: 清楚知道自己在编辑什么条件
  作为设计者，当我切到 hover 或 loading 状态编辑时，
  面板必须清晰告诉我「你正在编辑 XX 状态」，
  并且标出哪些值是覆盖的、哪些是从 default 继承的。

US-6: 控制子元素在不同状态下的显隐
  作为设计者，我希望容器在 loading 时隐藏列表、显示 spinner。
  我需要一个直观的勾选列表来控制每个子元素的可见性。
```

**P2 — 行为与数据（占 5% 使用时间）**

```
US-7: 给元素绑定事件
  作为设计者，我要给按钮绑定 click → 跳转注册页。
  事件配置应该是一个独立的区域，不与样式混在一起。

US-8: 属性绑定数据表达式
  作为设计者，我要把文本内容绑定到 {{data.user.name}}。
  绑定入口应该在属性旁边（🔗 图标），不需要切到「数据 Tab」。

US-9: 查看代码预览
  作为开发者，我偶尔想看看当前元素的 JSON/React 代码。
  代码预览应该在最底部折叠，不占用日常编辑空间。
```

## 四、信息架构 v2（面板结构）

### 4.1 整体布局

```
┌─ 右侧面板 ─────────────────────────────┐
│                                         │
│  A. 元素头部（固定）                      │
│     div · nd_6e291 · [🔒][👁]           │
│                                         │
│  B. 状态切换器（固定，紧凑）              │
│     [默认][悬停][按下][聚焦][禁用]        │
│     自定义: [default ▾] [+ 添加]        │
│     ── 当前编辑：hover 态 ──            │
│                                         │
├─ 滚动分界线 ────────────────────────────┤
│                                         │
│  C. 编辑区（可滚动，折叠 Section）        │
│                                         │
│     1. 内容 ──── 文本、元素属性、组件 Props │
│     2. 外观 ──── CSS 八组样式编辑器       │
│     3. 子元素 ── 子元素可见性（容器独有）  │
│     4. 行为 ──── 事件列表                 │
│     5. 高级 ──── 代码预览（默认折叠）      │
│                                         │
└─────────────────────────────────────────┘
```

### 4.2 与旧结构的核心差异


| 维度             | 旧实现                                     | 新设计                     | 理由           |
| -------------- | --------------------------------------- | ----------------------- | ------------ |
| 交互态切换          | 顶部 StateContextBar + StatesTab 内部（重复两次） | **仅在状态切换器里出现一次**        | 消除重复         |
| 业务态切换          | 顶部叫「组件态」+ StatesTab 内叫「业务状态」            | **统一叫「自定义状态」，仅在状态切换器里** | 统一术语         |
| 领域状态变量管理       | StatesTab 内的「全局状态变量」                    | **移到左侧面板的「数据」Tab**      | 页面级概念不应在节点面板 |
| 领域状态绑定         | StatesTab 内的「全局状态绑定」                    | **移到「高级」区域或右键菜单**       | 低频高级操作       |
| 环境变量切换         | StateContextBar 内                       | **移到底部工具栏或画布工具栏**       | 项目级概念        |
| 「状态与领域」section | 一个 section 塞 5 个子区域                     | **完全移除**                | 消除混乱根源       |
| 组合预览           | StatesTab 底部                            | **移到底部工具栏**             | 预览控制属于全局操作   |


### 4.3 关键设计决策：什么在右侧面板，什么不在

```
✅ 属于右侧面板（节点级）：
  ├── 元素的外观编辑
  ├── 元素的内容/属性编辑
  ├── 元素的事件绑定
  ├── 交互态切换与编辑（hover/active/focus/disabled）
  ├── 自定义状态的创建、切换与编辑（loading/error/...）
  ├── 容器的子元素可见性控制
  └── 代码预览

❌ 不属于右侧面板（应移到其他位置）：
  ├── 领域状态变量的定义和管理 → 移到左侧面板「数据」Tab
  ├── 环境变量（theme/locale）的预览切换 → 移到底部工具栏
  ├── 数据集管理 → 移到左侧面板「数据」Tab
  ├── 状态组合预览器 → 移到底部工具栏
  └── 领域状态绑定的详细配置 → 移到「高级」section 或专用弹窗
```

## 五、各区域详细设计

### 5.1 A 区：元素头部（固定）

**职责**：告诉用户「你在编辑谁」。

```
┌──────────────────────────────────────┐
│  div  nd_6e291     [🔒] [👁] nd_6e2│
│  ↑类型  ↑名称       锁定 可见  ↑ID  │
└──────────────────────────────────────┘
```

**规则：**

- 类型标签：蓝色 badge，显示 `div` / `button` / `img` / `component:LoginForm`
- 名称：可编辑，双击进入编辑模式
- 锁定/可见：图标按钮，状态改变有颜色反馈
- 多选时：显示「已选中 N 个元素」

### 5.2 B 区：状态切换器（固定，紧凑）

**职责**：告诉用户「你在什么条件下编辑」，并提供切换入口。

这是整个面板的**最高层上下文控制器**，始终可见，不随内容滚动。

```
┌──────────────────────────────────────┐
│                                      │
│  [默认] [悬停] [按下] [聚焦] [禁用]  │
│   ━━━━                               │  ← 当前选中有下划线
│                                      │
│  ── 以下仅在存在自定义状态时显示 ──   │
│  自定义: [default ▾]  [+ 新建]       │
│          ↑ 下拉选择已有的自定义状态    │
│                                      │
└──────────────────────────────────────┘
```

**交互态按钮的视觉语言：**

```
默认态（未激活）：     灰色文字，白色背景
当前编辑态（激活）：   蓝色文字 + 蓝色下划线
有覆盖（已定义差异）： 右上角蓝色小圆点
```

**状态提示条（非 default 态时显示）：**

```
┌──────────────────────────────────────┐
│  🔵 正在编辑「悬停」状态              │
│     覆盖的属性将以蓝色标记显示         │
│     编辑的值仅在此状态下生效           │
└──────────────────────────────────────┘
```

**自适应收起：**

- 元素没有自定义状态 → 「自定义」行隐藏，只显示交互态一行
- 这意味着 80% 的场景下，状态切换器只有一行高

**「+ 新建」自定义状态的流程：**

```
点击 [+ 新建] → 内联输入框:
┌──────────────────────────────────────┐
│  自定义: [loading    ] [✓] [×]      │
│          ↑ 输入名称    确认  取消     │
│                                      │
│  快捷选项:                            │
│  [loading] [success] [error] [empty] │
│  ↑ 点击直接创建                       │
└──────────────────────────────────────┘
```

**删除/重命名自定义状态：**

- 右键自定义状态下拉选项 → 弹出菜单：重命名 / 删除
- 或者下拉旁边显示管理入口（齿轮图标）

### 5.3 C 区：编辑区（可滚动）

#### Section 1：内容

**职责**：编辑元素「说什么」「是什么」。

```
┌──────────────────────────────────────┐
│  ▾ 内容                              │
├──────────────────────────────────────┤
│                                      │
│  ── 文本类元素（p/h1/button/span）── │
│  文本  [提交             ]  🔗       │
│         ↑ 直接编辑文本      ↑ 数据绑定│
│                                      │
│  ── 元素类型切换 ──                   │
│  类型  [button ▾]                    │
│                                      │
│  ── 元素特有属性 ──                   │
│  （按 node.type 动态渲染）            │
│  type     [button ▾]                 │
│  disabled [□]                        │
│                                      │
│  ── 组件实例时显示 ──                 │
│  📦 LoginForm（引用模式）            │
│  [脱离模板] [从模板同步]              │
│  标题  [用户登录       ]  str  🔗    │
│  记住密码 [✓]           bool  🔗    │
│                                      │
│  ── 列表绑定 ──                       │
│  重复渲染  [{{data.items}}    ]      │
│                                      │
└──────────────────────────────────────┘
```

**状态感知行为：**

- 在非 default 态下编辑内容 → 自动写入状态覆盖
- 被覆盖的属性旁显示 🔵 标记
- 🔵 点击 → 清除覆盖，回退到 default 值
- 未被覆盖的属性显示灰色值（继承自 default）

**空状态策略：**

- div/section 等容器没有特殊属性时 → 不显示「没有可编辑属性」
- 直接显示「此类型无额外属性，可在下方调整外观」（一行小字，不阻挡）

#### Section 2：外观

**职责**：编辑元素「长什么样」。

这是使用频率最高的区域（60%+），**默认展开**。

```
┌──────────────────────────────────────┐
│  ▾ 外观                              │
├──────────────────────────────────────┤
│                                      │
│  ▸ 布局 Layout ────────────────────  │
│  ▸ 尺寸 Size ──────────────────────  │
│  ▸ 间距 Spacing ───────────────────  │
│  ▸ 定位 Position ──────────────────  │
│  ▸ 背景 Fill ──────────────────────  │
│  ▸ 边框 Border ────────────────────  │
│  ▸ 文字 Typography ────────────────  │
│  ▸ 效果 Effects ───────────────────  │
│                                      │
└──────────────────────────────────────┘
```

**与原 StylesTab 的关系：** 内部完全复用现有 8 组样式编辑器，区别是：

- 每个属性值感知当前状态上下文
- 在 default 态编辑 → `updateStyle` Operation
- 在非 default 态编辑 → `updateState` Operation（写入覆盖）

**覆盖值的视觉语言：**

```
场景：当前在编辑 hover 态

  background  [#0056D2]  🔵      ← 蓝色圆点 = hover 态有覆盖
                                    hover 值 #0056D2
                                    default 值 #1677FF (灰色小字)

  color       [#FFFFFF]           ← 无标记 = 与 default 相同（继承）
                                    编辑此值 → 自动创建 hover 覆盖

  cursor      [not-allowed ▾] 🔵 ← hover 态覆盖了 cursor
                                    default: pointer
```

#### Section 3：子元素（仅容器显示）

**职责**：控制子元素在当前状态下的显隐。

```
┌──────────────────────────────────────┐
│  ▾ 子元素                             │
├──────────────────────────────────────┤
│                                      │
│  当前状态: loading                    │
│                                      │
│  ☑ header-text     「标题」     🔵   │
│  ☑ spinner         「加载动画」  🔵   │
│  ☐ content-list    「内容列表」  🔵   │
│  ☐ error-message   「错误提示」       │
│                                      │
│  🔵 = 当前状态下与 default 不同      │
│                                      │
└──────────────────────────────────────┘
```

**规则：**

- 叶子元素（img / input）不显示此 section
- 勾选 = 可见，取消勾选 = 隐藏
- default 态下修改 → 修改元素的 `visible` 属性
- 非 default 态下修改 → 写入状态的可见性覆盖

#### Section 4：行为

**职责**：定义元素「做什么」。

```
┌──────────────────────────────────────┐
│  ▸ 行为                              │
├──────────────────────────────────────┤
│                                      │
│  ┌────────────────────────────────┐  │
│  │  ● 点击 (click)               │  │
```

## 七、被移出右侧面板的功能去向

### 7.1 领域状态变量定义和管理 → 左侧面板「数据」Tab

**理由**：领域状态变量（taskStatus / formStep / loginPhase）是页面级概念，与选中哪个节点无关。放在节点编辑面板中违背了面板的隐含契约。

**在左侧面板的位置**：

```
左侧面板:
  [元素树] [页面] [数据]
                    ↑
                    ├── 数据源管理
                    ├── 领域状态变量管理 ← 移到这里
                    └── 数据绑定表达式帮助
```

### 7.2 环境变量预览切换 → 底部工具栏

**理由**：环境变量（theme / locale）是项目级概念，影响所有页面所有元素。它的预览切换应该在全局可见的位置。

**在底部工具栏的位置**：

```
底部工具栏:
  [预览] [视口: 375×812] [环境: theme=light ▾ | locale=zh ▾] [缩放: 100%]
                          ↑ 环境变量预览切换移到这里
```

### 7.3 状态组合预览 → 底部工具栏

**理由**：组合预览是一个全局操作（切换数据场景 + 全局状态 + 交互态的组合），不应塞在节点编辑面板里。

```
底部工具栏:
  [预览] [视口] [环境] [组合预览 ▾] [缩放]
                        ↑ 组合预览移到这里
```

### 7.4 领域状态绑定 → 右侧面板「高级」section

**理由**：节点对领域状态的响应（当 taskStatus=done 时 → 样式覆盖）是低频高级操作，不应与日常编辑混在一起。放在「高级」section 里，需要时展开即可。

## 十二、设计决策记录


| 决策             | 选择             | 备选                    | 理由                           |
| -------------- | -------------- | --------------------- | ---------------------------- |
| 交互态控件位置        | 仅在状态切换器        | 状态切换器 + StatesTab 双入口 | 双入口导致重复和混淆，是当前的核心问题          |
| 自定义状态管理位置      | 状态切换器内         | 单独的状态管理 section       | 创建/切换/编辑应在同一位置，减少跳转          |
| 「状态与领域」section | 完全移除           | 保留但简化                 | 这个 section 是混乱的根源，移除后信息架构更清晰 |
| 领域状态变量管理       | 左侧面板「数据」Tab    | 保留在右侧面板               | 页面级概念不应在节点编辑面板，作用域不对         |
| 环境变量预览切换       | 底部工具栏          | 保留在 StateContextBar   | 项目级概念，在底部工具栏全局可达             |
| Section 命名     | 用户语言（内容/外观/行为） | 技术语言（属性/样式/交互）        | 用户不按技术维度思考                   |
| 高级 section     | 默认折叠，收纳低频功能    | 每个低频功能单独 section      | 减少信息噪音，保护主编辑流                |
| 状态提示条          | 非 default 态时显示 | 始终显示                  | default 态是最常见状态，无需提醒         |
