## 一、第一性原理：预览模式解决什么根本问题？

### 1.1 编辑模式的天然矛盾

```
编辑模式下存在一个无法调和的矛盾：

  点击一个按钮 → 应该选中它（编辑行为）？还是触发它的 onClick（交互行为）？
  hover 一个元素 → 应该高亮选区（编辑行为）？还是显示 hover 样式（交互行为）？
  在输入框中打字 → 应该编辑元素名称？还是输入文本内容？

编辑模式的答案是：所有交互都是"编辑行为"
  · 点击 = 选中
  · hover = 显示选区
  · 拖拽 = 移动元素

这意味着设计师永远无法在编辑模式下体验真实的交互效果。
```

### 1.2 本质推导

```
预览模式 = 编辑器的"运行时"

  编辑模式 → 构建 Schema（结构 + 样式 + 交互 + 状态 + 数据）
  预览模式 → 运行 Schema（事件生效、状态切换、页面跳转）

预览模式的本质：
  · 移除 Canvas 编辑覆盖层（不再拦截事件）
  · React DOM 层变为可交互（事件冒泡正常工作）
  · Schema 定义的事件绑定真实执行
  · 交互状态（hover/active/focus）通过 CSS 伪类自动触发
  · 数据绑定正常解析，数据集切换继续有效

预览模式不是"另一个应用" → 它是同一份 Schema 的另一种呈现方式。
```

### 1.3 设计原则

```
1. 零构建 → 不需要"编译""导出"，一键进入，即时预览
2. 真实性 → 交互行为与最终产品一致（hover 是真 hover，点击是真点击）
3. 可控性 → 虽然是"运行"，但设计师仍可切换数据集/全局状态
4. 可退出 → 随时一键退出回到编辑模式，状态不丢失
5. 不修改 → 预览模式下不产生 Operation，Schema 不变
```

## 二、模式切换

### 2.1 进入预览模式

```
触发方式：
  · 顶部工具栏 → [▶ 预览] 按钮
  · 快捷键: Cmd+P

进入时的状态变化：

  ┌──────────────────────────────────────────────────────────┐
  │  编辑模式                  →    预览模式                   │
  ├────────────────────────────┼─────────────────────────────┤
  │  Canvas 覆盖层: 显示       │    Canvas 覆盖层: 隐藏       │
  │  DOM 层 pointer-events:    │    DOM 层 pointer-events:    │
  │    none                    │      auto                    │
  │  底部工具栏: 显示           │    底部工具栏: 隐藏          │
  │  左侧面板: 显示            │    左侧面板: 隐藏            │
  │  右侧面板: 显示            │    右侧面板: 隐藏/简化       │
  │  顶部工具栏: 完整          │    顶部工具栏: 简化          │
  │  鼠标: 编辑光标            │    鼠标: 默认/pointer        │
  │  事件绑定: 不执行          │    事件绑定: 真实执行         │
  │  交互状态: 手动切换查看     │    交互状态: CSS 伪类自动触发 │
  └────────────────────────────┴─────────────────────────────┘

进入动画（300ms ease-out）：
  · 左侧面板向左滑出
  · 右侧面板向右滑出
  · 底部工具栏向下淡出
  · 画布居中放大到视口实际尺寸（如果之前缩放过）
  · 视口容器边框从编辑灰色变为透明
  · 顶部出现预览控制条
```

### 2.2 退出预览模式

```
触发方式：
  · 预览控制条 → [■ 退出预览] 按钮
  · 快捷键: Escape 或 Cmd+P
  · 预览控制条的 [×] 按钮

退出时的状态恢复：
  · 所有面板恢复显示
  · Canvas 覆盖层恢复
  · 画布缩放/平移恢复到进入预览前的状态
  · 选中状态恢复（进入预览前选中的元素仍然选中）
  · 当前页面：
    - 如果在预览中跳转了页面 → 停留在跳转后的页面
    - 设计师可能想继续编辑跳转后的页面
  · 数据集/全局状态：保持预览期间的修改
    - 设计师可能切换到"空数据"后想继续编辑空状态下的 UI

重要：预览模式下不产生任何 Operation
  · 页面跳转只是切换预览内容，不是 setActiveScreen
  · 状态切换只是临时预览，不修改 node.activeState
  · 全局状态/数据集切换是例外 → 这些确实执行 Operation（因为设计师可能想保留切换结果）
```

## 四、预览模式下的交互行为

### 4.1 事件执行引擎

```
预览模式下，事件系统激活：

  Schema 中的 ComponentEvent → 注册到 DOM 真实事件

  注册策略:
    · 遍历所有有 events[] 的节点
    · 为对应 DOM 元素注册事件监听器:
      trigger: "click"     → addEventListener("click", handler)
      trigger: "hover"     → addEventListener("mouseenter", enterHandler)
                              addEventListener("mouseleave", leaveHandler)
      trigger: "focus"     → addEventListener("focus", handler)
      trigger: "blur"      → addEventListener("blur", handler)
      trigger: "longPress" → addEventListener("pointerdown", startTimer)
                              addEventListener("pointerup", cancelTimer)

  事件处理器 (handler) 执行 actions 数组中的行为:
    navigate     → 切换预览页面 + 转场动画
    setState     → 临时改变目标元素的 activeState → 重渲染
    openUrl      → window.open(url, target)
    setGlobalState → 执行 Operation（这个真正修改 Schema）
    toggleVisible  → 临时改变目标元素的 visible → 重渲染
```

### 4.2 页面跳转（navigate）

```
预览模式下触发 navigate 行为：

Step 1: 执行转场动画
  · 当前页面根据动画类型退出
  · 目标页面根据动画类型进入

  转场动画实现:
    slideLeft:  当前页面向左滑出，目标页面从右侧滑入
    slideRight: 当前页面向右滑出，目标页面从左侧滑入
    slideUp:    当前页面向上滑出，目标页面从底部滑入
    slideDown:  当前页面向下滑出，目标页面从顶部滑入
    fadeIn:     当前页面淡出，目标页面淡入（交叉淡化）
    zoomIn:     目标页面从中心放大进入
    none:       直接切换，无动画

  动画时长: 300ms
  缓动函数: ease-out

Step 2: 更新导航栈
  navigationStack.push(currentScreenId)
  currentPreviewScreen = targetScreenId

Step 3: 更新预览控制条
  · 页面名更新为目标页面名
  · [← 后退] 按钮变为可用

Step 4: 重新注册事件
  · 新页面的 DOM 渲染完成后
  · 为新页面的所有节点注册事件监听器
```

### 4.3 导航栈与后退

```
导航栈管理:

  navigationStack: string[] = []  // 存储访问过的 screenId
  currentIndex: number = 0

  跳转时: push 当前页面到栈中
  后退时: pop 栈顶页面，切换回去

  [← 后退] 按钮:
    · 栈非空时可用 → 点击回到上一个页面
    · 栈为空时置灰
    · 后退时的动画 = 进入时动画的反向
      (slideLeft 进入 → slideRight 后退)

  边界:
    · 后退到初始页面后 → [← 后退] 置灰
    · 不支持前进（只能后退），简化模型
```

### 4.4 交互状态（hover / active / focus）

```
预览模式下，交互状态通过 CSS 伪类自动触发：

  方案 A（推荐）: CSS 样式注入
    · 渲染引擎在预览模式下为每个有交互状态覆盖的节点注入 CSS:

      #node-xxx:hover {
        opacity: 0.85;
        cursor: pointer;
      }
      #node-xxx:active {
        opacity: 0.7;
        transform: scale(0.98);
      }
      #node-xxx:focus {
        border-color: #0D99FF;
        box-shadow: 0 0 0 2px rgba(13,153,255,0.2);
      }

    · 浏览器原生处理 :hover / :active / :focus
    · 零 JavaScript，性能最好

  方案 B（备选）: JavaScript 事件监听
    · 为每个节点注册 mouseenter/mouseleave 等事件
    · 手动切换样式
    · 更灵活但性能开销大

  推荐方案 A: CSS 注入 + 伪类
    · 在预览模式进入时，生成一个 <style> 标签注入所有节点的伪类样式
    · 退出预览时移除该 <style> 标签
```

### 4.5 表单输入

```
预览模式下，表单元素真实可交互：

  input → 可聚焦、可输入文字、显示 focus 样式
  textarea → 可聚焦、可输入多行文字
  select → 可展开下拉菜单、可选择
  checkbox → 可勾选/取消
  button → 可点击（触发绑定事件）

  表单输入的值不持久化:
    · 用户在 input 中输入的文字不写入 Schema
    · 退出预览后输入值丢失
    · 这是正确行为 → 预览模式只验证交互效果，不修改设计数据
```

### 4.6 滚动

```
预览模式下，视口内容可正常滚动：

  · overflow: auto / scroll 的元素 → 滚动条出现且可用
  · 列表数据渲染的长列表 → 可滚动浏览
  · 页面整体内容超出视口 → 页面级滚动

  这在编辑模式下不可用（因为 Canvas 覆盖层拦截了滚动事件）
  预览模式移除覆盖层后 → 滚动恢复原生行为
```

## 五、预览模式下的数据集与全局状态

### 5.1 数据集切换

```
预览控制条中的 [数据集▾] 下拉：

  · 切换后立即生效 → 画布内容根据新数据集重渲染
  · 列表长度变化 → 滚动位置重置到顶部
  · 数据绑定表达式重新解析
  · 列表重复渲染的数量跟着变

使用场景:
  预览模式下切换到"空数据" → 看到空状态引导页
  切换到"满数据" → 看到长列表的滚动和性能表现
  切换到"异常数据" → 看到错误提示 UI
```

### 5.2 全局状态切换

```
预览控制条中的 [全局状态▾] 下拉：

  · 切换后立即生效 → 所有绑定了该全局状态的元素联动变化
  · 样式/Props/可见性 根据新状态值重新计算

使用场景:
  taskStatus: "pending" → "done" → 看到页面从"进行中"变为"已完成"的全部视觉变化
  userRole: "admin" → "guest" → 看到管理员功能隐藏
```

### 5.3 数据集 + 全局状态 + 事件的组合验证

```
预览模式的终极价值 = 验证三者的组合效果:

  场景: 验证"空数据 + 访客用户"的完整交互流程

  1. 切换数据集 → "空数据"
  2. 切换全局状态 → userRole: "guest"
  3. 看到: 空列表 + 无管理功能 + "请登录" 引导
  4. 点击"请登录" → 跳转到登录页（navigate 事件执行）
  5. 在登录页输入用户名/密码（表单交互）
  6. 点击"登录" → 按钮状态变为 loading（setState）
  7. → 跳转到首页（navigate）

  整个流程无需导出代码、无需启动开发服务器。
```

  · 完整的组件树结构（Schema）

## 九、边界情况与异常处理

| 场景 | 预期行为 |
|------|---------|
| 预览模式下 Schema 被 AI 修改（WebSocket 推送）| 实时生效 → 画布重渲染（保持在预览模式） |
| navigate 目标页面不存在 | 控制台警告；预览控制条显示 ⚠️ "目标页面不存在" |
| 事件执行报错（如 openUrl 格式错误）| 控制台输出错误；不中断其他事件执行 |
| 连续快速点击导致多次跳转 | 导航做 debounce（300ms），避免动画冲突 |
| 预览模式下浏览器窗口 resize | 视口大小不变（设备尺寸固定），但居中位置更新 |
| 超长页面内容 | 页面级滚动正常工作；滚动条在视口内显示 |
| 列表数据极多（>1000 项）| 按数据量完全渲染（预览要真实）；如果性能差 → 这也是有价值的发现 |
| 进入预览时画布缩放为 30% | 自动重置为适合预览的缩放（视口原始尺寸或 zoomToFit） |
| 预览中修改了数据集/状态后退出 | 修改保留（它们是 Operation，已持久化） |
| 无任何事件绑定 | 预览模式仍可进入（验证静态 UI + 滚动 + hover 效果） |

## 十一、核心设计决策记录

| 决策 | 选择 | 理由 |
|------|------|------|
| 预览模式的实现方式？ | 隐藏覆盖层 + 启用 DOM 事件（同一 DOM 实例）| 零构建时间，即时切换；不需要新开窗口/iframe |
| 交互状态如何实现？ | CSS 伪类注入（:hover / :active / :focus）| 浏览器原生处理，性能最佳，与真实产品行为一致 |
| 预览模式下能否修改 Schema？ | 不能（除数据集/全局状态切换外）| 预览是"运行"不是"编辑"；保持关注点分离 |
| 页面跳转是否修改 activeScreen？ | 否（预览用独立导航栈）| 预览中的跳转是临时的，不应影响编辑器状态 |
| 退出后保留跳转页面？ | 是（停留在最后一个页面）| 设计师可能想编辑跳转后的页面 |
| 预览背景色？ | 深灰 #2C2C2C | 与编辑模式视觉区分明确；沉浸式预览感 |
| AI 测试何时实现？ | 远期（Phase 6），但架构预留 | 先满足人工预览需求；程序化 API 预留保证未来可扩展 |
| 面板在预览中显示？ | 全部隐藏（最大化预览区域）| 预览要沉浸；数据/状态切换通过控制条完成 |
