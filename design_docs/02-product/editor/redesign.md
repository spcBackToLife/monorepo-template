# 右侧属性面板 —— 信息架构重组方案（v2）

> **核心洞察：状态不是样式/属性/交互的附属品——状态是一切的上下文。**
>
> 设计一个按钮的 loading 态，要改的不只是颜色——还有文字（"提交"→"加载中"）、
> 子元素可见性（spinner 出现）、交互行为（点击失效）、透明度。
> 这些变化横跨样式、属性、结构、行为四个维度，但在用户心智中是**一个动作**：
> 「定义 loading 状态下的完整表现」。
>

## 一、现状问题分析

### 1.1 当前设计

```
┌────────────────────────────────────────────────┐
│  [样式]  [属性]  [状态]  [交互]  [数据]  [代码] │
│   ━━━                                          │
├────────────────────────────────────────────────┤
│  （Tab 对应内容区）                              │
└────────────────────────────────────────────────┘
```

6 个 Tab 在视觉上完全平等，暗示它们同等重要、同一层级。

### 1.2 四个结构性问题

**问题 1：使用频率严重不对等**

```
样式 Tab  ████████████████████████████████  60%+
属性 Tab  ████████████                      20%
状态 Tab  ████                              8%
交互 Tab  ███                               6%
数据 Tab  ██                                4%
代码 Tab  █                                 2%
```

将 60% 使用率的功能和 2% 使用率的功能放在同一级，
意味着高频操作区域被压缩（每个 Tab 只占 1/6 宽度），
Tab 栏文字挤成一团（320px / 6 ≈ 53px/Tab），可读性差。

**问题 2：作用域不一致**

```
节点级（选中哪个元素就编辑哪个）：
  ├── 样式  ── 该节点的 CSS
  ├── 属性  ── 该节点的 props
  ├── 状态  ── 该节点的状态变体
  └── 交互  ── 该节点的事件

页面级（与选中节点无关）：
  └── 数据  ── 当前页面的数据集

产出级（非编辑，只读输出）：
  └── 代码  ── JSON/React/HTML 预览
```

数据 Tab 是页面级概念，放在节点级面板中不自洽——
切换选中元素时数据 Tab 内容不变，违背面板的隐含契约。

**问题 3：操作流被割裂**

编辑一个按钮：改文字（属性 Tab）→ 调颜色（样式 Tab）→ 加 hover（状态 Tab）。
三个动作在用户心智中是一个连续的过程，但要切 3 次 Tab。

**问题 4（核心）：状态被错误地降维为「一个 Tab」**

这是最根本的设计缺陷。当前设计把「状态」当成和「样式」「属性」平级的一个维度，
但实际上状态是一个**横切所有维度的上下文**：

```
场景：设计一个提交按钮的 loading 状态

需要修改的不只是「状态 Tab」里的东西，而是：
  ├── 属性变化  ── text: "提交" → "加载中..."
  ├── 样式变化  ── backgroundColor: #1677FF → #999
  ├── 样式变化  ── cursor: pointer → not-allowed
  ├── 子元素变化 ── <Spinner> 从 hidden → visible
  └── 交互变化  ── click 事件被禁用
```

```
场景：设计一个用户列表模块在「空数据」场景下的表现

需要修改的更多：
  ├── 子元素 A（列表）    ── visible: false
  ├── 子元素 B（空状态图） ── visible: true
  ├── 子元素 C（提示文案） ── text: "暂无数据"
  ├── 子元素 D（操作按钮） ── text: "去添加", click → navigate: /add
  └── 自身样式            ── minHeight: 300px, justifyContent: center
```

在当前的 Tab 设计下，用户要完成上面的操作需要在
样式/属性/状态/交互之间来回反复切换。

**根因：Tab 是按「技术维度」（CSS/Props/Events/States）分的，
但用户的心智模型是按「状态上下文」来工作的。**

## 二、第一性原理：重新推导

### 2.1 用户的核心工作流是什么？

```
用户选中一个元素后，做的事情本质上只有两步：

第一步：定义这个元素在某个状态下的「完整表现」
        包括：长什么样（样式）、内容是什么（属性）、
              子元素怎么展示（可见性）、能做什么（交互）

第二步：切换到另一个状态，定义差异
        只编辑不同的部分，其余继承 default
```

### 2.2 状态的本质是什么？

状态是**对元素一组属性的条件覆盖**。它不局限于样式，而是横跨所有维度：

```
State = {
  条件（什么时候激活这个状态）
  +
  覆盖集（在这个状态下，哪些东西不同于 default）
    ├── 样式覆盖    styles: { backgroundColor: '#999' }
    ├── 属性覆盖    props: { text: '加载中...' }
    ├── 子元素覆盖  children: { spinner: visible, text: hidden }
    └── 行为覆盖    events: { click: disabled }
}
```

### 2.3 状态的四个层次

从作用域和触发方式来分，状态有四个层次：

```
┌─────────────────────────────────────────────────────┐
│                                                     │
│  层次 1：交互状态（CSS 伪类级）                      │
│  ─────────────────────────────                      │
│  scope: 单个元素                                    │
│  触发: 用户直接交互（鼠标/键盘）                     │
│  例子: hover, active, focus, disabled                │
│  影响: 主要是自身样式，偶尔影响子元素                 │
│                                                     │
│  层次 2：业务状态（组件级）                           │
│  ─────────────────────────                          │
│  scope: 组件及其子树                                 │
│  触发: 应用逻辑（API 返回、用户操作结果）             │
│  例子: loading, success, error, empty                │
│  影响: 自身样式 + 属性 + 子元素可见性 + 交互行为      │
│                                                     │
│  层次 3：数据场景（页面级）                           │
│  ─────────────────────────                          │
│  scope: 当前页面的多个组件                            │
│  触发: 数据源变化（不同的 Mock 数据集）               │
│  例子: 正常数据, 空数据, 异常数据, VIP 用户数据       │
│  影响: 页面内所有与数据绑定相关的元素                  │
│                                                     │
│  层次 4：全局状态（应用级）                           │
│  ─────────────────────────                          │
│  scope: 所有页面                                     │
│  触发: 全局配置/环境变化                              │
│  例子: theme=dark, locale=zh, role=admin             │
│  影响: 所有响应了该全局状态的元素                      │
│                                                     │
└─────────────────────────────────────────────────────┘
```

这四层共同构成一个**状态组合空间**：

```
某一时刻的 UI 表现 = f(
  全局状态:  { theme: 'dark', role: 'admin' },
  数据场景:  { dataSet: '空数据' },
  业务状态:  { loginForm: 'error' },
  交互状态:  { submitButton: 'hover' }
)
```

编辑器的职责是让用户**在这个状态组合空间中自由导航和编辑**。

## 三、新方案：状态驱动的统一编辑面板

### 3.1 核心设计理念

```
旧模型：  Tab 切换 → 编辑不同维度 → 状态是其中一个维度
新模型：  状态切换 → 在当前状态下编辑所有维度 → 状态是最高层上下文
```

**不再按技术维度分 Tab，而是在一个统一面板中，用「状态上下文」驱动编辑内容。**

### 3.2 面板整体结构

```
┌──────────────────────────────────────┐
│  右侧编辑面板                         │
├──────────────────────────────────────┤
│                                      │
│  ┌─ 元素信息条 ─────────────────────┐│
│  │  div.submit-btn  [🔒][👁] nd_xx ││
│  └──────────────────────────────────┘│
│                                      │
│  ┌─ 状态上下文栏（始终可见）────────┐ │
│  │                                  │ │
│  │  ● default ○ hover ○ loading    │ │
│  │  ○ disabled ○ error    [+]      │ │
│  │                                  │ │
│  │  数据场景: [正常数据 ▾]          │ │
│  │                                  │ │
│  └──────────────────────────────────┘ │
│                                      │
│  ═══ 以下为「loading」态的编辑 ═══    │
│                                      │
│  ┌─ 滚动内容区 ─────────────────────┐│
│  │                                  ││
│  │  ▸ 属性 ─────────────────────── ││
│  │  ▸ 样式 ─────────────────────── ││
│  │  ▸ 子元素可见性 ─────────────── ││
│  │  ▸ 交互行为 ─────────────────── ││
│  │                                  ││
│  │  ── 辅助 ───────────────────── ││
│  │  ▸ 代码预览                     ││
│  │                                  ││
│  └──────────────────────────────────┘│
│                                      │
└──────────────────────────────────────┘
```

### 3.3 状态上下文栏设计

状态上下文栏是整个面板的**最高层控制器**，始终固定在顶部不随内容滚动。

```
┌──────────────────────────────────────┐
│  状态上下文                           │
├──────────────────────────────────────┤
│                                      │
│  ── 交互态 ──                        │
│  ◉ default  ○ hover  ○ active       │
│  ○ focus  ○ disabled                │
│                                      │
│  ── 业务态 ──                        │
│  [default ▾]  [+添加状态]            │
│  可选: loading / success / error     │
│                                      │
│  ── 数据场景 ──                      │
│  [正常数据 ▾]  [管理数据集]          │
│  可选: 空数据 / 异常数据 / VIP数据   │
│                                      │
│  ── 全局状态 ──                      │
│  theme: [light ▾]  role: [user ▾]   │
│                                      │
└──────────────────────────────────────┘
```

**交互规则：**

- 切换任何状态 → 下方编辑区内容立即更新 + 画布实时预览
- 当前非 default 状态 → 编辑区顶部显示醒目提示条
- 被覆盖的字段显示蓝色标记 🔵，表示「在此状态下有定制值」
- 未被覆盖的字段显示灰色值（继承自 default），编辑即创建覆盖

**状态上下文栏的自适应收起：**

- 若元素没有定义业务状态 → 业务态行隐藏
- 若页面没有数据集 → 数据场景行隐藏
- 若没有全局状态变量 → 全局状态行隐藏
- 极简元素（无状态）→ 状态上下文栏只显示交互态一行，占用极少空间

## 四、编辑区各节详细设计

### 4.1 属性区

紧贴状态上下文栏下方，展示当前元素在**当前状态**下的属性值。

```
┌──────────────────────────────────────┐
│  ▾ 属性                              │
├──────────────────────────────────────┤
│                                      │
│  文本  [加载中...      ]  🔵 🔗     │
│        ↑ loading 态覆盖     ↑ 数据绑定│
│        default: "提交"               │
│                                      │
│  类名  [submit-btn     ]             │
│                                      │
│  (按元素类型动态展示不同属性)         │
│                                      │
└──────────────────────────────────────┘
```

**状态覆盖的视觉语言：**

```
普通值 (default 态):     [提交         ]         正常显示
覆盖值 (非 default 态):  [加载中...    ]  🔵     蓝色圆点标记
继承值 (未覆盖):         [提交         ]  灰色    值显示为灰色，编辑即创建覆盖
```

- 🔵 点击蓝色圆点 → 删除此属性在当前状态下的覆盖，回退到继承值
- 🔗 点击绑定图标 → 弹出表达式编辑器 `{{ data.user.name }}`

### 4.2 样式区

CSS 八组编辑器，内容与原「样式 Tab」基本一致，
但每个属性值都感知当前状态上下文：

```
┌──────────────────────────────────────┐
│  ▾ 背景 Fill                         │
├──────────────────────────────────────┤
│                                      │
│  颜色  [#999999] [🎨]  🔵           │
│        ↑ loading 态覆盖值             │
│        default: #1677FF              │
│                                      │
│  cursor [not-allowed ▾]  🔵          │
│        default: pointer              │
│                                      │
└──────────────────────────────────────┘
```

- 在 default 态下编辑 → 调用 `updateStyle` Operation
- 在非 default 态下编辑 → 调用 `updateState` Operation（写入状态覆盖）
- 蓝色标记 🔵 只在非 default 态时出现，表示「此值被当前状态覆盖」

### 4.3 子元素可见性区（新增）

这是本次重组的**关键新增区域**。解决的问题：
在不同状态下，一个容器的子元素应该展示哪些、隐藏哪些。

```
┌──────────────────────────────────────┐
│  ▾ 子元素可见性                       │
├──────────────────────────────────────┤
│                                      │
│  当前状态: loading                    │
│                                      │
│  ☑ icon-submit     (图标)            │
│  ☑ spinner         (加载动画)   🔵   │
│  ☐ text-submit     (提交文字)   🔵   │
│  ☐ text-error      (错误提示)   🔵   │
│                                      │
│  🔵 = 此状态下与 default 不同        │
│                                      │
│  default 态:                         │
│  ☑ icon-submit  ☐ spinner            │
│  ☑ text-submit  ☐ text-error         │
│                                      │
└──────────────────────────────────────┘
```

**设计要点：**

- 只有容器类元素（有 children）才显示此区域
- 叶子元素（img、input 等）不显示
- 勾选/取消 → 调用 `setNodeVisible` Operation（状态感知）
- 支持画布上直接点击子元素的「眼睛」图标切换可见性
- 在 default 态下修改 → 修改元素的 `visible` 属性
- 在非 default 态下修改 → 写入状态的 `visibleWhen` 条件

### 4.4 交互行为区

展示当前元素在**当前状态**下的事件绑定。

```
┌──────────────────────────────────────┐
│  ▾ 交互行为                           │
├──────────────────────────────────────┤
│                                      │
│  当前状态: loading                    │
│                                      │
│  click  →  (已禁用)            🔵   │
│            default: navigate /home   │
│                                      │
│  hover  →  setState: hover           │
│            (与 default 相同)          │
│                                      │
│  [+ 添加事件]                         │
│                                      │
└──────────────────────────────────────┘
```

**设计要点：**

- 事件卡片显示在当前状态下的行为
- 若当前状态禁用了某事件 → 显示「已禁用」+ 删除线
- 若与 default 相同 → 正常显示但不带蓝色标记
- 若与 default 不同 → 蓝色标记 🔵
- 添加事件 → 事件自动带上当前状态上下文

### 4.5 代码预览区

折叠在最底部，低频使用。

```
┌──────────────────────────────────────┐
│  ▸ 代码预览                           │
├──────────────────────────────────────┤
│  (折叠状态，点击展开)                 │
│                                      │
│  展开后:                              │
│  [节点] [页面]    [JSON] [React]     │
│  ┌────────────────────────────────┐  │
│  │  <button disabled ...>         │  │
│  │    <Spinner />                 │  │
│  │    加载中...                   │  │
│  │  </button>                     │  │
│  └────────────────────────────────┘  │
│  [📋 复制]                           │
└──────────────────────────────────────┘
```

## 五、数据面板 —— 提升为页面级独立面板

### 5.1 为什么要独立？

数据集管理是**页面级**概念（与选中哪个节点无关），
放在节点级面板中概念不自洽。

但数据**绑定**（某个属性引用某个数据路径）是节点级操作，
保留在属性区中以 🔗 图标内联。

### 5.2 位置

```
左侧面板新增「数据」入口:
  [组件树] [页面] [数据]

数据面板内容（页面级）:
  ├── 数据集选择/切换
  ├── JSON 编辑器
  ├── 数据预设模板
  └── 全景截图任务
```

数据场景的切换同时出现在两个位置：

1. 左侧数据面板 → 管理和编辑数据集详情
2. 右侧状态上下文栏 → 快速切换当前数据场景（下拉选择）

## 六、完整用户工作流示例

### 6.1 场景：设计提交按钮的多状态表现

```
1. 选中 submit-button

2. 状态上下文: ● default
   属性:  text = "提交"
   样式:  bg=#1677FF, color=#FFF, cursor=pointer
   子元素: ☑ icon ☐ spinner ☑ text
   行为:  click → navigate: /dashboard

3. 切换状态: ○ default ● hover
   样式:  bg=#0056D2 🔵  (其余继承 default)

4. 切换状态: ○ default ○ hover ● loading
   属性:  text = "加载中..." 🔵
   样式:  bg=#999 🔵, cursor=not-allowed 🔵
   子元素: ☑ icon ☑ spinner 🔵 ☐ text 🔵
   行为:  click → (禁用) 🔵

5. 切换状态: ● disabled
   样式:  opacity=0.5 🔵, cursor=not-allowed 🔵
   行为:  click → (禁用) 🔵
```

**全程在同一个面板中完成，零切换。
每个状态下的完整表现一目了然。**

### 6.2 场景：设计用户列表模块在不同数据下的表现

```
1. 选中 user-list-module

2. 状态上下文: 业务态=default, 数据场景=[正常数据]
   子元素:  ☑ list-header ☑ list-body ☐ empty-state ☐ error-state
   样式:    正常布局
   行为:    scroll → loadMore

3. 切换数据场景: [空数据]
   画布实时切换到空数据 Mock → 列表区域无内容
   子元素:  ☐ list-body 🔵 ☑ empty-state 🔵
   属性:    empty-state.text = "暂无数据" 🔵
   行为:    empty-state.button → click → navigate: /add 🔵

4. 切换业务态: error
   子元素:  ☐ list-body 🔵 ☐ empty-state ☑ error-state 🔵
   属性:    error-state.text = "加载失败" 🔵
   行为:    error-state.retry → click → reload 🔵

5. 切换全局状态: theme = dark
   样式:   bg=#1a1a1a 🔵, color=#eee 🔵
```

### 6.3 对比旧方案


| 操作       | 旧方案（6 Tab）              | 新方案（状态驱动）              |
| -------- | ----------------------- | ---------------------- |
| 改文字      | 切到「属性」Tab               | 直接在属性区改                |
| 改颜色      | 切到「样式」Tab               | 直接在样式区改                |
| 定义 hover | 切到「状态」Tab               | 状态栏点 hover             |
| hover 改色 | 切回「样式」Tab               | 直接在样式区改（自动写入 hover 覆盖） |
| 子元素可见性   | 切到「状态」Tab → visibleWhen | 直接在子元素区勾选              |
| 禁用事件     | 切到「交互」Tab               | 直接在行为区修改               |
| 看代码      | 切到「代码」Tab               | 展开底部代码预览               |
| 管理数据集    | 切到「数据」Tab               | 左侧数据面板                 |
| 切换数据场景   | 切到「数据」Tab               | 状态栏下拉选择                |

## 七、与旧 Tab 的功能映射


| 原 Tab      | 原内容                         | 新位置                | 理由               |
| ---------- | --------------------------- | ------------------ | ---------------- |
| 样式         | CSS 八组编辑                    | **样式区**            | 核心编辑，内容不变，增加状态感知 |
| 属性         | 元素/组件 Props                 | **属性区**（样式区上方）     | 属性和样式是同一编辑流的两个层面 |
| 状态 → 交互态   | hover/active/focus/disabled | **状态上下文栏 → 交互态**   | 提升为顶层上下文选择器      |
| 状态 → 业务态   | 自定义状态定义                     | **状态上下文栏 → 业务态**   | 提升为顶层上下文选择器      |
| 状态 → 全局绑定  | 全局状态变量响应规则                  | **状态上下文栏 → 全局状态**  | 全局状态是上下文的一部分     |
| 状态 → 可见性条件 | visibleWhen                 | **子元素可见性区**        | 可见性是状态下的具体表现之一   |
| 状态 → 样式覆盖  | 状态下的样式差异                    | **样式区**（状态感知，蓝色标记） | 覆盖应在被覆盖的字段旁展示    |
| 交互         | 事件卡片、添加事件                   | **交互行为区**          | 内容不变，增加状态感知      |
| 数据 → 数据集管理 | CRUD、JSON 编辑                | **左侧面板 → 数据子面板**   | 页面级，独立出来         |
| 数据 → 场景切换  | 切换当前数据集                     | **状态上下文栏 → 数据场景**  | 数据场景是状态的一个层次     |
| 数据 → 绑定表达式 | `{{data.xxx}}`              | **属性区内联 🔗**       | 绑定是属性编辑的一部分      |
| 代码         | JSON/React/HTML             | **代码预览区**（底部折叠）    | 低频，不值得占一个 Tab    |

## 八、需要额外思考的设计问题

### 8.1 状态的定义与编辑分离

```
问题：状态的「定义」（创建新状态、命名、设置条件）
     和「在状态下编辑」（调整样式/属性/可见性）
     是两个不同的操作。界面上如何区分？

方案：
  · 状态上下文栏负责「选择」和「创建」状态
  · 下方编辑区负责「在选中状态下编辑」
  · 创建新状态 → 弹出小面板（输入名称、选择类型）
  · 编辑状态条件 → 长按/右键状态标签打开编辑
```

### 8.2 子元素的状态如何跟随父元素

```
问题：父容器切换到 loading 状态时，子元素的可见性变化
     是记录在父元素的状态覆盖中，还是记录在子元素自己的条件上？

方案：两层机制
  · 父元素的「子元素可见性区」→ 控制直接子元素在此状态下的 visible
  · 子元素自己的 visibleWhen → 基于表达式的条件可见性

  优先级：子元素自己的 visibleWhen > 父元素的子元素可见性控制
  
  场景对比：
  · 简单开关（loading 时隐藏列表显示 spinner）→ 用父元素子元素可见性
  · 条件逻辑（当 data.count > 0 时显示）→ 用子元素自己的 visibleWhen
```

### 8.3 状态组合时的冲突

```
问题：如果 hover 态覆盖了背景色，loading 态也覆盖了背景色，
     当 hover + loading 同时激活时，用哪个？

方案：明确的优先级层次（与 CSS 级联类似）
  交互态 > 业务态 > 全局状态 > 数据场景 > default
  
  同层内：后定义的覆盖先定义的

编辑器中的表达：
  当处于 loading + hover 组合时，样式区显示最终计算值
  鼠标悬停在属性上 → tooltip 显示覆盖链：
  "背景色: #333
   ← hover 覆盖 (最终生效)
   ← loading 覆盖 (#999)
   ← default (#1677FF)"
```

### 8.4 模块级状态的编辑体验

```
问题：一个「用户卡片」模块在 loading 态下，
     需要同时编辑自身 + 5 个子元素的表现。
     在当前面板中只能编辑「选中的一个元素」。
     如何高效地编辑整个子树的状态差异？

方案（渐进式）：

  MVP：逐个选中子元素，在同一个状态上下文下分别编辑。
       父元素的「子元素可见性区」能快速批量设置 visible/hidden。

  进阶：选中父元素时，子元素可见性区支持「展开子元素属性」
       ——在不切换选中的情况下，快速调整子元素的关键属性（文本、颜色）。

  远期：组件模板的「状态蓝图」功能——
       在组件模板定义中，预定义整个子树在各状态下的差异。
       实例化时自动继承，支持覆盖。
```

### 8.5 数据场景与业务状态的关系

```
问题：数据场景（来自数据集切换）和业务状态（组件自身定义的状态）
     有时是同一件事：「空数据」既是数据场景，也是组件的业务状态。
     如何避免重复定义？

方案：
  · 数据场景影响的是「数据绑定表达式的值」——自动生效
  · 业务状态影响的是「样式/属性/可见性的覆盖」——手动定义
  · 两者通过 visibleWhen 表达式连接：
    visibleWhen: "{{ data.users.length === 0 }}" → 自动响应数据场景
  · 编辑器在状态上下文栏同时提供两个维度的切换，
    用户切换数据场景后，可以看到 visibleWhen 的实时效果
```

## 九、MVP 与渐进实施

### Phase 1：状态上下文栏 + 样式/属性合并（核心骨架）

```
改动范围：PropertyPanel, StylesTab, PropsTab, StatesTab
交付目标：
  · 6 Tab → 统一面板 + 状态上下文栏
  · 属性区 + 样式区在一个滚动面板中
  · 状态上下文栏支持交互态和业务态切换
  · 切换状态 → 样式区显示覆盖值 + 蓝色标记
  · 编辑值 → 自动写入当前状态的覆盖

技术要点：
  · RightTabType 从 6 值 → 废弃或保留为 section 折叠记忆
  · activeRightTab → 替换为 activeStateContext
  · StylesTab 和 PropsTab 合并为 DesignPanel
  · handleChange 感知当前状态上下文（default → updateStyle, 非 default → updateState）
```

### Phase 2：子元素可见性区 + 交互行为区

```
改动范围：新增 ChildrenVisibilitySection, 改造 InteractionsTab
交付目标：
  · 选中容器 → 显示子元素可见性区，支持按状态切换 visible
  · 交互行为区感知状态上下文（显示当前状态下的事件列表）
  · 事件支持条件执行（仅在某状态下触发）

技术要点：
  · setNodeVisible Operation 需要支持状态覆盖
  · 事件模型增加 condition 字段：{ state: 'loading', action: 'disable' }
```

### Phase 3：数据面板独立 + 数据场景联动

```
改动范围：DataTab → 左侧 DataPanel, 状态上下文栏增加数据场景
交付目标：
  · 数据面板从右侧移到左侧
  · 状态上下文栏的「数据场景」下拉与数据集联动
  · 切换数据场景 → 画布实时预览 + 属性区绑定值更新
  · 属性区 🔗 内联数据绑定保持可用

技术要点：
  · LeftPanel 新增 data tab
  · 状态上下文栏的数据场景切换 → editorStore.switchDataSet
  · 属性区的绑定表达式渲染需要感知当前数据集
```

### Phase 4：精打细磨

  · 状态覆盖链的 tooltip（显示各层覆盖来源）

## 十、设计决策记录


| 决策         | 选择                 | 备选                 | 理由                          |
| ---------- | ------------------ | ------------------ | --------------------------- |
| 面板顶层组织方式   | 状态上下文 + 统一编辑面板     | 三模式 Tab / 保持 6 Tab | 状态是一切的上下文，不应被降维为一个 Tab      |
| Tab 是否完全消除 | 是，改为可折叠 Section    | 保留 2-3 个 Tab       | 所有内容属于同一个编辑流，Section 足够分隔   |
| 状态上下文栏位置   | 固定在面板顶部不滚动         | 随内容滚动 / 浮层         | 状态选择器必须始终可见，是最高频操作          |
| 覆盖值的视觉语言   | 蓝色圆点 🔵 + 灰色继承值    | 高亮背景 / 边框 / 删除线    | 蓝色圆点简洁不侵入，灰色值明确区分「继承 vs 覆盖」 |
| 子元素可见性的位置  | 独立 Section（样式区下方）  | 属性区内 / 左侧组件树内      | 可见性是状态的重要表现维度，值得独立区域        |
| 交互行为的位置    | 独立 Section（子元素区下方） | 保持为独立 Tab          | 行为也受状态影响，应在统一面板中            |
| 数据面板位置     | 左侧面板独立             | 保留在右侧 / 顶部工具栏      | 页面级概念不应与节点级面板混在一起           |
| 代码预览位置     | 底部折叠 Section       | 独立模式/Tab           | 使用频率极低，不值得占用一个 Tab          |
| 数据场景在哪切换   | 状态上下文栏 + 左侧数据面板双入口 | 仅左侧 / 仅右侧          | 右侧快速切换 + 左侧详细管理，两种场景都覆盖     |
