## 一、第一性原理：资产系统解决什么根本问题？

### 1.1 本质推导

```
设计的核心矛盾：
  一致性 vs 灵活性

  一致性要求: 同一个按钮在 50 个页面中长得一样、行为一样
  灵活性要求: 这个页面的按钮文案是"登录"，那个页面是"注册"

没有资产系统 → 设计师复制粘贴组件 → 50 个页面 50 份独立副本
  · 修改按钮圆角 → 需要改 50 次
  · 忘改了 3 个 → 一致性破坏

有资产系统 → 组件模板 + 实例引用
  · 修改模板的圆角 → 50 个实例自动同步
  · 每个实例只配置自己的 Props（文案、颜色等）
  · 一致性和灵活性同时满足
```

### 1.2 两层资产的推导

```
更深一层的矛盾：
  复用效率 vs 设计自由度

  场景 A: 从零开始设计一个表单
    需要最大自由度 → 但效率低

  场景 B: 快速搭建一个后台页面
    需要最大效率 → 但自由度低（直接用模板）

解法: 两层资产模型

  骨架组件 → 给场景 A：只定义结构和行为，不含视觉风格
    · 拖入后设计师自由发挥视觉设计
    · 最大自由度 + 结构保障

  风格组件 → 给场景 B：包含完整的视觉风格
    · 拖入后直接可用，微调 Props 即可
    · 最大效率 + 可配置性
```

### 1.3 资产系统的三个组成部分

```
资产系统 = 组件资产 + 静态资源 + 素材设计器

  组件资产: 可复用的 Schema 片段（带 Props + 状态 + 事件）
    · ComponentTemplate → 模板定义
    · instantiateTemplate → 实例化
    · reference / detached → 引用/脱离模式

  静态资源: 项目中使用的图片、图标、动画、视频等文件
    · 上传到项目 assets/ 目录
    · 通过 asset:// 协议在 Schema 中引用
    · 统一管理、去重、CDN 化

  素材设计器: 为元素制作精细视觉素材的子面板
    · Canvas 绘图器（简单 SVG 图标）
    · 背景拼装（渐变 + 纹理 + 图片叠加）
    · 右键 → "设计素材" 进入
```

## 二、组件资产

### 2.1 数据结构

```typescript
// 已有（design-schema 中定义），扩展字段
interface ComponentTemplate {
  id: string;
  name: string;                     // "LoginForm" / "NavBar" / "UserCard"
  description?: string;             // "通用登录表单，支持用户名密码登录"
  category: string;                 // "表单" | "导航" | "卡片" | "布局" | "数据展示" | "反馈"
  tags: string[];                   // ["登录", "表单", "认证"]
  thumbnail?: string;               // 缩略图 base64 或 URL

  // 核心内容
  schema: ComponentNode;            // 组件的 Schema 片段（子树）
  propDefinitions: ComponentPropDefinition[];  // 暴露的可配置项（见 06-component-props）
  propBindings: PropBinding[];      // Props 到内部节点的映射

  // 分类
  kind: "skeleton" | "styled";     // 骨架组件 or 风格组件
  scope: "project" | "team" | "global";  // 可见范围

  // 元数据
  version: number;                  // 版本号（每次更新 +1）
  author?: string;                  // 创建者
  createdAt: string;
  updatedAt: string;
}
```

```

│  │ └──────────────────┘ │          │ │ └──────────────┘ │ │    │

│  特点:                              特点:                       │
│  · 无色彩、无圆角、无阴影           · 完整的视觉风格             │
│  · 有结构、有交互、有状态定义        · 可直接使用                 │
│  · 最小默认样式 (border:1px)        · 适合快速搭建               │
│  · 设计师自由发挥视觉               · 微调 Props 即可            │

│                                                                 │
└─────────────────────────────────────────────────────────────────┘

判定标准:
  骨架组件 → kind: "skeleton"
    schema 中的样式仅包含: 布局属性 (display/flex/gap/padding) + 最小边框
    不包含: 颜色/渐变/阴影/圆角/字体风格

  风格组件 → kind: "styled"
    schema 中包含完整的视觉样式
    通常从骨架组件 + 设计师的视觉设计生成
```

### 2.3 组件资产的生命周期

```
创建 → 发布 → 实例化 → 更新 → 版本管理

Step 1: 创建
  · 在画布上设计一组元素
  · 右键 → "保存为组件资产"
  · 定义名称/分类/Props（见 06-component-props 第四节）
  · 选择: 骨架组件 or 风格组件
  · 选择: 项目级 / 团队级

Step 2: 发布
  · 项目级 → 仅当前项目可用，立即可用
  · 团队级 → 团队内所有项目可用，需审核（远期）

Step 3: 实例化
  · 从资产库面板拖入画布 → instantiateTemplate Operation
  · 生成一个组件实例节点: type = "component:LoginForm"
  · 默认为引用模式 (reference)

Step 4: 更新
  · 修改模板 → 所有引用模式的实例自动同步
  · 版本号 +1
  · 实例的自定义 Props 值保留（只同步模板结构/样式/propDefinitions）

Step 5: 版本管理（远期）
  · 记录每次更新的 diff
  · 实例可以选择锁定到特定版本
  · 版本回滚
```

```

### 4.3 从骨架组件生成风格组件

```

  可以将"骨架 + 风格"保存为新的风格组件:

  1. 从资产库拖入 "通用表单骨架" → 得到实例
  2. 在实例上配置 Props + 设计视觉风格（颜色/圆角/阴影...）
  3. 右键 → "保存为风格组件"
  4. 自动继承骨架的 Props 定义 + 添加当前的视觉样式
  5. 新的风格组件出现在资产库中

  骨架组件是风格组件的"母版" → 风格组件是骨架的"皮肤"
```

## 五、静态资源管理

### 5.1 资源类型

```
┌──────────────┬───────────────────┬────────────────────────────┐

│  视频         │  MP4 / WebM       │  video.src / 背景视频       │

└──────────────┴───────────────────┴────────────────────────────┘

### 5.2 asset:// 协议

```
在 Schema 中引用静态资源使用 asset:// 协议:

  img.props.src = "asset://images/logo.png"
  div.styles.backgroundImage = "url(asset://images/bg-pattern.jpg)"
  img.props.src = "asset://icons/arrow-right.svg"

协议格式:
  asset://<类别>/<文件名>

解析规则:
  · 编辑器中: 解析为项目 assets 目录的文件 URL
  · 导出代码时: 解析为相对路径 "./assets/images/logo.png"
  · 发布后: 解析为 CDN URL

存储结构:
  project_root/
    assets/
      images/      → PNG / JPG / WebP
      icons/       → SVG 图标
      animations/  → Lottie / PAG
      videos/      → MP4
```

### 5.4 上传流程

```
点击 [上传] 或拖拽文件到素材面板:

  1. 文件选择/拖拽
  2. 格式校验（是否为支持的格式）
  3. 大小校验（单文件 < 10MB，可配置）
  4. 自动分类（根据文件扩展名放入对应目录）
  5. 重名检查:
     · 同名文件已存在 → 提示"替换" / "重命名" / "取消"
  6. 上传到服务端 → 返回 asset:// 路径
  7. 素材面板中出现新素材

批量上传:
  · 支持一次选择/拖拽多个文件
  · 进度条显示整体上传进度
  · 单个文件失败不影响其他文件
```

## 六、素材设计器

### 6.1 入口与定位

```
素材设计器是一个 Overlay 子面板，用于为元素制作精细视觉素材。

入口: 选中元素 → 右键 → "设计素材..."
场景:
  · 为按钮设计一个精细的渐变背景
  · 绘制一个简单的 SVG 图标
  · 拼装一个由多层图片叠加的背景
  · 调整插入素材的混合模式和滤镜

这不是一个完整的矢量编辑器（那是 Figma 的领域），
而是一个"素材加工站"——将素材处理为可用于元素的 background / icon。
```

### 6.3 素材设计器能力

```
图层类型:
  · 纯色层 → 单色填充
  · 渐变层 → 线性/径向/角度渐变（多色标）
  · 图片层 → 从素材库选择图片，调整位置/大小/裁切
  · 图案层 → 重复平铺的纹理图案
  · SVG 层 → 简单的矢量形状（矩形/圆/路径）

每个图层的通用属性:
  · 混合模式 (blend mode): normal / multiply / screen / overlay / ...
  · 透明度 (opacity): 0-1
  · 可拖拽排序（调整图层堆叠顺序）
  · 可显示/隐藏

导出方式:
  · 应用为 CSS → 生成 background / background-image CSS 属性
  · 导出为 SVG → 保存为项目素材 (asset://icons/xxx.svg)
  · 导出为 PNG → 保存为项目素材 (asset://images/xxx.png)
  · 导出为 CSS 代码 → 复制到剪贴板

[应用到元素]:
  · 将设计结果写入当前元素的 styles.background 等属性
  · 如果涉及图片 → 自动上传到项目素材 + 使用 asset:// 引用
```

## 七、实例管理

### 7.1 引用模式 (reference)

```
引用模式是默认模式——实例跟随模板更新:

  · 实例的结构和样式来自模板
  · 实例只能修改 Props 值（propDefinitions 定义的可配置项）
  · 不能进入实例内部选中子节点编辑
  · 模板更新 → 实例自动同步

画布中的表现:
  · 选中组件实例 → 蓝色边框 + 紫色角标 📦
  · 双击实例 → 不进入内部（提示"引用模式下不可编辑内部结构"）
  · 右侧属性面板 → 只显示 Props 配置

组件树中的表现:
  · 📦 LoginForm (引用)
  · 不可展开查看子节点（或展开为只读灰色）
```

### 7.2 脱离模式 (detached)

```
脱离模式 = 实例变为独立副本:

操作: 属性面板 → [分离实例] 或 右键 → "分离组件实例"
确认: "分离后将不再跟随模板更新，确认？"

脱离后:
  · 实例变为普通的节点子树
  · 可进入内部选中任何子节点编辑
  · 可修改结构（增删子节点）
  · 模板更新不再影响此实例
  · node.templateRef.mode 从 "reference" 变为 "detached"
  · 组件树中: 📦 LoginForm (已脱离) → 可展开
```

### 7.3 重新同步

```
引用模式下，手动触发同步:

操作: 属性面板 → [重新同步]
行为:
  · 将实例的 Props 值重置为模板默认值
  · 清除所有自定义 Props
  · 结构更新到模板最新版本

场景: 实例的 Props 配置乱了，想恢复到"出厂设置"
```

## 九、AI 集成

```
AI 使用资产系统的场景:

1. 搜索组件:
   MCP Tool: search_components({ query: "登录", category: "表单" })
   → 返回匹配的组件模板列表

2. 实例化组件:
   MCP Tool: instantiate_template({ templateId: "tpl-login", parentId: "root", mode: "reference" })
   → 在画布中创建组件实例

3. 配置实例 Props:
   MCP Tool: update_component_props({ nodeId: "xxx", props: { title: "欢迎回来" } })
   → 修改实例的 Props

4. 保存为资产（远期）:
   MCP Tool: save_as_template({ nodeId, name: "NewCard", category: "卡片", kind: "styled" })
   → AI 设计完一个组件后自动保存为资产

AI 操作的安全边界:
  · 引用模式实例 → AI 只能修改 propDefinitions 暴露的 Props
  · 不能修改实例的内部结构（除非先脱离）
  · 遵循与人类用户相同的约束
```

## 十、边界情况与异常处理

| 场景 | 预期行为 |
|------|---------|
| 删除被实例引用的模板 | 提示 "该模板有 N 个实例引用，确认删除？" → 删除后实例自动脱离 |
| 模板更新后实例布局错乱 | 实例上显示 ⚠️ "模板已更新" 通知，用户可选择同步或脱离 |
| 上传同名文件 | 提示 "替换" / "重命名 (logo-2.png)" / "取消" |
| 上传超大文件 (> 10MB) | 拒绝上传 + 提示 "文件大小超过限制 (10MB)" |
| 上传不支持的格式 | 拒绝 + 提示 "不支持的文件格式" |
| 素材被删除但 Schema 仍引用 | 画布中该 img 显示破碎图标 🖼️‍💔；组件树标记 ⚠️ |
| 团队组件与本地组件同名 | 允许（用 ID 区分）；面板中各自显示在不同区域 |
| 骨架组件中有样式（创建者违规）| 不强制校验，但在分类标签旁显示 ℹ️ 提示 |
| 实例化时 Props 默认值缺失 | 使用 propDefinition.defaultValue；如果也缺失 → 空值 |
| 素材设计器中图层过多 (> 20) | 允许但给出性能提示；图层列表可滚动 |

## 十二、核心设计决策记录

| 决策 | 选择 | 理由 |
|------|------|------|
| 资产分几层？ | 两层：骨架组件 + 风格组件 | 覆盖"最大自由度"和"最大效率"两个极端场景 |
| 实例默认模式？ | 引用模式 (reference) | 一致性优先——大多数场景需要跟随模板更新 |
| 脱离是否可逆？ | 不可逆（单向操作） | 简化心智模型；重新关联的场景极少且复杂 |
| 静态资源引用方式？ | asset:// 自定义协议 | 解耦存储位置与引用路径；编辑/导出/部署各自解析 |
| 素材设计器的定位？ | "素材加工站"而非"矢量编辑器" | 不与 Figma 竞争矢量编辑；聚焦于背景/图标的快速制作 |
| 组件内部能否编辑？ | 引用模式不可，脱离模式可 | 引用模式保护一致性；需要自由编辑时明确脱离 |
| 资产存储在哪？ | project.componentAssets（项目级） | 项目是资产的最小管理单元；团队级通过复制/共享实现 |
| 缩略图如何生成？ | 保存时自动截图 | 零成本；可手动替换 |
