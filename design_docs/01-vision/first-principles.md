# 第一性原理分析

> 本文档记录了 DesignUI 项目从第一性原理出发的所有核心推导。这是整个项目的"思想地基"，很少修改，但会在遇到新的根本性问题时追加新的 Q&A。

---

## 1. 当前设计工具的本质问题

回到最底层思考：**设计工具的本质是什么？**

设计工具 = **生成一份"描述"，让开发者把描述变成产品**。

但现有工具（Figma/Sketch）的"描述"是**视觉像素**——一张静态图片。这带来三个根本性断裂：

| 断裂 | 原因 | 代价 |
|------|------|------|
| **交互断裂** | 像素无法表达"点击后跳到哪、状态怎么变" | 需要额外的连线图、交互说明文档、反复对齐 |
| **语义断裂** | 像素没有语义，一个矩形是 Button 还是 Card 不确定 | 设计稿转代码靠猜，需要人工翻译 |
| **状态断裂** | 像素是一个瞬间的快照，无法描述组件的多种状态 | 每个状态画一份，维护成本指数增长 |

## 2. 什么是正确的"描述"？

既然最终产物是**可运行的 UI 代码**（React/Vue/...），那最正确的描述应该**天然就是代码可以直接消费的结构化数据**。

**推导链条：**

```
设计的目的 → 最终产出可运行的 UI
         → 描述应该直接就是 UI 的结构化表达
         → 这个表达必须包含：组件树 + 样式(CSS) + 交互(事件/跳转) + 状态
         → 这就是一个 UI Schema 协议
         → 渲染引擎 consume 这个协议 → 就是最终产品
         → 设计过程 = 通过操作修改这个 Schema
         → AI = 用自然语言调用这些操作
```

## 3. 核心洞察

> **设计即产物：设计过程不是"画图"，而是"构建 Schema"。Schema 本身就是最终产品的源数据。**

这意味着：
- 不需要"设计稿转代码"这一步——Schema 直接就能渲染成代码
- AI 不需要"生成图片"——它调用结构化操作来修改 Schema
- 交互逻辑天然包含在 Schema 中——不需要额外说明

## Q&A：关键子问题的推导

### Q1：画布应该用 Canvas 渲染还是 React DOM 渲染？

画布同时承担两个完全不同的职责：

| 职责 | 本质 | 最佳技术 |
|------|------|---------|
| **A. 内容预览** — 展示设计结果 | "所见即所得"，用户看到的应该就是最终产品 | React DOM |
| **B. 编辑交互** — 选区、拖拽、缩放、对齐线 | 像素级精确控制的几何操作 | Canvas |

**推导：**
- 纯 Canvas 渲染内容 = 自己实现文本排版、Flex 布局、CSS 属性映射 = 重写浏览器
- 纯 React DOM = 内容完美，但选区框、拖拽手柄等编辑器 UI 用 DOM 很别扭
- Schema 样式就是 CSS → 浏览器天然渲染 → DOM 渲染内容零翻译成本

**结论：双层架构**
```
┌─────────────────────────────┐
│  Canvas 覆盖层（透明）        │ ← 选区框、拖拽手柄、对齐线、hover 高亮
├─────────────────────────────┤
│  React DOM 渲染层            │ ← Schema → 真实 React 组件树，CSS 1:1 对应
└─────────────────────────────┘
```

> **核心原则：内容层用产品的真实技术渲染（DOM），编辑层用最适合几何操作的技术（Canvas）。**

### Q2：Schema 能否转换成 Flutter、React、Vue 等任意前端代码？

**什么数据才能被"万能翻译"？** 语义足够、抽象正确的中间表示（IR）。

类比：LLVM IR 能编译到 x86/ARM/WASM，因为它描述的是计算的本质，不是某平台的细节。

Schema 的样式层分为两层：
```
┌──────────────────────────────────────────────┐
│  语义层（layout intent）                      │
│  flexDirection, gap, padding, alignment...    │ ← 所有平台都有等价概念
├──────────────────────────────────────────────┤
│  视觉层（visual properties）                  │
│  backgroundColor, borderRadius, fontSize...   │ ← CSS 属性名作为通用语言
└──────────────────────────────────────────────┘
```

**结论：完全可以。** CSS 属性集合是所有 UI 框架的"最大公约数"。通过插件化 Codegen 管道翻译到任意平台。

→ 详细设计见 `03-tech/design-codegen.md`

### Q3：屏幕选择的本质是什么？

选择屏幕 ≠ 锁死屏幕。**屏幕选择 = 选择初始视口（Viewport）**。

```
Schema 不变 → 切换视口只是改变观看窗口 → 验证适配效果
```

- 初始选择：决定默认画布宽高
- 后续切换：只改 viewport，Schema 不变
- 未来可扩展：响应式断点规则

### Q4：元素添加应该分几个层次？

Figma 的"矩形/椭圆/文字"太底层没语义；只有"Button/Card"又不够灵活。

**结论：双层体系**
- **原子元素**：div、button、img 等 HTML 标签，完全样式自由度
- **组件资产**：之前设计好的 Schema 片段，可复用

→ 详细设计见 `02-product/component-assets.md`

### Q5：操作层应该做成 MCP Server 还是内置 SDK？

**AI 调用操作的本质 = 外部智能体通过标准协议控制系统**

MCP 更好的 4 个原因：
1. **关注点分离**：核心竞争力是 Schema+渲染+操作，不是"AI 对话 UI"
2. **即插即用**：Cursor/Claude Code 零成本接入
3. **前期快后期不受限**：MCP Server 是不变的基座，Client 可以随时换
4. **生态开放**：任何 MCP Client 都能操控设计系统

**结论：** design-operations SDK 仍保留（手动操作+核心逻辑），上面加一层 MCP Server 壳。

→ 详细设计见 `03-tech/design-mcp.md`

### Q6：大型 Schema 如何存储与版本管理？

**系统天然就是 Operation-based 的：**
```
design-operations 的操作 = 编辑器的命令 = MCP 的 Tool Call = 版本管理的事件日志
                           四者合一！
```

**结论：Event Sourcing + 周期快照。** 不需要 CRDT/OT。

| 好处 | 说明 |
|------|------|
| 增量存储 | 每次只存一条 Operation（~200 bytes） |
| 精确版本 | 可回溯到任意一次操作 |
| 天然 Undo/Redo | 操作日志正反向执行 |
| 天然审计 | 谁在什么时候做了什么，AI 操作也记录 |

→ 详细设计见 `03-tech/event-sourcing.md`

### Q7：为什么不能“每个操作都同步等待保存成功”？

如果把“编辑反馈”绑定到“网络持久化完成”，就会把不稳定网络延迟直接暴露给用户手感，违背设计工具的核心要求：**连续操作必须即时响应**。

**推导：**
- 用户编辑的直接目标是“看到画布变化”，不是“等待数据库 ACK”
- Schema 的即时真相在前端执行器内存里，后端是持久化与协作分发层
- 因此持久化应当异步化、批量化、可重试，而不是逐操作阻塞

**结论：**
- 前端：本地先执行（立即可见）
- 后端：按时间窗批量写入 Operation Log（失败重试）
- 协作：通过 ws 增量广播，客户端去重回声

这保证了“编辑流畅性”和“数据可靠性”同时成立。

### Q8：拖拽定位为什么存局部坐标而不是屏幕坐标？

拖拽的视觉语义是“放进某个父容器中的某个位置”，不是“屏幕绝对像素点”。

如果存屏幕坐标，会在视口缩放、父容器移动、响应式适配时产生语义漂移；存父容器局部坐标可保持结构稳定，且天然可导出为 CSS `left/top`。

**结论：**
- 拖拽落点转成父容器局部坐标
- Schema 持久化的是容器内定位信息
- 导出代码时保持可解释、可还原
