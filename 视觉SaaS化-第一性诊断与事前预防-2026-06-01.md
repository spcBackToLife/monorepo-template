# 视觉 SaaS 化 — 第一性诊断与事前预防方案

> 写作日期：2026-06-01
> 触发：校园社交登录页项目（d84c140e-0437-4c80-a786-c1f389bcbb02）走完 4 阶段 SKILL 流水线、产出 70+ 文档、schema 全字段对账通过，**最终视觉=纯蓝色 SaaS 登录页**，看不出任何"校园风"
> 范围：SKILL（提示词层）+ MCP（接口层）+ features/design-engine（渲染层）+ apps/design-mcp / design-api（服务层）+ analysis-notes 流水线产物
> ⚠️ 本文不写"事后修字段"——上一份 `视觉简陋诊断报告-2026-06-01.md` 已作废
> ⚠️ 本文不加"出场门禁 / integrity checker / expectedArtifacts 红线"——那些是事后防御
> 本文只回答两个问题：**为什么这套流水线必然产出 SaaS 风？怎么从源头让它必然产出校园风？**

---

## 0. 残酷事实

```
              用户期望              ↔              实际产出
┌──────────────────────────┐   ↔   ┌──────────────────────────┐
│ 校园社交登录页              │   ↔   │ 蓝色 SaaS 登录页           │
│ 95-00 后大学生氛围         │   ↔   │ Linear / Notion 既视感    │
│ "清晨教室的光"             │   ↔   │ 一片几乎纯白               │
│ "暖白米 / 大圆角柔和"      │   ↔   │ #FCFCFD（与 #FFF 差 0.4%）│
│ "校园清晨日常感"           │   ↔   │ 12% 看不见的蓝紫光斑       │
└──────────────────────────┘       └──────────────────────────┘
                          ↑
              而 schema 字段全部对账通过
              70+ 文档全部 done
              5 阶段 SKILL 流水线全部出场
```

**不是哪个字段错了。是这套流水线无论拿到什么 product description，最后都会收敛到训练分布的均值——也就是 SaaS 风。**

## 1. 第一性根因 8 条

每一条都是"机制本身的设计错误"，不是"AI 这次没好好做"。换一个 AI 来跑，结果一样。

### 根因 1：AI 没有视觉感知，整套流水线靠文字推理 → 温度感只活在 md 里

**机制现状**：
- visualConcept.soulSentence = `"像清晨教室的光，温暖但不打扰"` ← 文字
- visualStrategy.color.ratio = `{60: 暖白米, 30: 卡片+主文字, 10: 主色}` ← 文字
- styles.backgroundColor = `"$token:colors.background"` ← 文字（解析后还是文字 #FCFCFD）
- self-review = AI 给自己打 5 维分 ← 文字

**第一性问题**：
> AI 写文档时给自己建立了"温度感"的幻觉，因为文字层面满足了温度的所有要件（mood board / 灵魂句 / 60-30-10 / 装饰节制）。但**这种温度感从未被翻译成像素**——AI 没看到自己最终画了什么。等到 self-review 截图出现时，11 个 styles/states/materials 任务都 done，没人愿意回头改。

**为什么必然产出 SaaS 风**：
> AI 在 pixel 层面没有"温度感"概念，它默认走训练分布——训练数据里"登录页"95% 是 SaaS 模板（Stripe/Linear/Notion 那一类），所以**当文字描述与训练 prior 冲突时，pixel 输出永远向 prior 收敛**。"克制"是 prior 高质量信号，"#FCFCFD" 是 prior 的"暖白"翻译，"12% alpha 光斑"是 prior 的"装饰节制"翻译——**全是 prior 的"安全选择"**。

**证据**：
- 视觉概念三选 A/B/C 中，AI 选 A（克制温度风）否决 C（校园活力）。否决理由："登录页活泼降低安全可信度"——这是 SaaS 设计师的世界观，不是大学生用户的世界观。SaaS prior 在替 AI 拒绝候选 C。
- visualConcept.moodBoard = ["晨光教室窗格","木桌笔记本","操场跑道弧线","公告板便签"] —— **这 4 个场景词在 schema 中没有任何节点引用，没有任何素材翻译**。它们是 AI 写给自己看的安慰剂。

---

### 根因 2：目标用户被压成抽象标签，缺少"视觉文化基因"载荷

**机制现状**：
- product-analyst 写 `targetUser.summary = "95-00 后大学生"`
- theme-generator 读到这个 summary，按 `methodology/01-intent-extraction.md` 的 8 个 aesthetics 标签（minimal / glassmorphism / luxury / playful / cyberpunk / natural / bold / pastel）选 1-3 个
- 选了 `minimal+flat`，seedColors=#5B6CFF

**第一性问题**：
> "95-00 后大学生" 在 AI 训练分布里是什么？是 K12 教育产品（极简）+ 校园 OA（务实）+ 大学生求职（SaaS 简约）的语料并集。**不是小红书/B 站/即刻/Soul/Tap/Vsco 这种"95-00 实际每天在用的产品"的视觉文化**。
>
> AI 看到 "95-00 大学生" 调用的是教育/求职/SaaS prior，因为这些 prior 在训练时被打上"专业、安全、可信"的高质量标签——而真实大学生用的产品视觉语言（多色/插画/贴纸/手写/破格排版）在 prior 里是"低质量"或"难维护"的负面标签。

**为什么必然产出 SaaS 风**：
> targetUser 字段的语义是"用户是谁"，不是"用户的视觉文化基因是什么"。前者会调用 SaaS prior，后者才能调用真实的视觉文化样本。`product-analyst` 的提示词从未要求 AI 写"参考产品视觉案例"——所以这个关键载荷在流水线起点就丢了。

**证据**：
- `methodology/01-positioning.md` §3 目标用户字段建议："年龄段 / 职业身份 / 主要使用场景 / 核心痛点 / 现有替代方案"——**没有"视觉文化对照"项**
- `methodology/01-intent-extraction.md` 的 aesthetics 8 类全集：glassmorphism / minimal / luxury / playful / cyberpunk / natural / bold / pastel——**这是 SaaS 设计师的术语表，不是大学生用户的视觉词汇**
- theme.references 字段（参考品牌）默认值 `[]`——可空，AI 默认空

### 根因 3：流水线"视觉质量"的衡量标准 = AI 自评，AI 看不见 → 必然胡评

**机制现状**：
- design-planner 每个 craft/styles/states 任务的 Step 6.5 强制 self-review 5 维评分
- methodology/13-self-review-rubric.md：识别度 / 层次 / 状态 / 主题契合 / 情绪传达 各 0-5 分
- expectedArtifacts: `selfReviewAllPassed minScore: 4` ← service 端机器对账"评分≥4"
- service 端**只验"评分字段非空且数值≥4"**，不验"评分是否真实"

**第一性问题**：
> AI 不能自己看到自己写的截图（Claude/GPT 没有持续视觉），但 SKILL 让 AI 给自己的视觉打分。这是个**逻辑闭环**：
>
>   AI 不知道画得好不好 → 但必须给评分（任务才能 done）→ AI 编一个 ≥4 的分数 → service 接受 → 任务 done

**为什么必然产出 SaaS 风**：
> 当 AI 的"通过门槛"=自填字段，AI 会为了"通过"而填看起来合理的字段——"识别度 4/5：主 CTA 一眼可见 ✓"。这种"评分"不带任何客观锚点。**结果是任意视觉都能通过 self-review**。

**证据**：
- 实际 `analysis-notes/d84c140e/design/00-login/self-review.md` 行 11：
  > "ISSUE-1：mcp/generate_snapshots 服务端持续返回与 schema 无关的简化登录页（重试 3 次，每次都是 5 元素「登录/邮箱/密码/...」）—— 这是 snapshot 服务管线 bug，不影响 design schema 落库的真实性。降级方案：用户从 editor 直接刷新提供截图，design 阶段按 13-self-review-rubric.md 5 维度评分"
- 文件第 32-37 行 5 维评分模板**全是 "__/5"** —— 没填，因为没截图
- task 状态 `D-00-login-self-review` 即便如此仍**完成了**——因为 AI 把"等待用户截图"当成"任务完成的合理状态"

**这就是流水线最后一公里的崩溃**：snapshot 服务有 bug → AI 主动降级 → AI 自评等于不评 → 视觉质量门槛形同虚设。

### 根因 4：MCP 粒度 = 单字段写入，AI 在写 schema 时**从未"画一幅画"**

**机制现状**：
- design-planner Phase 1 挂 11+ 屏级任务（briefing / concept / strategy / task-planning / emotion / hierarchy / budget / decorations / styles / states / materials / self-review / meta / tree-redlines / coverage / integrity）+ 9 项目级任务
- 每个任务负责一类字段（styles 任务 → batch_update styles；states 任务 → 48 个 visualState）
- MCP 工具：`element/add` / `style/update` / `style/batch_update` / `visual_state/add` —— **全是字段级单点操作**

**第一性问题**：
> 真正的设计师工作流：先打草图 → 看整体 → 调主角 → 调配角 → 调装饰 → 反复迭代。AI 工作流被 SKILL 设计成：
>
>   先写 briefing.md → 写 concept.md → 写 strategy.md → 拆 task → 跑 emotion 任务 → 跑 hierarchy 任务 → 跑 budget 任务 → 跑 styles 任务（11 节点 batch_update）→ 跑 states 任务（48 visualState）→ 跑 materials 任务 → ... → 跑 self-review 任务（这时第一次截图）
>
> **AI 跑前面 10 个任务时，整屏视觉从未在它脑海/上下文里成型过**。AI 在"挨个填表格"，11 张表格全部填完才能"看一眼"——而这一眼如果发现整体不对，回去改要走 fix 任务（流程成本极高），所以 AI 倾向于"算了 4/5 给个面子通过"。

**为什么必然产出 SaaS 风**：
> 任务=填字段≠创作。AI 没有"先画草图反复调整"的最小循环——只有"先想好 11 个字段填完"的瀑布流程。瀑布流程的特征：起点偏一点点，终点差很多。起点（concept "克制温度"）偏向 SaaS prior，瀑布走完无人纠偏。

**证据**：
- design-planner SKILL.md 行 305-330（Phase 2 执行流程）：明确要求 read_file→md→schema→update_plan_task→下一任务，**Step 6.5 自审是任务结束才做**
- styles.md 写了 33 节点全量 styles 一口气 batch_update。完成时 schema 已经定型，没有"草图—调整—再草图"循环
- task-planning.md 列的 5 个 craft 任务（brandlogo / decoration-rebalance / tab-indicator / checkbox / typography-refresh）—— **全是单点 craft，不是"整屏校园感达标"这种作品级任务**

### 根因 5：素材是"可选附件"不是"概念翻译产物"

**机制现状**：
- design-planner 创作权 6："素材绘制权——自调 material-painter 子技能画素材"
- methodology/06-decoration.md §2：装饰密度档位 = 极少 / 节制 / 中等 / 丰富，登录页归"极少 0-1 / 节制 1-2"
- C 端每屏至少图标/装饰之一，"特殊场景需逐条论证"
- materials 任务可 skipped（"全 CSS 无素材需求时论证后跳过"）

**第一性问题**：
> visualConcept.moodBoard = ["晨光教室窗格", ...4 项] 是 visualConcept 的核心字段，但这 4 项**只读不写**——AI 写完 mood board 就完了，没有任何机制要求"每个 mood word 必须有 ≥1 个翻译节点"。
>
> 当 SKILL 把素材定义成"可选附件"，AI 的省力路径就是：跑 self-review 时说"为节制起见装饰仅 css-gradient blob，符合 minimal 主题"——理由完美，预算合规，concept 字段"已落库"。**整张 mood board 在最终视觉里完全不存在**。

**为什么必然产出 SaaS 风**：
> 素材是把"校园温度"从文字翻译到像素的**唯一通道**。当这个通道被设计成可选/可跳过/默认极少，AI 的训练 prior（节制=高质量）会让它倾向跳过。结果 mood board 永远是死字段，concept 永远是文字描述，整屏永远没有具象视觉信号——**所有"校园特征"都被节制吃掉了**。

**证据**：
- 该项目 4 个 materialSpec：BrandLogo / LockedIcon / SubmitSpinner / BgBlobTopRight
  - BrandLogo 真画了（type=img 强迫式必须有）
  - LockedIcon 没画（kind=icon, renderHint=svg, 因为 LockedView 默认隐藏 → AI 主观判定"不重要"）
  - SubmitSpinner 是纯 CSS（renderHint=css-only）
  - BgBlobTopRight 是纯 CSS（renderHint=css-gradient）
- mood board 4 词 ["晨光教室窗格","木桌笔记本","操场跑道弧线","公告板便签"] **没有任何 schema 节点引用，0 个素材**
- material-painter SKILL 在该项目仅被调用 1 次（画 BrandLogo），其他 0 次
- design-planner 的 Phase D task-planning 没有"扫 mood board → 自创翻译任务"的强制动作

### 根因 6：SKILL 之间是单向接力，theme 决策一旦落地 design 就视为"色板事实"

**机制现状**：
- 流水线：product → theme → interaction → design → executor，单向
- 跨阶段反向修订只能走 UpstreamChallenge（高仪式：写 challenge md / ≥3 候选 / 影响面声明 / decision md / resolve）
- 该项目 challenges 目录只有 `C-INT-00-login-001`（interaction 内部结构补充，不是跨阶段修订）

**第一性问题**：
> theme-generator T1-intent.md 选了 `aesthetics: [minimal, flat]`、`decoration: minimal`、`seedColors: [#5B6CFF]`。这套决策是**完全自洽**的——SaaS 蓝紫单色 + 极简极简极简——**单看 theme 阶段挑不出毛病**。
>
> 等流水线流到 design 阶段，AI 看到的是"已确定的 token 池"+"主题已 customized=true 出场门禁全过"。design 的入场门禁第 ③ 条："theme.customized=true 且 theme/validate 0 error"——**通过表示 theme 是真理**。
>
> design 阶段不会问"#5B6CFF 真的代表大学生吗？"——这是 theme 的事，theme 已通过。**没有任何 SKILL 在出场时被要求"过下游用户视角的预审"**。

**为什么必然产出 SaaS 风**：
> 单向接力让"上游产物 = 下游真理"。任何上游的 prior 偏移（如 theme 选了 SaaS 蓝紫）会原封不动传到下游。下游想纠偏只能 UpstreamChallenge——成本高、仪式重、AI 倾向不发起。

**证据**：
- 该项目 0 条跨阶段 design→theme 的 UpstreamChallenge
- design 阶段 emotion.md 把 user mood 写成"克制温度风"，concept.md 把候选浓缩成"校园清晨/公文专业/校园活力"——**都在 theme 给的色板（蓝紫单色 + 暖白底）上做选择**。这套色板本身就把候选 C "校园活力" 挤死了：纯蓝紫单色色板**画不出活泼**。
- theme-generator T7-handover.md 出场只跑 `theme/validate`（R-THEME-01~10），**0 条规则验证"色板与 targetUser 的视觉文化是否匹配"**——因为 targetUser 字段里也没载荷视觉文化（根因 2）

### 根因 7：渲染层与 SKILL 教法没有契约 → 谣言被写进方法论

**机制现状**：
- features/design-engine/src/styles/resolveTokens.ts：`value.includes(' ') && value.includes(TOKEN_PREFIX)` 走 split-by-space + 单段 startsWith 替换 → 实测**支持**字符串内嵌 token（如 `"radial-gradient(... $token:colors.primary 0%, ...)"`）
- design-planner methodology/06-decoration.md §9 红线："css-gradient 装饰节点 styles 不用字符串内嵌 token（拆字段 + currentColor）"
- 该项目 craft-decoration-rebalance.md §2："v2 BgBlobTopRight `background: 'radial-gradient(... $token:... )'` —— 字符串内嵌 token 不被渲染层解析（methodology/06 §7 + 12 §6 已标），所以渲染端把整个字符串当无效值，导致装饰完全没出现"

**第一性问题**：
> SKILL methodology 是"AI 写 schema 时的真理"。但 methodology 里的"渲染层不支持 X" 这种判断从哪来？**靠 SKILL 作者凭印象写**。一旦写错，下游所有 AI 永远凭这个错误判断行事。
>
> 该项目 design-planner v3 craft 任务 craft-decoration-rebalance.md **基于错误信仰** "字符串内嵌不解析"，把 BgBlobTopRight 改成 `rgba(91,108,255,0.12)` 硬编码——理由完美（trade-off 接受 R-STRUCTURE-02），结果**没修原本 token 形式（其实能渲染）+ 引入了新硬编码**。

**为什么必然产出 SaaS 风**：
> 这条根因不直接产出 SaaS 风，但它放大了所有其他根因——SKILL 凭印象写规则 → AI 凭印象做决策 → 渲染层实际能力闲置或被错误绕过 → 设计师工具链效率打折。

**证据**：
- features/design-engine 行 51-55 实现了空格分隔的内嵌 token 解析
- methodology/06-decoration.md 行 91 + 红线 §9 都说"不支持内嵌"
- 该项目 craft-decoration-rebalance.md 把内嵌 token 当根因，引入 rgba 硬编码作为 trade-off

### 根因 8（最毒）：SKILL 提示词的奖励函数错了 — 鼓励"跑完流程"，不鼓励"作品好"

**机制现状**：
- design-planner SKILL.md 行 21："**核心信念**：视觉决定结构、视觉决定装饰、视觉决定素材。**视觉先行 = 绝对红线**"
- design-planner SKILL.md 第 6 行："视觉创作者，不是字段填写员"

**但 SKILL.md 全文 881 行的实际内容分布**：
- 任务驱动 / plan / expectedArtifacts / 红线 / integrity 校验 / 出场门禁 ≈ 600 行
- 创作方法论引用（methodology/00 ~ 13）≈ 100 行
- 真正"教 AI 怎么把校园温度画出来"的内容 ≈ 0 行

**第一性问题**：
> SKILL 在第一段反复声明"AI 是创作者"，但全文 95% 在讲流程合规、字段对账、红线避免。AI 内化的奖励函数是：
>
>   做完所有强制项 = 任务通过 = 奖励
>
> 没有"画得好不好"的奖励，只有"流程跑没跑完"的奖励。当 AI 的优化目标是"跑完流程"，它会找最省力的合规路径——这条路径恰好是 SaaS prior。
>
> 训练 SKILL 的人当时在防"AI 偷懒不跑流程"，结果矫枉过正变成"AI 只跑流程"。

**为什么必然产出 SaaS 风**：
> AI 是奖励函数的优化器。当奖励函数=合规，AI 输出=合规的最低成本路径=训练 prior 均值=SaaS。这是**根因之根因**——前 7 条都是"机制设计的结果"，第 8 条是"机制设计的原因"。

**证据**：
- design-planner SKILL.md 行 393-433：第 5 节"关键红线"含 6 个红线小节，列了几十条"❌"
- 同一文件没有任何"如何让画面真的有校园感"的具体方法论
- methodology/13-self-review-rubric.md 评分维度都是抽象（识别度/层次/状态/契合/情绪），**没有"画面是否真有校园特征 / 是否真像 95-00 后会用的产品"这种产品级判据**
- 在 v3 升级文档（视觉设计技能升级方案-v3-从对账员到创作者.md）里，作者意识到这个问题，但 v3 落地后流程仪式反而更重——增加了 briefing/concept/strategy/task-planning/self-review 5 个 Phase，本意是"让 AI 像设计师"，结果变成"让 AI 像更细致的设计字段填写员"

## 3. 事前预防方案 — 8 个机制重构（不是事后 checker）

### 重构 1：把"视觉锚点"作为流水线的物理输入，不是文字输入

**问题**：现状用户给文字 prompt（"做个校园社交登录页"），AI 用文字推理生成视觉。

**重构**：

#### 1.1 product-analyst 入场强制收集"视觉参考"

```jsonc
// product-analyst Phase 0：入场动作改造
1. 用户描述需求
2. 【新增】问用户："你心里这个产品长得像哪几个产品？给 2-5 个真实产品名"
   - 候选触发词："像 X 但更 Y" / "类似 Z" / "参考过 A"
   - 用户答："像即刻但更校园" / "类似小红书的创作者中心"
3. 【新增】调 web-search/web-fetch 拉取这些产品的截图（首页/登录页/列表页 各 1）
4. 【新增】把截图存到 project.assets.visualReferences[]
   {
     productName: "即刻",
     screens: [{ purpose: "登录", url: "...", filePath: "assets/refs/jike-login.png" }],
     extractedTraits: { dominantColors: [...], decorationDensity: "中等", ... } // 由 vision 模型提取
   }
5. project.meta.visualReferences = [...]  ← 进入流水线下游 SKILL 的强制读取项
```

#### 1.2 theme-generator T2 强制读 visualReferences 推 seedColors

```
现状：T2 用 HSL 色轮 + APCA 验证从 seedColor 推完整色板
重构：T2 第一步先 image-color-extract 跑 visualReferences[].screens[*]
     → 提取 dominant + accent + neutral 实际色值
     → 这些色值是 seedColors 的强约束（不是凭 keyword "minimal" 抽 #5B6CFF）
```

#### 1.3 design-planner 每个 craft/styles 任务前必须读参考截图

```
现状：design 任务 read_file 读 methodology/recipes 等文档
重构：增加强制 read_image visualReferences[] —— 不是用文字描述，是真实图像作为上下文
     LLM 看图后写 styles 时会自然趋近参考的视觉气质
```

**为什么这是事前**：把视觉锚点变成物理输入后，AI 的 prior 被参考图像"压制"——参考图是即刻/小红书，AI 不会自然走向 Linear。

**实现层**：
- `apps/design-mcp` 新增 `visual_reference/upload` MCP action
- `features/design-engine` Schema 增加 `project.assets.visualReferences[]` 字段
- `.claude/skills/product-analyst/SKILL.md` Phase 0 改造（必收 ≥2 参考产品）
- `.claude/skills/theme-generator/SKILL.md` T2 改造（image-color-extract）
- `.claude/skills/design-planner/SKILL.md` 改造每个 craft 任务前 `read_image` 步骤

### 重构 3：把"视觉验证"从 AI 自评改为机器对账 + 修复 snapshot 服务

**问题**：snapshot 服务有 bug → AI 自评降级 → 视觉门槛形同虚设。

**重构**：

#### 3.1 P0 修复 generate_snapshots 服务（非 P0 不接受）

```
位置：apps/design-mcp/src/<snapshot 服务源码>
当前问题：服务端持续返回与 schema 无关的简化登录页
排查路径：
  - 检查 snapshot 渲染器是否正确加载 themeConfig
  - 检查是否使用了 PreviewRenderer 还是 SchemaRenderer
  - 检查 staticAssetOrigin 注入
  - 检查 viewport 模拟（headless browser 配置）
```

这是流水线最后一公里基础设施，**不修复其他重构都白搭**。这是真正的根因 0。

#### 3.2 把 self-review 5 维评分改为机器对账

| 现 5 维（主观自评） | 改造后（机器对账） |
|---|---|
| 识别度 | OCR 截图 → 主 CTA 文本 + 字号占比是否 ≥ 阈值 |
| 层次 | 视觉权重计算（节点面积×对比度×饱和度）是否成金字塔 |
| 状态 | 切换 setVisualState 截图差异 ≥ N 像素差 |
| 主题契合 | dominantColors 直方图与 visualReferences 直方图距离 ≤ 阈值 |
| 情绪传达 | 使用 multimodal LLM 看 reference 图 + 实际截图，回答"风格相似度 0-10" |

#### 3.3 加 MCP `visual_diff_with_reference`

```
visual_diff_with_reference {
  projectId, screenId, referenceId
}
→ 返回：
  {
    colorHistogramDistance: 0.42,      // 0=完全一样
    decorationDensityRatio: 0.15,      // 实际/参考
    fontFaceFamilyDelta: ["actual sans-serif", "ref handwriting"],
    similarityScore: 0.31              // multimodal 综合
  }
```

**为什么这是事前**：当 self-review 是机器对账，AI 不能给自己打 4/5 蒙混过关。AI 会主动调整 styles 直到 colorHistogramDistance ≤ 阈值——这种调整自然趋近参考图的视觉气质。

**实现层**：
- P0 修复 `apps/design-mcp` snapshot 渲染管线
- 新增 `apps/design-mcp/src/visual-diff/` 模块（image-color-extract、SSIM、multimodal LLM 调用）
- design-planner SKILL 把 self-review 改为"调 visual_diff_with_reference + 自动修复循环"

### 重构 4：MCP 增加"作品级"操作 + 把任务粒度从"字段类别"改成"视觉信号达标"

**问题**：现状 MCP 都是单字段操作，AI 跑完 11 任务前看不到整屏。

**重构**：

#### 4.1 新增 MCP `preview_iteration`

```jsonc
preview_iteration {
  projectId, screenId,
  intentNote: "强化校园温度感"  // 自由文字描述
}
→ 服务端：
  1. 跑 generate_snapshots 拿当前截图（before）
  2. 等待 AI 调多个 style/update + element/add 等
  3. 跑 generate_snapshots 拿新截图（after）
  4. visual_diff_with_reference(reference, after) - visual_diff_with_reference(reference, before)
  5. 返回 { delta, stillFar, shouldContinue, suggestion }
```

这把"草图—调整—再草图"做成了原生 MCP 循环。

#### 4.2 design-planner 任务粒度改造

| 现 styles 任务（按字段类别）| 改造后（按视觉信号达标）|
|---|---|
| `D-X-styles` 一次性 batch_update 33 节点 | `D-X-signal-brand` BrandLogo + Slogan 视觉权重达标 |
|  | `D-X-signal-cta` SubmitBtn 主角识别度达标 |
|  | `D-X-signal-mood` mood board ≥1 词翻译为可见素材 |
|  | `D-X-signal-cultural` 与 visualReferences 色彩气质相似度 ≥ 0.6 |
| `D-X-states` 48 visualState | `D-X-signal-state-distinct` 每业务态截图与 default 像素差 ≥ N |

每个 signal 任务 = 一个最小可视化的视觉目标 + 必须看到截图达标才能 done。

**为什么这是事前**：把任务粒度切成"视觉信号"后，AI 的工作循环天然变成"画一笔→看一眼→再画"——这正是设计师的工作模式。

**实现层**：
- `apps/design-mcp` 新增 `preview_iteration` action
- `.claude/skills/design-planner/SKILL.md` Phase 1 任务清单重写
- 旧的 styles/states/materials 巨任务保留为"兜底"，不是默认路径

### 重构 5：mood board 是强制翻译产物，不是字符串数组

**问题**：moodBoard 4 词在 schema 落库后 0 翻译节点。

### 重构 8：SKILL 提示词奖励函数从"合规"改为"作品"（最关键）

**问题**：SKILL 全文 95% 在讲流程合规、字段对账，AI 内化为"做完=完成"。

**重构**：

#### 8.1 SKILL.md 结构重写：把"作品质量目标"放在最前 + 全程

  3. 双产出原则（md + schema）

  5. 关键红线                     ← 占 20% 篇幅

  2. 你不能做的事：把视觉相似度凭印象当成 "看起来差不多就行"

## 7. 关键不要做什么

为了避免再走偏，列一份**反清单**：

### ❌ 不要再加新的 expectedArtifacts 红线
- expectedArtifacts 是事后机器对账，无法补救事前的 prior 偏移
- 越多 expectedArtifacts，AI 越觉得"通过 = 作品好"，奖励函数错位越严重

### ❌ 不要再加更多 methodology 文档
- design-planner 已经有 17 个 methodology 文档
- 加文档不解决 AI 没视觉感知问题
- 与其加文档，不如让 AI 看图

### ❌ 不要再加更多 plan 任务
- design-planner 已经 11 屏级 + 9 项目级 + 4 全局 overlay 任务
- 加任务不解决"任务=填字段≠创作"问题
- 真正要做的是改任务粒度（按视觉信号），不是加任务数

### ❌ 不要再做"AI 自审"机制
- 任何"AI 给自己打分"机制必然失效——AI 看不见
- 自审只能是"调机器对账工具"，不能是"自己评 5 维"

### ❌ 不要再写视觉简陋根因分析-XX-XX.md 这类报告
- 该项目根目录已经有 5+ 份诊断报告（视觉简陋根因分析-2026-05-31.md / 视觉设计技能升级方案-v3-从对账员到创作者.md / 视觉设计渲染链路根治方案.md / ...）
- 写报告 ≠ 改机制
- 这一份不补改造代码 + 改 SKILL 提示词 + 改 MCP，下一份诊断又会写"为什么 v4 还是产出 SaaS"
