## 1. 第一性原理：专业视觉设计师真实工作流

### 1.1 视觉设计的本质

```
视觉设计 ≠ 给已有节点配 styles
视觉设计 = 用视觉语言把「产品意图 + 主题语言 + 交互骨架」重新讲一遍故事，
          让用户在 0.5 秒内识别 + 共鸣
```

讲故事需要**创作权**——不只是填字段权。当前 SKILL 把 design-planner 限定为"严禁动节点结构、严禁画素材、严禁自创任务"——剥离了所有创作权，只剩"按既定字段填值"，所以输出物一定是简陋的。

### 1.2 设计师工作的 8 阶段（从纸到屏）

```
阶段                   设计师在做什么                              产物
────────────────────────────────────────────────────────────────────
A. 取景（Briefing）   读产品 / 主题 / 交互骨架 / 用户场景        理解纪要
B. 概念（Concept）    脑子里浮现 N 种风格方向，选 1 推进           mood board + 灵魂句 + 风格关键词 3
C. 策略（Strategy）   定调色配比 / 字号节奏 / 形状语言 /           visualStrategy 5 维：
                      装饰系统 / 间距韵律 / 动效语言               色 / 字 / 形 / 饰 / 律
D. 拆任务（Plan）     基于策略 + 本屏特征，自创出 N 个视觉任务      n 条自创 plan tasks（不是模板硬列）
E. 创作（Craft）      每个任务：可能改布局 / 加装饰 / 画素材 /     节点 add/move/wrap、装饰节点、
                      调 styles + visualStates                     素材 PNG/SVG、styles 全量
F. 自审（Review）     截图自看 + 5 维度评分（识别/平衡/一致/契合/  审稿评分 + 重做记录
                      情绪），不达标重做
G. 一致性 audit       跨屏 + 跨组件统一性核查                      audit 报告
H. 出场               交付完整 schema + 截图 + 设计 token 用法     handover
────────────────────────────────────────────────────────────────────
```

**当前 SKILL 实际只做 D 局部 + E 局部**（按 Phase 1 写死的 11 任务"扫平"填字段）。**A/B/C/F 完全缺失**，**G/H 是 schema 字段层而非画面层**。

### 1.3 当前 SKILL 与真实工作流差距对照

| 阶段 | 真实设计师做的 | 当前 SKILL | 差距严重程度 |
|---|---|---|:---:|
| A 取景 | 读产品调性 / 主题 intent / 交互骨架 / 用户情境 | Phase 0 入场门禁只看"上游 phase 标对没"；不读产品/主题语义 | ⚠️ 中 |
| B 概念 | mood board → 风格关键词 3 个 → 灵魂句 | 完全无 ❌ | ★ 致命 |
| C 策略 | 5 维（色/字/形/饰/律）+ 60-30-10 + 装饰单一族 | methodology/04 装饰分类（一维），无策略统筹 | ★ 致命 |
| D 拆任务 | 基于策略自创 n 个任务（如"做 hero 渐变背景"） | Phase 1 写死 11 任务模板 ❌ | ★ 致命 |
| E 创作 | 改布局 + 加装饰 + 画素材 + 调 styles | §5.4 严禁动结构、素材踢给 executor | ★ 致命 |
| F 自审 | 截图 + 5 维度评分 + 重做 | 完全无 ❌ | ★ 致命 |
| G audit | 跨屏一致 + 装饰系统统一 | 现有 D-audit 任务但只对账字段 | ⚠️ 中 |
| H 出场 | 完整画面 + 素材 + token 用法 | 出场门禁全是 schema 字段非空 | ⚠️ 中 |

5 个 ★ 致命缺口正对应用户的 Q1-Q7。

## 2. 升级方案总览：把 7 项能力补齐

### 2.1 角色重新定义

```diff
SKILL.md §1 角色定位：

- 资深企业级 UI/视觉设计师。你不是写 CSS 的，你是给整个产品定调、定层级、定氛围、定品牌的设计师。
+ 资深企业级 UI/视觉设计师。**视觉创作者，不是字段填写员**。
+
+ 拥有六项创作权：
+ 1. 视觉概念决策权 —— 为每屏定 mood / 灵魂句 / 3 个风格关键词
+ 2. 视觉策略制定权 —— 在 token 上定 60-30-10 调色 / 字号节奏 / 形状语言 / 装饰系统 / 间距律 / 动效律
+ 3. 视觉任务自创权 —— 基于策略与本屏特征自创 plan task，不局限于模板
+ 4. 布局调整权 —— add/move/wrap 视觉容器节点（不动业务节点的事件/绑定/数据）
+ 5. 装饰节点新建权 —— 4 类装饰节点 + 单一装饰系统（光斑系/几何系/插画系/纹理系任选其一）
+ 6. 素材绘制权 —— 自调 material-painter 画完素材 + applyMaterialDesign 应用，给 executor 留交付物而非规格
+
+ 出场契约：交付**完整可识别的画面**（截图 5 维度评分 ≥4/5），不是字段非空。
```

### 2.2 工作流重构：8 个 Phase，对应专业设计师 8 阶段

```
Phase A — 取景（Briefing）            ⭐ 新增
Phase B — 视觉概念（Concept）         ⭐ 新增
Phase C — 视觉策略（Strategy）        ⭐ 新增（替代当前 D-X-emotion/hierarchy/budget 散点）
Phase D — 任务自创（Plan）            ⭐ 升级（当前是 SKILL 写死 11 任务，改为基于 C 自创）
Phase E — 创作执行（Craft）           ⭐ 大改（含布局调整、装饰新建、素材绘制、styles、states）
Phase F — 自审重做（Review）          ⭐ 新增
Phase G — 跨屏 audit                  保留升级
Phase H — 出场                        保留升级
```

每个 Phase 在 SKILL.md 中是 ≤ 200 字的"做什么 + 何时进入下一 Phase"，详细方法在 references。

## 3. 执行机制保留与强化（循序渐进 / 任务驱动 / 过程留痕 / 上下文聚焦）

> 这是 v3 的**不动条款 + 强化条款**。视觉设计是大工程，必须循序渐进——顶层分析 → 规划方向 → 拆解任务 → 逐个细化。**当前 SKILL 已经设计了这套机制**；v3 一行不动，只把新内容（创作权 / methodology / recipes / 自创任务）叠加进入这套机制的对应位置。

### 3.1 现有机制盘点（一行不动，全部保留）

当前 SKILL 已实现"大型设计项目循序渐进"的标准做法，盘点如下——v3 一律保留：

| 机制 | 落点 | 解决什么 |
|---|---|---|
| **Phase 0 入场门禁 5 查** | SKILL §4 Phase 0 | 上游未完成则不得开始；新会话续接通过这一步定位状态 |
| **Phase 1 首次挂 plan tasks** | SKILL §4 Phase 1 | 把整屏 + 项目级 + 全局 overlay 工作拆成 ~21 个小任务，每任务带 expectedArtifacts |
| **Phase 2 每轮一个最小任务循环（雷打不动 8 步）** | SKILL §4 Phase 2 | 一次只聚焦一个任务；token 不背历史包袱；写完一个落库一个 |
| **每任务一份 md（过程留痕）** | analysis-notes/<projectId>/design/.../*.md | 推理过程可见、可审、可复用；多会话续接看 md 不看历史聊天 |
| **md ↔ schema 双产出 + 末段「★ 沉淀到 schema 的结论」** | SKILL §3.4 | md 末尾与 MCP 调用 1:1 对应；既保留过程又保证落库一致性 |
| **expectedArtifacts 机器对账** | service 端 update_plan_task done 校验 | "看似 done 但产物不齐"会被拒；防漂移 |
| **plan 状态机 (pending → doing → done/skipped/blocked)** | meta/update_plan_task | 任务有显式生命周期；blocked 任务支持回流 |
| **schema 自身就是状态（§10 续接）** | 无外部 progress.json | 任意会话进来 query/next_pending_task 直接续做；token 焦虑大幅缓解 |
| **强制 read_file 触发加载（§4.X 任务→必读文件映射）** | SKILL §4.X | 每任务执行 Step 3 时按映射加载本任务所需文件，其他文件不入 context |
| **跨阶段回流 UpstreamChallenge** | SKILL §5.6 + §11 | 发现上游瑕疵不当场补，挂 challenge 让上游重做；保留阶段边界纯净 |
| **跨任务模板复用 + schema-spec 每次重读** | SKILL §4.X 末段 | 连续做多个同类任务模板可复用，spec 必重读防拼写错 |

**这 11 项机制 v3 一行不改**。下面 §3.2/§3.3 说明 v3 新内容如何叠加进入这套机制。

#### 叠加点 C：expectedArtifacts 加新机器对账条目

每个新任务定义自己的 expectedArtifacts，沿用现有机制。例：

```jsonc

    { kind: "compositionApplied", recipe: "checkbox", participants: [...] },    // B2 工具
    { kind: "recordInMd", section: "★ 沉淀到 schema 的结论", taskId: "$" }
  ]
}
```

**service 端 update_plan_task done 时校验**——这是现有机制的扩展，不是新机制。

#### 叠加点 D：UpstreamChallenge 触发条件扩展

现有 §11 处理"interaction 缺字段、theme 缺 token"等。v3 新增触发：

```
+ 自创 D-X-craft-N 时发现：theme.tokens 缺关键 token（如缺 elevation-2 阴影、缺 successBg 色）→ challenge theme-generator 补
+ 自创任务时发现：interaction 没建对应衍生视图节点（如错误页 ErrorView）→ challenge interaction-designer
```

通过**现有 challenge 通道**回流，机制 0 改动。

### 3.3 v3 升级后 AI 每轮执行循环（与现有 8 步雷打不动流程一致）

```
（每轮一个最小任务，循环执行直到 next_pending_task 返回 null）

  ┌─ Step 1. query/next_pending_task → 任务 T
  │
  ├─ Step 2. update_plan_task { T, status:'doing' }
  │
  ├─ Step 3. ★ 强制 read_file（按 §3.2 叠加点 B 的扩展映射表）
  │           - 只读本任务对应的 template + methodology + schema-spec + recipes
  │           - 其他文件不入 context（progressive disclosure）
  │
  ├─ Step 4. query/screen_schema { projectId, screenId } → 看最新状态
  │
  ├─ Step 5. ★ 写 md（按对应 template）
  │           - 推理过程段：对照 methodology / recipes 论证本次设计选择
  │           - 末尾「★ 沉淀到 schema 的结论」段：与 Step 6 MCP 调用 1:1 对应
  │
  ├─ Step 6. ★ MCP 落 schema
  │           - styles / states / 装饰节点 / materialProjectId（v3 放开）/ 自创下游任务
  │           - 每行 styles 必须 $token: 引用
  │
  ├─ Step 6.5（v3 新增）. 如本任务是 D-X-craft-* 或 D-X-self-review：
  │           generate_snapshots → 取截图 → md 末尾加自审段（5 维评分）
  │           任一维 < 4 → 不进 Step 7，回 Step 5 重做
  │
  ├─ Step 7. update_plan_task { T, status:'done', expectedArtifacts:[...] }
  │           - service 端机器对账，不齐则 done 失败 → 回 Step 5 补
  │
  └─ Step 8. 简短回复（§7 格式）→ 进下一轮
```

**关键**：循环颗粒不变（一轮 = 一个最小任务），只在 Step 3 / Step 6 / Step 6.5 上**叠加内容**。token 焦虑控制因 progressive disclosure + schema 自身是状态而保持。

### 3.4 token 焦虑控制策略（继承现有 + v3 强化）

| 策略 | 现有 | v3 强化 |
|---|---|---|
| 一次只做一个最小任务 | ✅ Phase 2 已实现 | 保留 |
| 每任务一份 md（不一次性写完整屏） | ✅ analysis-notes 分文件 | 保留；新增 craft 任务一份 md，避免单个 styles.md 200 行膨胀 |
| 模板 / 方法论按任务触发加载 | ✅ §4.X 映射表 | 扩展映射表至新 methodology / recipes，加载粒度仍是"按需" |
| schema 自身是状态（续接零成本） | ✅ §10 | 保留 |
| schema-spec 每次落 schema 前重读 | ✅ §4.X 末段 | 保留 |
| 跨任务模板复用 | ✅ 内存中保留 | 保留 |
| **★ v3 新增：recipes 按"本任务实际用到的效果/控件"加载** | — | 做 D-X-craft-form 只读 recipes/compositions/{checkbox,input,tab}.md；不 read 其他 |
| **★ v3 新增：theme-element-dict 只读匹配 theme.intent 那一份** | — | 8 份字典只 read 1 份；其余不入 context |

**结果**：v3 单轮任务的 context 占用 ≈ 当前单轮任务的 context 占用。**不会因 references 体系扩到 50+ 文件就全部加载**。

### 3.6 一图说明：循序渐进的全过程

```
顶层目标：本屏视觉设计交付完整可识别画面（5 维度均 ≥4/5）
   │
   ├─ A 取景    1 任务 1 md │  briefing.md           （理解产品+主题+交互骨架）
   ├─ B 概念    1 任务 1 md │  concept.md            （mood + 灵魂句 + 关键词 3）
   ├─ C 策略    1 任务 1 md │  strategy.md           （色 30/字 30/形 20/饰 15/律 5）
   ├─ D 自创    1 任务 1 md │  task-planning.md      （基于 C 自创 N 个 craft）
   │              ↓ N 个独立小任务，每个独立 md，每个独立 expectedArtifacts
   ├─ E 创作  N 任务 N md  │  craft-hero.md / craft-form.md / craft-deco.md ...
   ├─ F 自审  M 任务 M md  │  review-X.md           （每个 craft 独立自审，不达标重做）
   │              ↓ 反复 E↔F 直到所有 craft + review 都过
   ├─ G audit 1-3 任务 1-3 md │ decoration-audit.md / color-ratio.md / weight-pyramid.md
   └─ H 出场  1 任务         │ handover            （integrity 0 错 + 全 self-review ≥4 + 移交 executor）

每个任务（A 到 H 所有节点）都共用同一套 8 步循环：
  next_pending_task → doing → read 模板/方法论/recipes → 写 md → 落 schema → [自审] → done
```

**循序渐进 = 大目标拆成 ~25 个小任务，每个小任务一轮聚焦完成、一份 md 留痕、机器对账后才能 done**。这套机制 v3 完全保留。

## 4. 升级如何回答用户的 7 问

每问对应：**SKILL 落点 / methodology / recipes / 机器对账**。

### Q1. AI 知道何时用素材装饰 / 用什么色 → ✅ 通过 Phase B+C 解答

#### 设计师真实做法
1. 先看本屏情绪（Phase B）—— "焦虑/期待/平静/兴奋"
2. 选风格 mood —— "温暖治愈/冷静专业/活泼俏皮"
3. 据 mood 推导**装饰用量**（一个 mood 配一个装饰用量等级：极简 0-1 处 / 节制 2-3 处 / 丰富 4-6 处）
4. 据 mood 推导**装饰系统**（光斑系 / 几何线条系 / 插画系 / 纹理系 / 有机曲线系——选 1）
5. 据 mood 在 token 池里挑 60-30-10 三色（主导/次要/强调）

#### 落点
- `methodology/02-visual-concept.md`：mood → 装饰用量等级映射表（决定何时用）
- `methodology/06-visual-strategy-decoration.md`：装饰系统单一族（决定用什么形态）
- `methodology/03-visual-strategy-color.md`：60-30-10 调色法 + token 池筛选（决定用什么色）
- `recipes/decoration-systems/*.md`：5 大装饰系统每族一份配方（含具体 CSS / 素材规格）
- SKILL Phase C 强制产出 `screen.meta.design.visualStrategy = { color: { primary, secondary, accent, ratios }, decoration: { system, density, instances } }`

#### 机器对账
- 装饰系统必须填一个枚举（不能不写、不能选多个）
- 60-30-10 比例必须基于实际 styles 自动统计（B2 工具）
- 装饰实例数必须 ∈ [density.min, density.max]

---

### Q2. AI 知道每个元素怎么契合主题 + 丰富呈现 → ✅ 通过主题→元素词典

#### 设计师真实做法
拿到主题"温暖治愈"，资深设计师立刻浮现：
- button：圆角大、边角软、主色填充 + 极淡渐变、按下态有 spring
- input：边框 1.5px、focus 时微光晕、字号大避免压迫
- card：大圆角、内部留白多、阴影超柔（rgba alpha 0.04）
- empty：用插画 or 大图标 + 暖色调
- error：不用纯红，用珊瑚橘（warm error）

而拿到"科技专业"主题：
- button：直角或微圆角、单色、按下态硬切换无 spring
- input：边框直角、focus 用单色高亮
- card：直角或 2-4px 圆角、紧凑、阴影锐利
- empty：用数据可视化图表
- error：纯红 #ff3333

**这种"主题 → 元素表达"的对应关系是设计师的肌肉记忆**——AI 没有，必须显式喂。

#### 落点
- `recipes/theme-element-dict/{playful,minimal,premium,trustworthy,warm,clean,bold,natural}.md` —— 每种 theme intent 一份，每份覆盖 ~15 种核心元素的视觉表达 spec
- theme-generator SKILL 出场契约升级：必须给每个 theme 标 `intent` 枚举（已有），design-planner Phase A 入场必读对应词典
- SKILL Phase E 每个执行任务模板末尾要求：**"对照 theme-element-dict/X.md 第 N 节，本节点视觉是否符合？"**

#### 机器对账
- 弱：用 LLM-as-judge 跑词典对照（成本高，可选）
- 强：每个词典里给可机器校验项（如 minimal 主题 → button border-radius ≤ 4px；warm 主题 → 至少 1 个色相在 0-30°/300-360° 暖色区）

---

### Q3. AI 知道权重 + 整体和谐 → ✅ 权重金字塔 + 数值化对账

#### 设计师真实做法
和谐 = 视觉权重呈金字塔，每层之间有**数量级差异**：

```
主角层（CTA / Hero / 品牌符号）        weight 8-10  ← 最多 1-2 个节点
配角层（信息容器 / 主要内容）          weight 5-6   ← 占 30-40% 节点
工具层（输入 / 切换 / 链接）           weight 2-3   ← 占大多数
氛围层（装饰 / 背景）                  weight 1     ← 大面积低密度

强约束：
- 主角与配角 weight 差 ≥ 2
- 配角与工具 weight 差 ≥ 2
- 任意层节点数 ≤ 6（除工具层）
- 主角 ≤ 2 个，否则没主角
- 全屏 weight 总和 ≤ 30
```

权重通过具体视觉手段量化：
```
weight = f(色对比强度) + f(尺寸比) + f(字重差) + f(装饰密度) + f(动效)
每个 f 给 0-3 分，加起来就是这个节点的 weight
```

#### 落点
- `methodology/08-weight-pyramid.md` 替换当前 02-visual-budget.md：
  - 金字塔结构强约束
  - weight 量化公式（5 维度评分）
  - 最低识别阈值表（A2 v2 已写）
- `schema-spec/screen-meta-design.md` 升级：componentBudgets 加 `weightBreakdown: { color, size, weight, decoration, motion }` 子字段，每项 0-3
- SKILL Phase C 强制产出权重金字塔

#### 机器对账
- B2 工具集 `query/visual_weight_audit { screenId }` 返回：
  - 每节点实测 weight（基于 styles 自动算）
  - 与声明 weight 差异
  - 金字塔结构是否成立（差值 / 节点数 / 总和）

---

### Q4. AI 知道一个交互效果需要多元素协作 → ✅ 协同视觉 + 配方库

#### 设计师真实做法
"按钮浮出感"不是只改按钮：
```
按钮浮出感 = 多元素合奏：
  1. 按钮本身：transform: translateY(-2px) + boxShadow 加深
  2. 父容器：保留 padding 不变（防止跳动）
  3. 兄弟元素：opacity 微降至 0.95（focus 焦点）
  4. 背景：subtle 暗色蒙层（可选，强 focus 场景）
  5. transition：所有上述变化共用同一 ease-out 200ms
```

"表单聚焦感" / "提交紧迫感" / "支付信任感" / "上传期待感" / "搜索灵敏感"——每个都是 3-5 元素合奏。

**这种"视觉效果 = 配方"的思维必须明文教**——AI 默认会在单节点维度想，写不出合奏。

#### 落点
- `recipes/visual-effects/*.md` 配方库：浮出感 / 聚焦感 / 紧迫感 / 信任感 / 亲切感 / 灵敏感 / 期待感 / 完成感（8-12 份）
- 每份配方：
  ```
  名称：浮出感 (floating)
  典型使用场景：主 CTA hover / Card hover / 表单 focus 跟随
  参与元素：[主体, 父容器, 直接兄弟, 装饰]
  CSS 配方（含 token 引用）：...
  动效（含 transition）：...
  注意事项：transform 不能产生布局跳动 / shadow 别叠加重 / 避免对低性能设备用 blur
  对账信号：query/visual_state_distinctness 在该 hover 态下应有 ≥ 3 个 distinct override
  ```
- SKILL Phase E 每个执行任务模板要求：**"本视觉效果是否参考了某个 recipe？参考的 recipe 名称 + 参与节点列表 + 配方差异说明"**

#### 机器对账
- 任务的 expectedArtifacts 加 `recipeApplied: { name, participants: [...] }`
- B2 工具校验 participants 节点的 styles/visualStates 是否符合配方关键约束

---

### Q5. AI 知道基于视觉目的新建任务而非简单 CSS 调试 → ✅ 任务自创权

#### 设计师真实做法
设计师不会被一份"待办清单"束缚——发现"这屏需要 hero 渐变背景"，就**自创**一项新工作：
1. 设计 hero 区域几何
2. 选 3 色 / 比例 / 角度
3. 决定用 CSS 还是 PNG
4. 如果 PNG → 调 material-painter 画
5. 应用到节点 + 微调位置

**这一整套 = 一个任务**，不是"在 D-X-styles 里给一个 div 加 background"。

#### 落点
- SKILL Phase D「任务自创」工作流：
  ```
  Phase D 入场：拿到 Phase C 的 visualStrategy
  
  对每屏：
    1. 检视 strategy.decoration.instances —— 装饰需求点出来
    2. 检视 strategy.color.accent 用法 —— accent 色应在哪几个节点出现
    3. 检视 hero/CTA/品牌区是否需要重点雕琢任务
    4. 检视 recipes/visual-effects 哪些适用本屏交互
    5. 检视 recipes/compositions 哪些复合控件本屏出现 → 各开一个雕琢任务
    
  对每个发现 → meta/add_plan_tasks 自创一个：
  {
    id: "D-X-craft-N",
    title: "<视觉目标>，参与节点：<list>，参考配方：<recipe>",
    expectedArtifacts: [...]
  }
  ```
- SKILL §5.4 阶段边界放开：**design 阶段允许 add_plan_tasks**（当前已默认允许，但 SKILL 没引导主动用）
- `note-templates/self-created-task.template.md` 模板规定自创任务必含字段：visualGoal / participants / referenceRecipe / referenceMethodology / expectedArtifacts

#### 机器对账
- Phase D 必须自创 ≥ 3 个任务（少于 3 = 没真做策略）
- 每个自创任务必须挂 expectedArtifacts，否则 update_plan_task done 拒掉

### Q6. AI 系统性结合产品+主题→拆视觉任务 → ✅ Phase A→B→C→D 漏斗

#### 设计师真实做法
不是看到字段就填——而是：

```
Phase A 取景：
  read product.meta（用户/场景/情绪曲线）
  read theme.tokens + theme.intent
  read interaction 阶段 stateVisualMap、衍生视图、overlays
  → 写 briefing.md：本屏要解决什么用户问题、当前主题如何借用、interaction 给了哪些状态需要视觉化
  
Phase B 视觉概念：
  基于 briefing → 写 concept.md：
    灵魂句（一句视觉感受，如"清新校园社交，温度不轻浮"）
    风格关键词 3 个（如"暖白 / 极简 / 单色温度"）
    mood board 描述（如"晨光透过窗 / 笔记本上的便签 / 球场围栏的剪影"）
    candidate 风格 3 个 + 否决理由 + 选定理由
  
Phase C 视觉策略：
  基于 concept → 写 strategy.md（5 维齐全）：
    color: 60-30-10 = bg #FCFCFD / surface #FFFFFF / accent primary
    typography: display 32 → h2 22 → body 14 → caption 12 (1.4 比例)
    shape: 全屏统一圆角 12px（card）/ 8px（button）/ 4px（input）/ 6px（icon-button）
    decoration: 系统 = soft-glow，密度 = 节制（2 处），实例 = [hero 右上光斑, footer 浅渐变线]
    rhythm: 间距 = 4 → 8 → 16 → 24 → 32（2x 比例）；动效 = ease-out 200ms 主、150ms 辅
  
Phase D 任务自创（见 Q5）
```

四个 Phase 是漏斗——**前一棒输出是后一棒强制输入**。

#### 落点
- SKILL Phase A/B/C/D 是雷打不动顺序步骤

### Q7. AI 调整布局 + 装饰 → ✅ 放开布局调整权 + 装饰新建权 + 素材绘制权

#### 设计师真实做法
- **加一层视觉容器**：interaction 阶段建的 PolicyRow 是 div + Checkbox + Text 平铺，design 阶段为了"勾选 + 文字 + 图标可点击错落感"，可能需要再包一层 `<label>` 让整行点击；或为 hero 区加一个 `<HeroFrame>` 容纳 Logo + Slogan + 背景渐变
- **改装饰节点结构**：从 1 个 BgBlobTopRight → 拆成 BgBlobTopRight + BgBlobBottomLeft 形成视觉对角，营造空间纵深
- **画 + 应用素材**：BrandLogo 直接画一个简单几何 Logo + applyMaterialDesign

但当前 SKILL §5.4 严禁动节点结构——**这是错配**。

#### 怎么收紧 vs 放开（精确边界）

```
✅ design 阶段允许：
  - element/add  添加视觉容器（label 包裹、HeroFrame 包裹、装饰节点 div）
  - element/wrap 包裹现有兄弟节点
  - element/move 移动节点位置（同父级内）
  - 装饰节点 add（4 类：背景氛围 / 角落溢出 / 分割装饰 / 品牌点缀）
  - material-painter 调用 + 节点 materialProjectId 写入
  - 自创 plan tasks

❌ design 阶段禁止（这些是上游已定的业务真理，动了 = 推翻业务设计）：
  - element/remove 业务节点（product/interaction 建的）
  - 改任何节点的 events / bind / repeat / visibleWhen
  - 改 dataSources / 业务字段定义
  - 改 product.meta / theme.tokens（直接改 tokens 必须走 theme-generator UpstreamChallenge）

⚠️ 边界情况（需声明 + 走 UpstreamChallenge）：
  - 想改 interaction 建的衍生视图节点结构 → 走 UpstreamChallenge
  - 想加新业务字段（state.view）→ 走 UpstreamChallenge
```

#### 落点
- 重写 SKILL §5.4 阶段边界表（放开 / 收紧分明）
- `methodology/11-layout-adjustment.md` 详解何时该调、怎么调、对账规则
- `schema-spec/forbidden-fields-design.md` 升级，删去对 element/wrap、element/add、materialProjectId 的禁令

#### 机器对账
- `query/integrity` 不再因 design 阶段 add 节点 / wrap 而报警
- 但加新约束：design 阶段 add 的节点必须挂 `meta.design.kind ∈ [decoration, visual-container, material-frame]` 之一，否则视为"试图加业务节点" → 拒

## 10. 风险与权衡

| 风险 | 影响 | 缓解 |
